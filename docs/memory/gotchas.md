# Infrastructure Gotchas (Lessons Learned)

## CloudFormation

- **GroupDescription is immutable**: Changing `GroupDescription` on a security group forces REPLACEMENT. If `GroupName` is explicit, the new SG collides with the old name. Never change GroupDescription on deployed stacks — use option 3 (rename both GroupName + GroupDescription) or tear down first.

- **resolve:ssm is NOT re-evaluated on redeploy**: `{{resolve:ssm:/path}}` in templates only resolves when CloudFormation detects a resource change. Updating the SSM parameter value alone does NOT trigger re-resolution. Fix: use a `ForceUpdateToken` parameter referenced in a tag, pass `$(date +%s)` from Makefile.

- **Image Builder components are immutable**: Updating the CloudFormation template creates new component CONTENT, but the recipe continues using the old version until `RecipeVersion` is bumped in the parameter file. Always bump RecipeVersion when changing component content.

- **CFN template edits in git don't auto-deploy**: Editing `cloudformation/cf-*.yaml`, committing, and pushing does NOT update the AWS stack — `make deploy-<stack>` is still required. Easy to miss when juggling many stacks; a template change that's needed by another change can sit in the repo for days while debugging the downstream symptom. Discovered (again) 2026-05-14: `cf-iam.yaml` was edited 6 days ago to grant `worxco/<env>/drupal/*` read, never deployed, then `configure-php.sh` silently produced empty DB passwords for an entire week. **Followup**: a `make check-drift` target that compares `cloudformation/cf-*.yaml` mtimes against each stack's `LastUpdatedTime` and warns when local is newer.

- **`systemctl start` is a NO-OP when the unit is already running** — use `restart` after editing config: This bit us once we made the AMIs `systemd enable` PHP-FPM and nginx (so they auto-start at boot before cloud-init's scripts_user phase). By the time `configure-php.sh` runs and writes the marker block into `pool.d/www.conf`, FPM is already up with the AMI-baked www.conf (no marker block). `systemctl start php8.3-fpm` at the end of the script does nothing; FPM keeps serving with `env[]` from the empty AMI-baked pool config. Workers' `getenv('DRUPAL_DB_HOST')` returns empty, settings.php's `trusted_host_patterns` never gets `DRUPAL_SITE_NAME` appended, Drupal returns HTTP 400 "host name not valid" — even though the on-disk config dump (`php-fpm -tt`) shows everything is fine. **Fix**: use `systemctl restart` (not `start`) for any service whose config was just modified. Discovered 2026-05-18 after Drupal worked Friday (we kept manually `restart`-ing during debugging), then "stopped working" Monday because new boxes had only ever done a `start`.

- **SSM Session Manager bypasses ubuntu's login shell**: `chsh -s /bin/zsh ubuntu` updates `/etc/passwd` but Session Manager's SSM-SessionManagerRunShell document explicitly does `exec bash -l`, ignoring the user's preferred shell entirely. The bash that lands in your SSM session is from the SSM doc, not the user's login shell. To get a different shell inside SSM: either (a) edit the SSM doc to invoke zsh (account-level — affects every box you SSM into; needs zsh installed on each), or (b) keep bash as the SSM entry shell and append an auto-exec to `/etc/bash.bashrc` that swaps to zsh for interactive sessions (`exec zsh -l`). Pattern (b) is local to one host and degrades safely. Discovered 2026-05-15.

- **zsh prompts need `setopt PROMPT_SUBST` for `$(...)` to evaluate**: Without that option set, `RPROMPT='$(some_func)'` displays the literal string `$(some_func)` instead of running it. Classic zsh trap that catches everyone the first time. Set it once in zshrc before any prompt with command substitution. Discovered 2026-05-15.

- **PHP-FPM `listen.allowed_clients` does NOT accept the word "any"**: The directive takes a comma-separated list of IPv4 addresses. If left blank/unset, all IPs are allowed (this is the default). But if you set it to the literal string `any`, PHP-FPM treats that as "allow only the single IP literally named 'any'" — which never matches, so EVERY connection is rejected with `ERROR: Connection disallowed: IP <addr>` written to stderr (NOT to the FPM error log). nginx sees this as "premature close" or "connection reset by peer", which looks identical to a worker crash, an upstream/network issue, or a bad password — sending you on a 3-day wild-goose chase. **Fix**: omit the directive entirely (or set it blank), and rely on the security group for network-level ACL. Discovered 2026-05-15 via strace on an FPM worker that revealed the rejection happens BEFORE any FastCGI parsing.

- **AWS-generated passwords need shell-metachars excluded**: `AWS::SecretsManager::Secret` with `GenerateSecretString` honors `ExcludeCharacters`, but the default character set still permits shell metacharacters (`$`, `` ` ``, `~`, `*`, `<`, `>`, `|`, `!`, `(`, `)`, `{`, `}`, `^`). Any of those at the start of a value breaks PHP-FPM INI parsing of unquoted `env[KEY] = value` lines (`syntax error, unexpected '~' in Unknown on line 1`), AND those that survive into shell heredocs in install-drupal.sh undergo expansion (`$VAR` substitution, backtick command substitution). **Fix**: tighten `ExcludeCharacters` to also exclude all of those, AND have any consumer that writes the value into an INI/config file quote it (`env[KEY] = "value"`). Discovered 2026-05-14: a generated DB password `X~pz...` caused PHP-FPM to refuse loading www.conf, which combined with the workers from boot still being alive on an earlier (valid) config made the failure look like a NLB or network bug for hours.

- **`configure-php.sh` boot-script writes only run at instance launch**: The script is baked into the AMI at `/opt/worxco/configure-php.sh` and invoked from cloud-init UserData. Editing the script in S3 (via `make upload-build-configs`) does NOT propagate to running instances — only future AMI builds pick up the new version. To update existing boxes without an AMI rebuild you either need (a) the PHPInstanceRole's `S3Access` to include the `image-builder` bucket so the live script can refresh itself from S3, or (b) push the new script via SSM `send-command` + base64. Discovered 2026-05-14.

- **PHP-FPM `include=` only works at the GLOBAL level, not inside `[pool]` sections**: Tried to use a `pool.d/extra/*.conf` drop-in pattern by adding `include=/etc/php/8.3/fpm/pool.d/extra/*.conf` inside the `[www]` section of `www.conf`. PHP-FPM rejects this with `Unable to include /etc/php/8.3/fpm/pool.d/www.conf from /etc/php/8.3/fpm/php-fpm.conf at line 10` — the inner include directive itself is invalid in pool context. Boot fails, ELB health checks kill the instance, ASG cycles forever. **Fix**: write `env[...]` and `php_value[...]` directives directly into `www.conf` between `; BEGIN worxco-boot` / `; END worxco-boot` markers (idempotent strip+append on each boot via `sed -i` then `>>`). Discovered 2026-05-08 — first AMI rev that contained the wired Drupal env vars never reached InService, instances cycled with `php8.3-fpm.service: Main process exited, code=exited, status=78/CONFIG`.

- **AWS::ImageBuilder::Component `Data` field has a hard 16,000-char limit**: The component-data string AWS validates is post-`Fn::Sub` size, NOT the YAML source size. **cfn-lint's `W1031` warning ("Fn::Sub when resolved is longer than 16000") is the canary** — exit code is "warnings only" (4) but the AWS API will reject the deploy with `Model validation failed (#/Data: expected maxLength: 16000, actual: NNNNN)`, the stack rolls back, and `build-amis` will silently make AMIs from the previous (rolled-back) component versions — so subsequent compute deploys ship STALE code that LOOKS like it deployed cleanly. **Never dismiss W1031 on `AWS::ImageBuilder::Component` resources**. Fix: move large boot scripts to `image-builder/configs/<file>`, sync to S3 via `make upload-build-configs`, and have the component just `aws s3 cp` it down. Discovered 2026-05-08 — first attempt to deploy configure-php.sh changes hit 21,094/16,000 and rolled back undetected.

- **.env file format**: `.env` uses bare `KEY=VALUE` (no `export`). Make reads it via `-include .env` + bare `export`. Adding `export` keywords would break Make. For bash, use `set -a; source .env; set +a` or `export AWS_PROFILE=ZI-Sandbox` explicitly.

## VPC / Networking

- **VPC endpoints need BOTH AZs**: Interface endpoints (SSM, Secrets Manager) and Gateway endpoints (S3) must be in all AZs where instances run. Single-AZ endpoints cause failures for instances in the other AZ. Use `!If [HasTwoAZs, ...]` conditions.

- **S3 gateway endpoint uses public IPs**: Even though traffic routes through the VPC endpoint, S3 resolves to public IPs. Security groups need HTTPS egress to `0.0.0.0/0` (or the S3 prefix list), not just to the endpoint security group.

- **FSx NFS mount options**: The original tuned options (`nfsvers=4.1,rsize=1048576,wsize=1048576,hard,timeo=600,retrans=2,noresvport,noatime`) cause "Operation not permitted" from cross-subnet instances. Simplified `vers=4.1,port=2049` works reliably across all subnets.

- **Private instances can't reach internet**: No NAT gateway by design. Outbound AWS API access via VPC endpoints only. Can't `apt-get`, `curl`, or `pip install` from private instances. Use S3 as a relay for files.

## SSM

- **Snap SSM agent ignores /etc/amazon/ssm/**: The snap-installed SSM agent on Ubuntu doesn't read config from `/etc/amazon/ssm/amazon-ssm-agent.json`. RunAs configuration must be done via account-level `SSM-SessionManagerRunShell` document instead.

- **SSM-SessionManagerRunShell is account-level**: This document affects ALL SSM sessions in the account, not just the deploy host. It must be deleted before stack deletion and recreated on deploy (Makefile handles this).

- **SSM sessions start as ssm-user by default**: Without the RunShell document, sessions land as `ssm-user` in `/var/snap/amazon-ssm-agent/`. The document configures `ubuntu` user with `cd ~ && exec bash -l`.

- **`list-command-invocations` truncates Output at ~2500 chars**: Use `aws ssm get-command-invocation --command-id $CMD --instance-id $ID` to get the full `StandardOutputContent` / `StandardErrorContent` for a specific instance. Otherwise you'll see the FIRST 2500 chars of output and miss the actual error at the bottom. Discovered 2026-05-19 debugging install-drupal SSM dispatches where the composer install log filled the buffer before the actual error message.

- **AWS-RunShellScript runs as root with `HOME` unset**: Tools that require `HOME` (composer, drush, npm, anything that wants to read/write `~/.<tool>`) will fail with "Cannot find the home directory path" or similar. **Fix**: set `export HOME=/root` at the top of any script dispatched via SSM, or pass `HOME=/root` to specific tool invocations. Also `composer` refuses to run as root without `COMPOSER_ALLOW_SUPERUSER=1` (and disables plugins, breaking Drupal install). Set both at the top of install-drupal.sh. Discovered 2026-05-19.

- **The literal `(` in an SSM `commands=[...]` arg is a shell syntax error**: When you write `commands=["echo Running configure-php.sh again (now that SSM params exist)..."]`, the SSM agent writes the command to a script file and runs it as sh. Sh sees `(` outside quotes and tries to start a subshell, fails with `Syntax error: "(" unexpected`. **Fix**: either escape paren content properly inside the JSON-encoded SSM args, or (easier) wrap the inner script in base64 and dispatch as `["echo $B64 | base64 -d | bash"]` — sidesteps the whole double-escape problem. Discovered 2026-05-19.

- **Multiple SSM `--instance-ids` must be passed as separate AWS CLI args**: `--instance-ids "id1 id2"` (single quoted string with space) is interpreted as one invalid 37-char instance ID. `--instance-ids id1 id2` (two separate args) works. macOS-default zsh doesn't support bash's `read -a`; use either explicit hardcoded args or `for id in $LIST; do ...` to expand. Discovered 2026-05-19.

## IMDSv2

- **HttpTokens: required breaks IMDSv1 boot scripts**: If AMIs use `curl -s http://169.254.169.254/...` (IMDSv1), launch templates must set `HttpTokens: optional`. IMDSv2 requires token-based requests. Boot scripts updated in templates but need AMI rebuild to take effect.

## AWS Profile

- **ZI-Sandbox is account 033879516417**: The `ZI-Sandbox` AWS CLI profile assumes role into account 033879516417 via `OrganizationAccountAccessRole`. The parent account (645925380349, alias `worxco`) is the management account — NOT the sandbox.

- **Deploy host uses instance role, not profile**: No `~/.aws/credentials` needed. `AWS_DEFAULT_REGION` set via `/etc/profile.d/`. No `AWS_PROFILE` should be set.

## Deploy Host Lifecycle

- **`make deploy-deploy-host` ALWAYS replaces the EC2 instance** — even for output-only template changes. The UserData has tokens/timestamps that change each deploy, so CloudFormation tears down the old instance and launches a new one. **Do not run `make deploy-deploy-host` from a shell on the deploy host itself** — your SSM session dies with the box. Run it from your Mac (or another machine) instead. The new instance auto-clones the repo and is functionally identical to the old one. Discovered 2026-05-06 during VPC peering rollout.

- **`!Ref` on AWS::EC2::SecurityGroup without explicit VpcId returns the GroupName, not the GroupId**: Legacy EC2-Classic Ref behavior that AWS still falls back to when `VpcId` is unspecified, even though EC2-Classic itself was retired in 2022. The SG resource is created in the default VPC, but `!Ref` returns the literal name string (e.g., `cf-deploy-host-sg`), which is invalid wherever an `sg-xxxxxxxxx` ID is expected. **Fix**: always use `!GetAtt MySecurityGroup.GroupId` for outputs/exports — works regardless of how the SG was created. Discovered 2026-05-06 when cf-deploy-peering's `SourceSecurityGroupId` rules failed with `Invalid id: cf-deploy-host-sg (expecting sg-...)`.

- **`grep '^KEY=' env-file | cut ...` breaks under `set -euo pipefail` when the file uses `export KEY=value` lines**: grep returns 1 (no match because lines start with `export `), pipefail propagates the 1 through cut, set -e kills the script. **Better pattern**: source the file in a subshell and echo the variable. Works for both formats:
  ```bash
  value=$(. /path/to/envfile 2>/dev/null; echo "${SOME_VAR:-}") || true
  ```
  Discovered 2026-05-08 when bootstrap.sh's auto-mount step crashed silently (the script died before the MOTD step ran on a fresh deploy-host).

- **`source <file>` doesn't export vars to child processes — use `export KEY=value` lines**: A bash file with `KEY=value` (no `export`) sets variables in the current shell, but they're NOT exported. Child processes (drush running PHP, etc.) can't see them via `getenv()`. Symptom: `drush status` shows `DB hostname : localhost` (the settings.php fallback default) even though we just sourced an env file with `DRUPAL_DB_HOST=<actual-rds>`. **Fix**: env files written for sourcing into a shell that will spawn children must use `export KEY=value` for every variable. Applied 2026-05-08 to `/etc/worxco/envs/<env>` files (generated by `refresh-env-config`).

- **AWS-managed RDS master credentials are stored as JSON, not plaintext**: When RDS is configured with `ManageMasterUserPassword: true`, AWS creates a Secrets Manager entry with structure `{"username":"dbadmin","password":"..."}` — NOT a plaintext password. Symptom: `psql` connection fails with `password authentication failed for user "dbadmin"` because the entire JSON blob is being passed as the password. **Fix**: scripts that fetch the master password must extract `.password` via `jq`. The defensive pattern that handles both AWS-managed (JSON) and hand-created (plaintext) secrets:
  ```bash
  RAW=$(aws secretsmanager get-secret-value --secret-id $ID --query SecretString --output text)
  PASS=$(echo "$RAW" | jq -r '.password // empty' 2>/dev/null || true)
  [ -z "$PASS" ] && PASS="$RAW"
  ```
  Applied 2026-05-08 to psql-env, valkey-env, install-drupal.sh, remove-drupal.sh, and the Makefile verify-drupal target.

- **VPC peering does NOT enable cross-VPC DNS resolution by default**: After `aws ec2 create-vpc-peering-connection`, instances in one VPC can route packets to the other, but they CANNOT resolve the peer VPC's private DNS names. RDS and ElastiCache work across peering anyway because their hostnames are AWS-public DNS that resolve to private IPs via split-horizon (works from any VPC). **FSx is the exception — its hostname is resolvable only from within its home VPC's internal resolver, even with `AllowDnsResolutionFromRemoteVpc=true` set on the peering connection**. The flag controls Route 53 private hosted zones; FSx OpenZFS uses VPC-internal DNS, which is not affected by the flag. Setting the flag is still required (and `make deploy-peering` does it automatically) for any future Route 53 private zones, but it does NOT solve FSx. **Solution**: maintain `/etc/hosts` entries on the deploy host mapping each FSx hostname → private IP. The IP is stable for SINGLE_AZ_1 deployments. The `refresh-env-config` script does this automatically — looks up `aws fsx describe-file-systems` → `aws ec2 describe-network-interfaces` to find the private IP and rewrites a managed block in `/etc/hosts` (between `# BEGIN/END worxco-fsx-hosts` markers). Discovered 2026-05-06 during peering connectivity testing.

- **Unconditional pre-delete of `SSM-SessionManagerRunShell` silently breaks SSM sessions**: Earlier versions of `make deploy-deploy-host` always pre-deleted the SSM document before running CloudFormation. CFN's stack state already had `SSMSessionPreferences` as `CREATE_COMPLETE` and saw no template change, so it did nothing — leaving the stack happy but the document gone. Result: SSM sessions fall back to default `ssm-user` with `sh` instead of `ubuntu` with `bash`. **Fix** (2026-05-06): pre-delete only when the stack does NOT yet exist (orphan cleanup case). When the stack exists, CFN already owns the document. **Recovery for orphaned state**: manually `aws ssm create-document --name SSM-SessionManagerRunShell` with the same content as the template — CFN's physical-id is just the document name, so re-creating it puts CFN's state back in sync.

## Drupal Install (cloud path — RDS + FSx)

- **drush 13's `--db-url` doesn't round-trip aggressive URL encoding**: `urllib.parse.quote(password, safe='')` percent-encodes chars that are valid sub-delims in RFC 3986 userinfo (`'`, `,`, etc.). drush's URL parser doesn't decode them back symmetrically, so postgres gets a mangled password and rejects auth with `fe_sendauth: password authentication failed`. The same password works fine via `PGPASSWORD` env var with direct psql. **Fix**: pass the password raw in `--db-url`. `cf-app-drupal`'s `GenerateSecretString` `ExcludeCharacters` already excludes every char that genuinely needs URL encoding in userinfo (`@`, `:`, `/`, `?`, `#`, `%`, `[`, `]`, etc.); what remains (alphanumeric + `'`, `,`, `.`, `-`, `_`) is valid raw in userinfo. Discovered 2026-05-19. The "defense in depth" URL encoding I'd added in commit `a52e38a` was the actual cause of the bug, not a fix.

- **Drupal's nginx vhost lives on FSx, not in the AMI**: The base nginx config (baked into the AMI) is `/etc/nginx/nginx.conf` and ends with `include /etc/nginx/shared/sites-enabled/*.conf` — and `/etc/nginx/shared` is an NFS mount of `$FSX:/fsx/nginx`. The Drupal vhost (server_name = $SITE_NAME, root = /var/www/drupal/web, fastcgi_pass php83) is written to `/fsx/nginx/sites-enabled/drupal.conf` by `install-drupal.sh` (Phase 7 of deploy-allX). nginx instances see it instantly via NFS but only load config at startup — call `make reload-nginx ENV=<env>` (or rely on install-drupal-remote's auto-reload) after publishing or updating the vhost. Discovered 2026-05-19 when the first end-to-end deploy-allX returned 404 from nginx because no vhost existed.

- **`configure-nginx.sh` bails silently when `/fsx/nginx` doesn't exist on FSx**: At boot, the script tries `mount -t nfs4 ... $FSX:/fsx/nginx /etc/nginx/shared`. If `/fsx/nginx` doesn't exist on FSx (because no one has created it yet — chicken-and-egg with deploy-allX phase ordering), the mount fails. `set -e` at the top of the script means the script EXITS at that point — BEFORE the step that writes `/etc/nginx/conf.d/upstream-php.conf` (the file that defines the `php74` and `php83` upstream blocks). nginx still starts (systemctl loads it independently) but with no upstreams, so any vhost using `fastcgi_pass php83` fails `nginx -t` with `no port in upstream "php83"`. Discovered 2026-05-19. **Fix (pending)**: P0 in TODO — phase-reorder deploy-allX so `/fsx/nginx` is pre-created before nginx instances launch.

- **`configure-php.sh` skips the env[] block when `/sandbox/rds/endpoint` SSM param is missing**: PHP instances launch in deploy-allX Phase 4 (compute), BEFORE the data layer (Phase 5: RDS + ElastiCache). When configure-php.sh runs at boot and reads `/sandbox/rds/endpoint`, the param doesn't exist yet, so the script logs `WARNING: /sandbox/rds/endpoint not found - boot-config block NOT written` and exits without populating `www.conf` with `env[DRUPAL_*]` directives. PHP-FPM starts with no Drupal env vars; every request gets HTTP 500 ("Missing $settings['hash_salt']" or similar). Manual recovery: SSM-dispatch a re-run of `/opt/worxco/configure-php.sh` on every PHP box AFTER the data layer is up. Discovered 2026-05-19. **Fix (pending)**: P0 in TODO — phase-reorder deploy-allX so data layer precedes compute.

- **settings.php FSx paths were stale after the `/var/www` cutover**: Commit `ee319d5` (2026-05-16) cut over from per-env-prefix FSx mount paths (`/var/www/<env>/drupal-private/...`) to single-mount paths (`/var/www/drupal-private/...`). install-drupal.sh's own `INSTALL_DIR`/`PRIVATE_DIR`/`CONFIG_DIR` variables were updated, but the embedded settings.php template still constructed `'/var/www/' . $_env . '/drupal-private/salt.txt'` for the three FSx-path settings (`$_salt`, `file_private_path`, `config_sync_directory`). Result: install completed, salt.txt was written to the right place on FSx, but settings.php couldn't read it back. Drupal logged `RuntimeException: Missing $settings['hash_salt']` on every request → HTTP 500. Fix (commit `417f9f0`, 2026-05-19): drop the `$_env` segment from all three paths. Lesson: when cutting over mount layouts, grep for the OLD path pattern across ALL templates and scripts that construct paths — not just the variables that hold the new path.

- **PHP-FPM `clear_env = yes` makes `/proc/$PID/environ` empty even when `env[]` directives are set**: PHP-FPM's `env[]` populates PHP's SAPI environment (visible via `getenv()`, `$_SERVER`, `$_ENV`) but does NOT set the OS-level process environment. With `clear_env = yes` (our default — for security), worker processes have a completely empty `/proc/$PID/environ`. **Diagnostic trap**: don't verify env injection by looking at `/proc/$PID/environ` (it'll always be empty). Instead, write a tiny PHP probe and curl it through nginx: `<?php foreach($vars as $v) echo "$v: " . strlen(getenv($v)) . "\n";`. Discovered 2026-05-19 after ~30 min of false-positive "workers don't have env vars" diagnosis.

- **install-drupal.sh refuses to re-run when `.installed` marker exists**: install-drupal.sh has a marker check at the top (`if [ -f "$MARKER" ]; then exit 1`). For partial-failure retries where the marker was never written, that's fine — the next run cleans up and proceeds. But if you need to update something post-install (e.g., the nginx vhost on FSx wasn't written by the older install-drupal.sh), running install-drupal again won't work — it short-circuits at the marker. **Fix path**: either (a) `make reinstall-drupal ENV=<env>` (full wipe + reinstall — overkill for one-off fixes), or (b) use surgical Make targets that bypass the install: `make publish-drupal-vhost`, `make restart-php-fpm`, `make clear-drupal-cache`. Pattern established 2026-05-19.
