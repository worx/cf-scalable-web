## What's left for full Phase C

Today, Drupal is reachable only via `drush runserver` + SSH tunnel. The actual production path (browser → ALB → NGINX → NLB → PHP-FPM) doesn't yet serve the site because the **PHP-FPM auto-scaling group instances don't know how to talk to the Drupal install on FSx**.

The piece needed: extend the existing `/opt/worxco/configure-php.sh` boot script (already on the PHP AMIs) to:

1. Mount FSx at `/var/www/sandbox` (same path the deploy-host uses)
2. Read `/sandbox/drupal/*` SSM parameters and `worxco/sandbox/drupal/db-password` secret
3. Write env vars into the PHP-FPM pool config (`/etc/php-fpm.d/drupal.conf`):
   ```
   env[ENVIRONMENT_NAME] = sandbox
   env[DRUPAL_DB_HOST]   = sandbox-postgres.xxx.amazonaws.com
   env[DRUPAL_DB_NAME]   = drupal
   env[DRUPAL_DB_USER]   = drupal_user
   env[DRUPAL_DB_PASS]   = <fetched from Secrets Manager>
   env[DRUPAL_SITE_NAME] = drupal-sandbox.test
   ```
4. Restart PHP-FPM so the env vars take effect

After that, when an HTTP request hits the ALB → NGINX → NLB → PHP-FPM, the PHP process has the env vars set, settings.php's `getenv()` calls work, Drupal connects to RDS — exactly as the install-drupal flow proved.

This requires:
- Editing the PHP AMI (or its build script in cf-image-builder)
- Possibly a NGINX `server_name` for the trusted host pattern
- An AMI rebuild + redeploy of the PHP compute stack
