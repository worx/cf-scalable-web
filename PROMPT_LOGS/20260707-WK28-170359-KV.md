### 17:05:53 - Response

Good responses, all four. Let me address each:

---

**Item 1 — no read replica separation**

Understood. Single-instance clusters: the writer and `-ro` endpoints both point at the same node, so no isolation gained from the `-ro` endpoint. Given you're monitoring and prod is at 12% CPU on an r5.large, the actual load impact is fine. Just note for the runbook: `--single-transaction` in your mysqldump flags means no table locks, which is the other half of the safety story. **Does your current running mysqldump have `--single-transaction`?** If not, InnoDB tables might be getting brief metadata locks and MyISAM tables (unlikely in a Drupal 8+ DB but possible) would get full table locks for the duration of the dump. Worth confirming, but if CPU is calm you're probably fine.

---

**Item 2 — tt.sh `read -s` failure**

Almost certainly this: the script has no shebang. If you ran it as `sh scripts/tt.sh` (or a hook or something did), Ubuntu's `/bin/sh` is dash, and dash's built-in `read` doesn't support `-s`. Bash does.

Two fixes, either works:

```bash
# Fix A: add shebang, mark executable, run directly
sed -i '1i #!/bin/bash' scripts/tt.sh
chmod +x scripts/tt.sh
./scripts/tt.sh

# Fix B: just run under bash explicitly (no script edit)
bash scripts/tt.sh
```

I'd do Fix A — the shebang makes intent explicit for anyone (including future-you) who reads it later.

If you want to keep the script POSIX-portable (works under dash), the equivalent is:

```sh
printf 'Prod MySQL password for user worxco: '
stty -echo
read PROD_PW
stty echo
printf '\n'
```

But bash is fine for our purposes.

**Also:** the script as written points pgloader at `127.0.0.1:3306` which is the tunnel to prod. If we switch to the dump-restore-local approach (which I recommend below), we'll want to update the script — different port or different address for the local MySQL, and no tunnel needed during the pgloader phase.

---

**Item 3 — dump growing past 500 MB**

Rough shape of what to expect:

| Compressed dump (final) | Uncompressed | Restored MySQL data-dir |
|-------------------------|--------------|-------------------------|
| ~500 MB | ~1.5-2 GB | ~3-5 GB (with indexes) |
| ~1 GB | ~3-4 GB | ~6-10 GB |
| ~2 GB | ~6-8 GB | ~12-20 GB |

Drupal DBs are index-heavy (watchdog, cache_*, node_field_data indexes), so the restored footprint is typically 2-3× the uncompressed dump. Let's see where yours lands.

---

**Item 4 — MySQL over NFS (your actual question)**

Direct answer: **it works but I'd avoid it.** Not because of speed — because of MySQL's file-locking behavior over NFS. Options ranked:

| Option | Cost | Time | Risk | Notes |
|--------|------|------|------|-------|
| **Best: dedicated EBS gp3 volume** | ~$2-3 for a few days | 5 min to attach + mkfs + mount | Very low | 30 GB gp3, mount at `/var/lib/mysql`, install MySQL there. Detach and delete when done. |
| **OK: extend deploy-host root volume** | ~same $/GB | 3 min + resize2fs | Very low | Simpler if you don't mind the root volume growing. Won't shrink back easily though. |
| **Discouraged: MySQL data-dir on FSx (NFS)** | $0 (already have it) | 0 min | Moderate | MySQL's fcntl locking over NFS has historical issues. For a one-shot scratch DB the *practical* risk is low, but pgloader errors would be hard to diagnose ("is this a real schema issue or MySQL confused by NFS?"). Extra headaches you don't need. |

The dump *file* on FSx is totally fine — that's what NFS is for. It's live MySQL storage on NFS that's the anti-pattern.

**My recommendation:** attach a 30 GB EBS gp3 to deploy-host, put MySQL data there. Costs pennies for the migration timeframe. Something like:

```bash
# from your laptop or deploy-host with sandbox creds
aws --profile ZI-Sandbox ec2 create-volume \
  --availability-zone <deploy-host-AZ> \
  --volume-type gp3 \
  --size 30 \
  --tag-specifications 'ResourceType=volume,Tags=[{Key=Name,Value=migration-mysql-scratch},{Key=Purpose,Value=temporary-migration}]'

# then attach, mkfs.
