```
fix(refresh-env-config): Use DRUPAL_DB_* variable names that match settings.php

The cached /etc/worxco/envs/<env> files exported RDS_ENDPOINT,
RDS_PORT, RDS_USER, RDS_DATABASE — descriptive of what they are.
But settings.php (and PHP-FPM pool config) read DRUPAL_DB_HOST,
DRUPAL_DB_PORT, DRUPAL_DB_NAME, DRUPAL_DB_USER. Sourcing the file
gave variables that nothing actually consumed.

Symptom on the deploy host:
  source /etc/worxco/envs/sandbox
  export DRUPAL_DB_PASS=...
  vendor/bin/drush runserver 0.0.0.0:8080

  In BootstrapHook.php line 40:
    Bootstrap failed. Run your command with -vvv for more information.

(settings.php's getenv('DRUPAL_DB_HOST') returned null, fell back to
'localhost', and Drupal couldn't connect to the database.)

Fix: rename the variables in /etc/worxco/envs/<env> to match what
settings.php and the (future) PHP-FPM boot script expect:

  ENVIRONMENT_NAME      (was DRUPAL_ENV)
  DRUPAL_DB_HOST        (was RDS_ENDPOINT)
  DRUPAL_DB_PORT        (was RDS_PORT)
  DRUPAL_DB_NAME        (was RDS_DATABASE)
  DRUPAL_DB_USER        (was RDS_USER, now correctly drupal_user
                         instead of dbadmin — Drupal uses the limited
                         user, not the master)
  DRUPAL_SITE_NAME      (new — for trusted_host_patterns)
  DRUPAL_PATH           (kept — install path)

  FSX_DNS, VALKEY_HOST, VALKEY_PORT, ALB_DNS  (unchanged — these
  aren't Drupal-specific)

Now a single 'source /etc/worxco/envs/sandbox' is sufficient for
drush rs to find the env vars settings.php needs. User only needs
to additionally export DRUPAL_DB_PASS (which is in Secrets Manager,
not in the cached file by design).

Memory comment in bootstrap.sh MOTD updated to reflect new names.

To take effect on a running deploy host:
  cd ~/projects/cf-scalable-web && git pull
  sudo install -m 0755 scripts/deploy-host/refresh-env-config /usr/local/sbin/
  sudo refresh-env-config sandbox
  cat /etc/worxco/envs/sandbox    # confirms new names
```
