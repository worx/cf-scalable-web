fix(secrets): Defensive JSON-or-plaintext handling for Secrets Manager reads

RDS is configured with ManageMasterUserPassword: true, which makes AWS
generate and rotate the master password automatically. AWS stores it
as JSON, not plaintext:

  {"username": "dbadmin", "password": "..."}

Our scripts were fetching SecretString as-is and using the whole JSON
blob as the password — RDS rejected with:

  FATAL: password authentication failed for user "dbadmin"

Adopted a defensive parsing pattern that handles both formats:
  RAW=$(aws secretsmanager get-secret-value ... --query SecretString --output text)
  PASS=$(echo "$RAW" | jq -r '.password // empty' 2>/dev/null || true)
  [ -z "$PASS" ] && PASS="$RAW"

If the secret is JSON with a 'password' key, extract it. Otherwise
use the raw string. Forward-compatible: hand-created plaintext secrets
keep working, AWS-managed JSON secrets now work too.

Applied to:
  scripts/deploy-host/psql-env             (RDS master password)
  scripts/deploy-host/valkey-env           (Valkey AUTH token — defensive,
                                            currently plaintext)
  scripts/deploy-host/install-drupal.sh    (RDS master + drupal db pw +
                                            drupal admin pw)
  scripts/deploy-host/remove-drupal.sh     (drupal db pw)
  Makefile verify-drupal                   (drupal db pw)

Memory updated with the gotcha and the defensive pattern.

Discovered 2026-05-08 during the first cloud install attempt's pre-
flight sanity check (psql-env sandbox).
