### Design shape (proposed)

**S3 layout — adds `config/` to the existing three prefixes:**
```
s3://sandbox-migration-kv-worxco/
├── scripts/
├── dumps/
├── logs/
└── config/                                          ← NEW
    ├── jumpbox-state-latest.tar.gz                  ← pointer to newest (for auto-restore)
    └── jumpbox-state-YYYYMMDD-HHMMSSZ.tar.gz        ← timestamped archives
```

**Naming convention:**
- Timestamped archive is the durable record
- `-latest.tar.gz` is a *copy* of the newest timestamped archive, updated atomically at backup time
- Auto-restore reads from `-latest.tar.gz` — simple, no S3 listing needed on boot

**Scripts (~150 lines each, following our established pattern):**

| Script | Purpose |
|--------|---------|
| `migration/scripts/jumpbox/backup-state.sh` | Tar the "state paths" list → S3 (timestamped + latest) |
| `migration/scripts/jumpbox/restore-state.sh` | S3 `-latest.tar.gz` → extract to / |

**Make targets (invoke via same `_ssm-run-jumpbox.sh` helper):**

| Target | Behavior |
|--------|----------|
| `make backup-jumpbox` | On-demand backup — safe anytime |
| `make restore-jumpbox` | On-demand restore (rare — auto-restore usually handles it) |
| `make list-jumpbox-backups` | Show archives + sizes + timestamps |
