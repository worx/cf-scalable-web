```
fix(cf-app-drupal): exclude shell metachars from generated passwords

cf-app-drupal's GenerateSecretString ExcludeCharacters only excluded
quote/at/slash/backslash/etc., leaving shell metacharacters $/`/~/*/<>
free to appear in the auto-generated DB and admin passwords. A `~` near
the start of the DB password broke PHP-FPM INI parsing of www.conf
(`syntax error, unexpected '~' in Unknown on line 1`); separately, any
of those characters surviving into install-drupal.sh's psql heredoc
would undergo shell expansion.

Tighten the exclusion sets:
- Admin password: add ~$\`<>*|^!(){} to existing "@/\\'
- DB password:    add ~$\`<>*|^!(){} to existing "@/\\'#%&+:;=?[]

Note: deploying this change does NOT rotate the existing secret values.
A separate manual rotation + matching ALTER USER on the postgres side
is required to remove the in-place `X~pz...` password from circulation.

### 16:56:39 - Git Commit

**Commit:** `5073dc12ab42`

**Message:**
```
fix(configure-php): defensively quote every env[] value in www.conf

PHP-FPM's INI scanner rejects unquoted bareword values that contain
shell metacharacters (tilde, backtick, asterisk, angle brackets, etc.).
The cf-app-drupal-generated DB password can include any of those, and
when one happens to land near the start of the value PHP-FPM aborts
loading the entire pool config with `syntax error, unexpected '~'`.

Wrap each value in double quotes and strip any embedded double quote
defensively (the existing ExcludeCharacters list bans `"`, but defense
in depth is cheap). Behavior with a tame password is unchanged.

Bump RecipeVersion 1.0.7 → 1.0.8 so the next AMI build pulls the new
script. Existing instances still run the old (unquoted) version baked
into their AMI — for a runtime patch use SSM send-command + base64.
