```
fix(install-drupal): pass raw password in --db-url; drop URL-encoding

The Python urllib.parse.quote(safe='') step that I added in commit
a52e38a as "defense in depth" was actually CAUSING the auth failure
we hit today in Phase 7 of deploy-allX.

Diagnostic 2026-05-19:
  - ALTER ROLE drupal_user WITH PASSWORD <secret value> — succeeded.
  - Direct psql connect with PGPASSWORD=<secret value> — authenticated.
  - drush site:install with same value, URL-encoded in --db-url — failed
    with "password authentication failed for user drupal_user".

That triangulates the bug to drush 13.7's URL parser: it doesn't
round-trip Python's aggressive percent-encoding back to the original
bytes. Specifically, urllib.parse.quote(safe='') encodes EVERY char
outside [A-Za-z0-9_.-~] — including chars like ',' and "'" that are
sub-delims per RFC 3986 and valid in URL userinfo without encoding.

Resolution: just pass the raw password in --db-url. cf-app-drupal's
GenerateSecretString ExcludeCharacters already excludes every character
that would actually need URL encoding (@ : / ? # % [ ] etc.). The
remaining permitted characters are all valid unreserved or sub-delim
chars in URL userinfo per RFC 3986. No encoding needed.

The original 5-15 "password mangling" justification for adding URL
encoding was retrospectively wrong — the real 5-15 issue was operational
ordering (cf-app-drupal GenerateSecretString rotated value while
postgres still had old password). URL encoding never fixed anything;
it just added a new bug that took ~30 min of diagnostic to find.
```
