## On the backup/config-protection idea

Your instinct is dead right, and I think you're onto something worth doing. Let me riff on it.

### The state actually worth protecting

For the jumpbox specifically:
- **`/root/.ssh/id_ed25519` + `.pub`** — the SSH key React2 trusts. If we lose this, we're back to manually pasting a new pubkey into `~ubuntu/.ssh/authorized_keys` on React2.
- **`/home/ubuntu/.ssh/`** — same concern if we use ubuntu-user SSH
- **`~/.env` (either user)** — if we ever populate these with per-migration overrides
- **`~/bin/`** — if we ever put custom scripts there (you're right that we don't now, but might)

For the deploy-host, way more state matters (Drupal codebase on FSx, MariaDB scratch data, etc.), but FSx is on separate storage that survives instance replacement, so the operational concern is smaller there.

### S3 tar vs SSM Parameter Store — pick S3

You mentioned both. Real comparison:

| Aspect | S3 tar | SSM Parameter Store |
|--------|--------|---------------------|
| Cost | Bytes stored + puts (pennies/month) | **Free** for standard tier |
| Size limit | Effectively none | 4 KB per param (standard), 8 KB (advanced $$) |
| Structure | One tarball has everything | One param per value |
| Restore | Single `aws s3 cp | tar xzf -` | Loop over params, decode SecureString, write to file |
| Ergonomics | Trivial for arbitrary files | Trivial for individual keys |

For SSH keys alone, SSM works fine (ed25519 keys are ~400 bytes each). But once you add `bin/` folders with arbitrary scripts or want to preserve directory perms, S3 tar wins hands-down. And S3 is nearly-free at this size (a 100 KB nightly tar × 365 nights = 36 MB/year = $0.001/month).

**Verdict: S3 tar under a `config/` prefix.** Matches the pattern we already have.
