### 13:31:21 - Response

Good design instinct on keeping the keys out of the compute fleet — that preserves the "production traffic is SSM-only, no exceptions" boundary which is part of what makes the fleet testable as a production system.

**Recommended shape:**

1. **Storage**: SSM Parameter Store, single param `/worxco/admin/ssh-public-keys` as a StringList (newline-separated). Public keys aren't secret, but the *list* is sensitive (anyone whose key is added gets root), so the SSM IAM policy should grant write access only to admin roles and read access only to the specific instance roles that need it. Free, audited via CloudTrail.

2. **Install gating by stack type, not parameter**: Don't make the compute templates take an `InstallAdminSshKeys` parameter at all — that's a tripwire (someone passes `true` once and now sandbox has SSH that prod doesn't). Instead, only `cf-deploy-host` (and any future "operator" boxes) have UserData that reads the SSM path and writes `~/.ssh/authorized_keys`. The compute templates *physically can't* install the keys because their UserData doesn't reference the path. Security by exclusion is more robust than security by parameter defaults.

3. **Make targets**: `make admin-ssh-key-add NAME=alice FILE=alice.pub`, `-remove`, `-list`. Each commits the updated StringList to SSM with a CloudTrail-visible change.

**The non-trivial bit you should know about**: this also means opening **port 22 ingress** on the deploy-host SG (currently SSM-only, zero inbound). That's a real security choice, not just "copy some keys." Worth a CFN parameter `AdminSshSourceCidr` that defaults to a known admin VPN range or your IP — never `0.0.0.0/0`. If you don't want to pay that surface-area cost, **EC2 Instance Connect** gives you ephemeral SSH via the AWS console/CLI without ever opening a port — at the cost of operator UX (you can't `ssh deploy-host` from your terminal directly without the wrapper).
