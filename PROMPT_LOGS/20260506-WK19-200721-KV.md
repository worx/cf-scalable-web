### 20:08:01 - Response

Good question — by default no, the `ubuntu` user cannot write to `/etc`. Files there are root-owned and not writable by regular users.

Three ways to handle this, picking by tradeoff:

| Option | Pros | Cons |
|--------|------|------|
| **Sudo every refresh** | Standard /etc semantics — everyone respects "this is system config" | Annoying friction every time. `sudo refresh-env-config` |
| **`chown ubuntu:ubuntu /etc/worxco/envs/`** | No sudo for ubuntu user. Simple. | Slightly non-standard for `/etc` — but `/etc/worxco/` is yours anyway |
| **Move to `/var/lib/worxco/envs/`** | FHS-correct (`/var/lib` is for "variable state files that change over time") | Different path than what we discussed |

My recommendation: **option 2** — chown the `envs/` subdirectory specifically to `ubuntu:ubuntu`. Reasons:

- The `envs/` files are **regenerated**, not hand-edited system config. Their nature is closer to user state than to system config.
- The deploy-host is single-user-purpose. The "ubuntu" account is effectively the admin. Treating `/etc/worxco/envs/` as their territory makes sense.
- The marker file at `/etc/worxco/deploy-host-marker` stays root-owned (it's true system config, written once at boot, read-only thereafter). Only the `envs/` subdir gets relaxed permissions.
- No sudo means: `refresh-env-config sandbox` just works. Lazy-friendly.

In UserData this becomes a single line:
```bash
mkdir -p /etc/worxco/envs
chown -R ubuntu:ubuntu /etc/worxco/envs
```

The marker file at `/etc/worxco/deploy-host-marker` (already in Phase B) keeps its root ownership — it's untouched.
