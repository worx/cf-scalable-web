The error is telling you exactly what's wrong, but it's worded in a way that hides the actual cause. Let me decode it.

## What the error actually means

Look at the last three words: **`no encryption`**.

RDS PostgreSQL in this project has `rds.force_ssl = 1` in its parameter group (which is the correct security posture — it enforces encrypted-only connections). When a client connects **without SSL**, RDS synthesizes a `pg_hba.conf` on the fly that has *no matching entry for unencrypted connections*, so it rejects with that misleading "no pg_hba.conf entry" message.

The line "no encryption" at the end is the tell. If SSL had been used, the message would have ended with something like `SSL on` or `SSL off, cipher…`.

Translation: **"you tried to connect without TLS, and I don't accept that."**

Nothing to do with `pg_hba.conf` in the traditional sense — RDS doesn't let you edit that file anyway.

## The one-line fix in `scripts/tt.sh`

pgloader takes SSL options as query parameters on the PostgreSQL connection URL. Change this:

```
INTO      postgresql://${DRUPAL_DB_USER}:${SANDBOX_PW_ENC}@${DRUPAL_DB_HOST}:${DRUPAL_DB_PORT}/${DRUPAL_DB_NAME}
```

to this:

```
INTO      postgresql://${DRUPAL_DB_USER}:${SANDBOX_PW_ENC}@${DRUPAL_DB_HOST}:${DRUPAL_DB_PORT}/${DRUPAL_DB_NAME}?sslmode=require
```

Just add `?sslmode=require` at the end of the INTO URL.

Do you want me to make that edit to `scripts/tt.sh`? And once we get past this, we should also add `?sslmode=require` to the pgloader section of `docs/MIGRATION.md` so future-you doesn't hit the same wall.
