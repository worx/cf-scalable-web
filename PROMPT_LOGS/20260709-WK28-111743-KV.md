## The `?:` fallback pattern

Look at each line in the `$databases` block:

```php
'host'     => getenv('DRUPAL_DB_HOST') ?: 'localhost',
'port'     => getenv('DRUPAL_DB_PORT') ?: '5432',
'database' => getenv('DRUPAL_DB_NAME') ?: 'drupal',
```

The `?:` is PHP's "short ternary" — it reads:

> **"If `DRUPAL_DB_HOST` env var is set and non-empty, use that. Otherwise fall back to `'localhost'`."**

So it's not saying "connect to localhost." It's saying "connect to `$DRUPAL_DB_HOST`, and only fall back to localhost as a last-resort default if the env var is missing."

On the actual application servers (NGINX/PHP-FPM instances), those env vars come from `/etc/worxco/envs/sandbox` (sourced into the PHP-FPM pool at startup). So Drupal on those servers connects to your RDS endpoint. The `'localhost'` string is just belt-and-suspenders code — dead code path in normal operation.

If you want to prove it to yourself, source the env file and echo:

```bash
source /etc/worxco/envs/sandbox
echo "DB host env: $DRUPAL_DB_HOST"
```

You should see your RDS endpoint. That's what Drupal actually gets.

## Cache rebuild after the edit

After you save settings.php, Drupal needs to re-inspect the database schema. There's no drush on deploy-host normally — drush runs on the app instances. The easiest way is via SSM into one of the NGINX or PHP instances:

```bash
# From your Mac or deploy-host
make ssm-php  # or similar target — depends on your project's Makefile
# Once on the instance:
cd /var/www/drupal
drush cr
```

I don't recall if you have a specific `make` target for shelling into a PHP instance — check `make help`. If not, an SSM start-session against any PHP-FPM instance ID works.
