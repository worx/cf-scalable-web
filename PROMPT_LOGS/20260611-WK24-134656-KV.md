### 13:48:54 - Response

The `nc -vz` timing out is exactly what you want to see — confirms prod RDS isn't reachable from anywhere outside its VPC. Good defense posture.

For cross-account access without poking holes in prod, **AWS SSM Session Manager is the right tool**. It gives you:
- Cross-account access using your `ZoningInfoAdmin` profile, no VPC peering
- No SG changes on prod (RDS SG already allows the prod EC2 instances; SSM gives you a hop through one)
- Encrypted in transit, auditable in CloudTrail
- Tear-down is just closing the session — no persistent attack surface

**Two channels you need, both via SSM:**

| What | How | From |
|------|-----|------|
| MySQL connection to prod RDS | SSM port-forward via a prod EC2 to `d8zdb-1.cluster-ro-...:3306` → localhost:3306 on deploy-host | deploy-host |
| SSH for rsync of Drupal codebase | SSM SSH session to a prod Drupal server | deploy-host |

Both require **at least one prod EC2 instance that has SSM Agent + an IAM role with `AmazonSSMManagedInstanceCore` AND can reach the RDS / has the Drupal codebase**. Almost certainly true since you mentioned this is a working production site.

## Recommended path forward

**Step 1: Verify SSM-eligible prod targets.** From your Mac:

```bash
aws --profile ZoningInfoAdmin ssm describe-instance-information \
  --query 'InstanceInformationList[].[InstanceId,PingStatus,PlatformName,IPAddress]' \
  --output table
```

You want at least one `Online` instance. If they don't show up, prod EC2s either don't have SSM Agent or their IAM role lacks SSM permissions — that's a precondition we'd need to address first.

**Step 2: Set up the `ZoningInfoAdmin` profile on the deploy-host.** Either:
- Copy your `~/.aws/config` + `~/.aws/credentials` snippets for that profile to the deploy-host's `ubuntu` user, OR
- Better (longer-term): set up cross-account role assumption — the deploy-host's existing instance role assumes a role in the prod account via prod's IAM trust policy. Cleaner but takes some IAM work in prod first.

For one-off migration testing, just copying the profile is fine.

**Step 3: Pick the migration topology — I'd recommend "dump to file, ship via S3."**

Two ways to do the actual data move:

| Approach | Pros | Cons |
|----------|------|------|
| **Live MySQL→PostgreSQL via pgloader over SSM tunnel** | One command; pgloader's Drupal-aware preset handles schema translation | Tunnel needs to stay up for the full dump+load duration; if it drops, restart from scratch |
| **mysqldump on prod → S3 → pgloader on file on deploy-host** | Each step is independent; mysqldump on prod is fast and local; pgloader works against the file with retries | Two steps instead of one |

I'd vote **#2 (dump to file, ship via S3)** for the first cut — more robust to iteration, and you can keep the dump around for repeated pgloader experiments without re-fetching from prod each time.

**Step 4: For the codebase rsync** — SSM session direct to a prod Drupal server, run `tar` of the Drupal docroot to S3 (using the prod EC2's IAM permissions, or via stdin/stdout through the SSM session if perms are tight). Then pull from S3 on the deploy-host side.
