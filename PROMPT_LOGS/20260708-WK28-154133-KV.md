### 15:42:03 - Response

Not a dumb question at all — it's exactly the right question to ask before running a data migration. Let me decode what pgloader is doing.

## Look at your WITH clause

```
WITH include drop, create tables, create indexes, reset sequences, foreign keys,
     ...
```

The first item — **`include drop`** — is your answer. That instructs pgloader to **`DROP TABLE ... CASCADE`** on every target table before recreating it. So for each table that exists in the source MySQL, pgloader does:

1. `DROP TABLE IF EXISTS <table> CASCADE` (nuking any existing rows + indexes + FKs)
2. `CREATE TABLE <table> (...)` fresh from the MySQL schema
3. `COPY` data in
4. `CREATE INDEX`es
5. `ALTER TABLE ADD CONSTRAINT ... FOREIGN KEY ...`
6. `SELECT setval(...)` to reset any sequences to `MAX(id) + 1`

So it's **wipe-and-replace per table**, not merge. Every row you see in PostgreSQL after this run comes from your MySQL source — nothing from the fresh Drupal install survives inside those tables.

## What pgloader does NOT touch

- The `drupal` **database** itself — stays
- The `drupal_user` role and its permissions — stays
- Any PostgreSQL **schema** (namespace) — stays
- Any **table that exists in PostgreSQL but NOT in the source MySQL** — untouched

That last one is important and answers the "why did we run `install-drupal-full` first" question: it wasn't to put data in — it was to create `drupal_user` (a *role*, not a table). `install-drupal-full` also DID create all the Drupal tables with default install data as a side effect, but pgloader is dropping every one of them right now and replacing with your real production data. That side-effect data is gone the moment `include drop` fires on each table.

## So — nothing to worry about

You don't need to manually clear anything. The current tt.sh does exactly what you'd want: nukes any Drupal-install placeholder tables, recreates them from Aurora's schema, streams the data in.

If you wanted a "safety" checkpoint version, you *could* do a manual `psql \dt` before and after to compare table counts, but it's not required. The `include drop` semantics make this a deterministic full replacement.
