### 12:46:49 - Response

Good moment for this — let me push back on a couple of the assumptions and answer the concrete questions.

**Single-zone for test: I'd push back.**

The cost savings are smaller than they look. FSx OpenZFS Multi-AZ runs ~$0.36/GB-mo vs $0.18 Single-AZ; on a 100GB volume that's $18/mo. RDS db.t4g.micro Multi-AZ is ~$26 vs $13 Single-AZ. Combined: maybe ~$30/mo savings.

The cost you give up is the rehearsal value, which is the whole *point* of test as a prod-migration staging ground:
- Multi-AZ RDS failover behavior (connection string handling, replica lag)
- FSx Multi-AZ failover (OpenZFS Multi-AZ has specific failover semantics worth seeing in test before prod)
- ALB cross-AZ routing
- ASG distribution math (one-AZ ASG behaves differently than two-AZ at scale events)

The compute cost dominates anyway — and that's where ASG sizing + pause-compute does the real work. My recommendation: dual-AZ everywhere matching prod, **ASG desired=1 daytime** (one nginx, one php74, one php83) instead of 2-of-each. You get ~50% compute savings during business hours, full shutdown overnight, and your topology matches prod.

**What can actually be stopped vs not:**

| Component | Stoppable? | Behavior |
|-----------|-----------|----------|
| ASGs (NGINX/PHP) | ✅ | `pause-compute` (already exists) sets desired=0; resume sets back |
| RDS | ✅ | `aws rds stop-db-instance` — **7-day max**, AWS auto-starts after. Multi-AZ DB *instances* can stop; Multi-AZ DB *clusters* cannot. |
| ElastiCache Valkey | ❌ | No stop API. Only destroy/recreate. |
| FSx OpenZFS | ❌ | No stop API. Pay 24/7. Data persists indefinitely — no loss risk. |
| ALB | ✅ (sort of) | Can delete; you pay ~$16/mo otherwise. Recreating changes DNS though. |

So a realistic "stop-test" sequence:

1. `pause-compute` → ASGs to 0
2. `aws rds stop-db-instance` (script handles the 7-day re-poke if needed)
3. Optionally `destroy-cache` (Valkey only takes ~3 min to recreate, cold cache is fine on a test env)
4. Leave FSx + ALB running

Overnight cost drops to ≈ FSx + ALB + (cache if not destroyed). For a 100GB FSx that's roughly $20 + $16 + $15 = ~$50/mo overnight baseline vs ~$200+/mo full-running.

**Your specific questions:**

> "Does FSx survive stop/restart without losing data?"

FSx OpenZFS data is durable — there's no "stop" but also no risk. Snapshots are separate and you can take them on a schedule.

> "Can we offload cache before stopping it so we restart warm?"

Yes, sort of. Valkey supports `create-snapshot` → S3, then on recreate you `--snapshot-name` to restore. Drupal handles cold caches fine (first few page loads slower, then rebuilt) so I'd skip the snapshot-and-restore choreography for test. For prod, that's the right pattern.

**MySQL → PostgreSQL:**

I'd strongly endorse doing the migration now rather than punting to MySQL. Two reasons: (a) the rehearsal value is exactly the same as the Multi-AZ argument — you want test to teach you the migration before prod cutover, not insulate you from it; (b) running test on a different engine than prod is *less* useful than running test on a different engine than test's own current trajectory, which is what migration-rehearsal is for.

Toolchain choice: **pgloader from the deploy-host.** It's open-source, fast (1-2 minutes for typical Drupal DBs), has Drupal-aware presets, and runs as a single command:

```bash
pgloader mysql://user:pass@prod-mysql/drupal postgres://drupal_user:pass@test-rds/drupal
```

It handles the Drupal schema gotchas (LONGTEXT → TEXT, TINYINT(1) → BOOLEAN, sequences for auto-increment, etc.). The deploy-host can run it against prod-MySQL (over your existing peering or a one-off VPN/tunnel) and write directly into test-RDS-Postgres. After load, one `drush cache:rebuild` and you're in business.

The alternative is AWS DMS, which is more infrastructure for a one-shot job. pgloader is the right tool here.
