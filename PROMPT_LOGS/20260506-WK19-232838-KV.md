### 23:28:59 - Response

Yes — destroy/deploy of the deploy-host gives you a clean slate with UserData running from scratch. That's actually the cleaner path because it exercises the same flow a brand-new setup would use.

**The wrinkle**: the peering stack imports cf-deploy-host's SG via cross-stack reference, so CFN refuses to destroy deploy-host while peering is using it. The sequence has to be:

```bash
# From your Mac (NOT from the deploy-host shell — your session dies)
make destroy-peering ENV=sandbox
make destroy-deploy-host
make deploy-deploy-host
make deploy-peering ENV=sandbox
```

Each step is fast (~1-2 min for the destroys, ~2-3 min for the deploys, total ~10 min).

When the new deploy-host comes up:
- UserData runs end-to-end: installs all Phase B tools, copies helpers from repo, runs `refresh-env-config` once → cached config files written, FSx /etc/hosts entry created
- New SG ID gets exported, peering picks it up cleanly

Nothing important lives only on the deploy-host:
- Repo is auto-cloned from GitHub at boot
- No persistent state worth saving
- FSx, RDS, Valkey are untouched (different stack lifecycle)

After the four commands finish, SSM in and you should see everything working without any manual install steps. Run the verification checklist again to confirm.

Want me to also build out `scripts/deploy-host/bootstrap.sh` for the future "I changed a script and want to refresh without redeploying" workflow? Different tool from destroy/deploy — useful for quick iteration. Optional.
