## FSx Mount Point Summary

**Mount point**: `/var/www` on both NGINX and PHP-FPM instances

**FSx path**: `{fsx_dns}:/fsx` → `/var/www`

**Where it's defined** (across the stack):

### 1. Storage Layer (`cf-storage.yaml`)
- Creates the FSx OpenZFS file system
- Stores the DNS name in SSM: `/${ENV}/fsx/dns-name`
- Exports a convenience mount command: `mount -t nfs -o nfsvers=4.1 ${FSxDNSName}:/fsx /var/www`

### 2. VPC Layer (`cf-vpc.yaml`)
- Security group egress rules allow both NGINX and PHP-FPM to reach FSx on NFS ports (2049, 111, 20001-20003)

### 3. IAM Layer (`cf-iam.yaml`)
- Both NGINX and PHP-FPM roles have `fsx:DescribeFileSystems` and `fsx:DescribeVolumes` permissions

### 4. Image Builder (`cf-image-builder.yaml`) — This is where the actual mount happens

**At AMI build time:**
- Creates `/var/www` directory, owned by `www-data:www-data`
- Writes commented-out fstab templates as placeholders

**At instance boot time (configure scripts):**

**NGINX** gets two mounts:
- `$FSX_DNS:/fsx` → `/var/www` (webroot — shared with PHP)
- `$FSX_DNS:/fsx/nginx` → `/etc/nginx/shared` (shared NGINX site configs)

**PHP-FPM** gets one mount:
- `$FSX_DNS:/fsx` → `/var/www` (webroot — same as NGINX)

Both boot scripts:
1. Read `/$ENV/fsx/dns-name` from SSM Parameter Store
2. If the value exists and isn't "None", mount via NFS4 with performance-tuned options (`rsize=1048576,wsize=1048576,hard,timeo=600,retrans=2,noresvport,noatime`)
3. Gracefully warn but don't fail if mount fails

### 5. Compute Layer (`cf-compute-php.yaml`, `cf-compute-nginx.yaml`)
- UserData calls the configure scripts, which handle the mounting

**Key design point**: The mount is done at boot time via the configure scripts, not baked into the AMI. This means instances work without FSx (they just have an empty local `/var/www`), which aligns with the runtime-dependency pattern we discussed.
