### 13:05:54 - Response

Really good question — and an important source of confusion worth clearing up before we build another layer on top. You're partially right, but there's a subtle distinction to nail down.

## The mental model: host vs account

**The AWS CLI is a network client.** When you run `aws --profile X <command>`, three things are happening:

1. The CLI process runs **on whatever host you invoke it from** (your Mac, deploy-host, whatever)
2. It reads credentials for profile `X` from `~/.aws/credentials` on that host
3. Those credentials determine **which AWS account the API call authenticates against**

So **host ≠ account.** The physical machine is largely irrelevant to CFN operations — what matters is whether the credentials on that machine have the right permissions.

## What that means for these specific commands

| Command | Where it runs from | AWS profile used | What account it touches |
|---------|-------------------|------------------|-------------------------|
| `make deploy-migration-bucket` | **Your Mac** (or deploy-host) | `ZI-Sandbox` | Sandbox |
| `make deploy-migration-secret` | **Your Mac** (or deploy-host with the profile) | `ZoningInfoAdmin` | Prod |
| `make set-migration-secret-password` | **Your Mac** (or deploy-host with the profile) | `ZoningInfoAdmin` | Prod |
| `make deploy-migration-jumpbox` | **Your Mac** (or deploy-host with the profile) | `ZoningInfoAdmin` | Prod |

**Notice: none of these run on the jumpbox itself.** The jumpbox is a runtime worker, not an admin console. Its IAM instance role only grants three things:
- SSM management (for you to shell in)
- Cross-account S3 write to the sandbox migration bucket
- `GetSecretValue` on the specific migration secret ARN we just added

It **cannot**:
- Create or modify CloudFormation stacks
- Create or update Secrets Manager entries
- Access anything outside those scoped permissions

That's intentional — least-privilege. The jumpbox is a data-mover, not a control plane.

## When we DO care what host things run from

The scripts we'll build in Step 2+ (the `scripts/migration/jumpbox/*.sh` files) will run **on the jumpbox** because they:
- Need SSH access to React2 (only the jumpbox has that path)
- Rely on the jumpbox's instance role for `GetSecretValue` and S3 write
- Stream data through the jumpbox's network path

Contrast with `scripts/migration/deploy-host/*.sh` which will run **on deploy-host** because they:
- Talk to the local MariaDB scratch instance
- Access FSx-mounted files
- Sit inside the sandbox VPC for RDS access

So the split is:
- **Makefile-orchestrated CFN operations** → run wherever you have the right profile (typically Mac)
- **Actual data-mover scripts** → shipped to and executed on the right host by `aws ssm send-command`
