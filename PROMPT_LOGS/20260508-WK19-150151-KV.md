## Phase C step 3 plan

`cf-app-drupal.yaml` is a small CloudFormation stack that **formalizes** the Secrets Manager entries and SSM parameters that `install-drupal.sh` currently auto-creates on first run. Reasons to formalize:

1. **Lifecycle parity** — secrets and config managed via CloudFormation just like everything else (deploy / update / destroy)
2. **Resource policies** — explicit Secrets Manager rotation policy, SSM parameter tier, encryption keys
3. **Audit trail** — CFN events log the changes; today the install script silently auto-creates
4. **Tear-down clarity** — `make destroy-app-drupal ENV=sandbox` cleans up the stack-owned secrets/params, while preserving the FSx files for "remove install but keep history" scenarios
5. **Dependency declaration** — the install can later refuse to run unless the stack exists, signalling "infra prerequisites not met"

The stack will own:
- `worxco/${env}/drupal/admin-password` — generated at create time, rotation-ready
- `worxco/${env}/drupal/db-password` — generated at create time
- `/${env}/drupal/db-name`, `/${env}/drupal/db-user`, `/${env}/drupal/site-name`, `/${env}/drupal/install-path` — SSM parameters

After the stack creates the secrets/params, `install-drupal.sh` will read them as inputs (rather than auto-creating). Existing installs aren't disturbed — the script will detect "secret already exists" and use it.

Make targets:
- `make deploy-app-drupal ENV=sandbox` — create or update the stack
- `make destroy-app-drupal ENV=sandbox` — tear down the secrets/params (with the same CONFIRMED=yes pattern)
- `make verify-app-drupal ENV=sandbox` — list what the stack owns
