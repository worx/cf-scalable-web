fix(install-drupal): composer needs HOME + COMPOSER_ALLOW_SUPERUSER under SSM

Phase 7 of deploy-allX failed today on `composer create-project` because
AWS-RunShellScript dispatches scripts as root with a minimal environment
(no HOME, no shell profile). Composer refuses to run without HOME set
and refuses to run as root without explicit COMPOSER_ALLOW_SUPERUSER=1.

The 2026-05-15 install worked because Kurt was in an interactive SSM
Session Manager shell (HOME=/home/ubuntu, composer ran as ubuntu, plugins
allowed). The non-interactive SSM dispatch hits both guard rails.

Three fixes in scripts/deploy-host/install-drupal.sh:

1. Set HOME and COMPOSER_ALLOW_SUPERUSER at the top, before any
   composer invocation. Both behave identically whether running as
   ubuntu (interactive) or root (SSM dispatch).

2. Wipe $INSTALL_DIR contents before composer create-project if the
   directory exists with files but no .installed marker. Handles the
   partial-install-retry case where composer crashed mid-flight — its
   `create-project` refuses to write into a non-empty target. The
   $PRIVATE_DIR salt.txt is intentionally preserved (the later
   "Generate (or reuse) hash_salt" step is already idempotent).

3. Drive-by cleanup in Makefile: removed the legacy inline AMI polling
   loop from deploy-all Phase 2. `build-amis` has been blocking since
   commit d47a0d4 (the wait-amis addition); the inline loop ran a
   second time, saw everything was already AVAILABLE, and printed
   confusing "Waiting 20-30 minutes" output AFTER the wait already
   completed. ~75 lines removed. To get the old fire-and-forget
   behavior, use the `build-amis-async` target directly.
