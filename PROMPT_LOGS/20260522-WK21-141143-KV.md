### 14:14:00 - Response

**What happened:**

The Drupal install itself **succeeded** — the line `✓ install-drupal-remote: Drupal installed` confirms that. The failure was in the *post-install* step that reloads nginx to pick up the new vhost. Root cause is a race condition between ASG instance churn and stale instance-ID enumeration.

**Timeline (from ASG scaling activities):**

| UTC | Event |
|-----|-------|
| 17:28:30 | ASG scaled 0→2. Original instances `i-073ebc9f54074ee59` and `i-095a54c501a813daa` launched. |
| 17:34:37 | ELB health check failure → first original terminated. |
| 17:34:39 | Replacement `i-02130b4aa814a9c1f` launched (still alive today). |
| 17:38:48 | ELB health check failure → second original terminated. |
| 17:38:49 | Replacement `i-0b59bdd31af2fa1a7` launched (still alive today). |
| ~17:45+ | `install-drupal.sh` finished, wrote `drupal.conf` to FSx, then `reload-nginx.sh` ran. |

The ELB health failures during the install window are an unrelated chicken-and-egg (nginx 502s because Drupal isn't installed yet, ASG replaces nginx), but the install *did* eventually finish, and the replacement instances are healthy now.

**The bug in `scripts/reload-nginx.sh`:**

It enumerates targets via `aws ec2 describe-instances --filters tag:aws:autoscaling:groupName=...,instance-state-name=running`. That filter returned three instance IDs:
- `i-02130b4aa814a9c1f` (current, valid)
- `i-073ebc9f54074ee59` (already-terminated original — but tag index briefly stale)
- `i-095a54c501a813daa` (same)

SSM `send-command` got a list with 2/3 not-actually-running instances → `InvalidInstanceId` rejecting the whole batch. The authoritative source for "current ASG members" is `describe-auto-scaling-groups`, not a tag-filtered `describe-instances`.

**About your current state:**

The two nginx instances alive right now (`i-02130b4aa814a9c1f`, `i-0b59bdd31af2fa1a7`) launched **before** `install-drupal.sh` wrote `drupal.conf` to FSx, so they're running without the Drupal vhost loaded. They need a reload to serve Drupal.

**Drop-in to unstick *right now*:**

```bash
IDS=$(aws --profile ZI-Sandbox autoscaling describe-auto-scaling-groups \
  --auto-scaling-group-names sandbox-nginx-asg \
  --query 'AutoScalingGroups[0].Instances[?LifecycleState==`InService` && HealthStatus==`Healthy`].InstanceId' \
  --output text)
echo "Targeting: $IDS"
aws --profile ZI-Sandbox ssm send-command \
  --instance-ids $IDS \
  --document-name AWS-RunShellScript \
  --parameters 'commands=["sudo nginx -t","sudo nginx -s reload","systemctl is-active nginx"]'
```

After that, `make smoke-test-drupal ENV=sandbox` should pass.

**Want me to commit the fix to `reload-nginx.sh` now?** The change is one block — switch from tag-filtered `describe-instances` to `describe-auto-scaling-groups` with the `InService` + `Healthy` filter. Plus the same fix for any other places we enumerate ASG members via tag (there might be a few; I'd grep). That way the next deploy-allX won't trip on the same race.
