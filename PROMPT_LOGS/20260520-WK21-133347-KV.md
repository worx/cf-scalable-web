Three concrete findings:

**1. All Ubuntu, no Amazon Linux.** Confirmed at three spots:
- `cf-deploy-host.yaml:45` — deploy-host uses SSM param `.../ubuntu/server/24.04/stable/current/arm64/hvm/ebs-gp3/ami-id`
- `cf-image-builder.yaml:1278,1299,1320` — all three Image Builder pipelines (nginx, php74, php83) use `ubuntu-server-24-lts-arm64` as their ParentImage
- nginx + PHP install components reference `noble` (24.04's codename) in apt sources

**2. Ubuntu 26.04 LTS "Resolute" IS published on AWS.** Just 17 days ago, in fact — AMI `ami-053da03328707680f`, dated 2026-05-03. Canonical's description: *"Canonical, Ubuntu, 26.04, arm64 resolute image"*. 22.04 has aged out of the SSM stable channel; 24.04 last refreshed 5 days ago.

**3. My recommendation: hold off until 26.04.1 (~August 2026).** Reasons:

- 26.04 is 2-3 weeks old as a public AWS image. New-LTS-on-day-one is a known pain point — kernel regressions, glibc edge cases, distro-package version skew. Production-shy operators always wait for the `.1` (point) release where the dust has settled. That ships ~4 months after `.04`, so August.
- 24.04 is supported until 2029 free / 2034 with Ubuntu Pro. No urgency.
- If we DID move, the touchpoints are small but real:
  - `cf-deploy-host.yaml` SSM param path: `24.04` → `26.04` (1 line)
  - `cf-image-builder.yaml` ParentImage: `ubuntu-server-24-lts-arm64` → `ubuntu-server-26-lts-arm64` (3 lines, one per pipeline)
  - `noble` → `resolute` in nginx apt repo URL in two places
  - **Unknown risk**: nginx.org/packages/ubuntu may not have a `resolute` distribution mirror yet (they publish per-codename, can lag months behind new Ubuntu LTSes). If `resolute` isn't there, the nginx install during AMI bake will fail and we'd need a workaround (use Ubuntu's universe nginx instead, which is older but always available)
- PHP versions: php8.3 and php7.4 packages need to exist in the new Ubuntu's universe/ondrej PPA. Should be fine but worth verifying.

**Suggested next move if you want to explore:** spike it in a separate experimental env (`sandbox-26` or similar) without changing sandbox/production. We'd discover the nginx-repo + PHP-package issues without putting our working sandbox at risk. Add as a P3 TODO and revisit after 26.04.1 ships?
