## The Three Sections of a CloudFormation Template

Every template has three main sections that play different roles:

**Parameters** (lines 23-66) — Inputs. These are the knobs you turn before deploying. They don't create anything; they're just variables that get fed into the Resources section. Think of them like function arguments.

**Resources** (lines 72-261) — The actual AWS things being created. This is where AWS goes "OK, I'm making a file system, some buckets, some SSM parameters."

**Outputs** (lines 263-304) — Values published *after* everything is created. Two purposes: showing you useful info ("here's your mount command"), and exporting values so *other* CloudFormation stacks can reference them.

## Following the FSx DNS Name: Creation → Storage → Export

### Step 1: The File System Gets Created (line 77)

```yaml
FSxFileSystem:
  Type: AWS::FSx::FileSystem
  Properties:
    FileSystemType: OPENZFS
    StorageCapacity: !Ref FSxStorageCapacityGB
    ...
```

`FSxFileSystem` is a **logical name** — it only exists inside this template as a handle to refer to the resource. AWS doesn't see this name. AWS creates the file system and assigns it a real ID (like `fs-0c1e63cb3dbdeaa0b`) and a DNS name (like `fs-0c1e63cb3dbdeaa0b.fsx.us-east-1.amazonaws.com`).

### Step 2: `!GetAtt` — Asking AWS "What Did You Create?"

This is the key concept you were asking about. When CloudFormation creates a resource, that resource has **attributes** — properties that only exist *after* creation. You can't know the DNS name ahead of time; AWS assigns it.

`!GetAtt` means "get an attribute from a resource that was just created."

```yaml
!GetAtt FSxFileSystem.DNSName
```

Translation: "Go to the resource I called `FSxFileSystem`, and give me its `DNSName` attribute."

Every AWS resource type has documented attributes. For `AWS::FSx::FileSystem`, the available attributes include:
- `DNSName` — the hostname for NFS mounting
- `RootVolumeId` — the ID of the root volume

You'll see `!GetAtt` used three times for the DNS name in this template, all reading the same value from the same resource. They're not three different things — they're three places that need the same piece of information.

### Step 3: SSM Parameter — Storing It for Runtime Discovery (line 213)

```yaml
FSxDNSParameter:
  Type: AWS::SSM::Parameter
  Properties:
    Name: !Sub '/${EnvironmentName}/fsx/dns-name'
    Value: !GetAtt FSxFileSystem.DNSName
```

This creates an entry in **SSM Parameter Store** at the path `/sandbox/fsx/dns-name` with the DNS name as its value. This is what your PHP and NGINX instances read at boot time — they call `aws ssm get-parameter --name "/sandbox/fsx/dns-name"` and get back the DNS hostname to mount.

**Why not just use the Outputs?** Because EC2 instances can't easily query CloudFormation exports at boot time. SSM Parameter Store is a simple key-value API that any instance with IAM permissions can call. It's the bridge between "CloudFormation knows this value" and "a running instance needs this value."

### Step 4: Outputs — Publishing for Other Stacks and Humans (line 270)

```yaml
FSxDNSName:
  Description: FSx DNS name for NFS mounting
  Value: !GetAtt FSxFileSystem.DNSName
  Export:
    Name: !Sub '${EnvironmentName}-fsx-dns-name'
```

This does two things:
1. **Shows the value** in the CloudFormation console and CLI output (the `Description` + `Value`)
2. **Exports it** with a cross-stack name (`sandbox-fsx-dns-name`) so other CloudFormation templates can use `Fn::ImportValue` to reference it

The SSM Parameter (step 3) is for **instances at runtime**. The Export is for **other CloudFormation stacks at deploy time**.
