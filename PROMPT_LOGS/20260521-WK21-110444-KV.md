fix(install-drupal): tolerate update module already-uninstalled (Drupal 11.3.10 default)

Yesterday's commit added a final `drush pm:uninstall update -y` step
to silence the "can't contact updates.drupal.org" admin warnings (per
docs/memory/drupal-update-management.md — the module is architecturally
unreachable from our privatized compute fleet, so its UI lies about
having capabilities it doesn't).

But: Drupal 11.3.10's standard install profile no longer enables the
update module by default. Yesterday's tests used 11.3.9 which DID
enable it; today's install hit 11.3.10 which doesn't. drush pm:uninstall
on an already-not-installed module exits 1 with:

  The following module(s) are not installed: update. No modules to
  uninstall.

set -e in install-drupal.sh dutifully killed the install there, after
the rest of the install pipeline had completed successfully. The
already-desired state was being treated as failure.

Fix: capture drush's exit code, treat the "are not installed" message
as a successful idempotent no-op. Real drush failures (unexpected
exit codes with other error messages) still fail the install loudly.

Two-way compatible: works on the Drupal 11.3.9-and-earlier line (where
update IS enabled — uninstall proceeds normally) AND on 11.3.10+
(where update is already not enabled — log idempotent and continue).
Defensive against a hypothetical future Drupal that re-enables the
module too.
