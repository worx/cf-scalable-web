```
fix(boot): use systemctl restart (not start) after mutating service config

The boot scripts (configure-php.sh and configure-nginx.sh) end with
`systemctl start <service>` — which is a NO-OP when the unit is already
running. Both php-fpm and nginx are systemd-enabled, so they auto-start
at boot BEFORE cloud-init's scripts_user phase runs the boot scripts.
By the time the boot script writes its marker block into www.conf (PHP)
or mounts /etc/nginx/shared from FSx (nginx), the daemon is already up
with the AMI-baked config — and never re-reads the file we just changed.

Symptom (PHP): the on-disk www.conf has env[DRUPAL_DB_HOST] etc., but
workers serve with an empty environment. settings.php's getenv() calls
all return empty, trusted_host_patterns never gets DRUPAL_SITE_NAME,
Drupal returns HTTP 400 "host name not valid". This worked Friday only
because we kept manually restarting FPM during debugging; new boxes
that have only ever done `start` are broken from first request.

Symptom (nginx): the FSx-mounted vhosts in /etc/nginx/shared/sites-enabled/
are invisible to the running nginx — it serves only the AMI-baked health
check default server.

Fix: switch `start` to `restart` in both boot scripts. Bump RecipeVersion
1.0.9 -> 1.0.10 so the next AMI build bakes in the fix (configure-php.sh
also flows via S3 sync). Document the gotcha.

For the running fleet: a manual `sudo systemctl restart php8.3-fpm` on
each PHP box made everything work immediately (no AMI rebuild required
to recover existing boxes). The AMI rebuild is only needed so fresh
instance launches don't fall into the same trap.

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
```
