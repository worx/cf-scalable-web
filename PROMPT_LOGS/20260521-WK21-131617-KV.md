### 13:17:25 - Response

Good thinking — DNS gives the site real URLs you can hand to other people without explaining the ALB DNS soup. A few angles worth deciding before I start wiring anything:

**Cross-account model — two reasonable approaches:**

**A. Direct cross-account writes.** A make target runs `aws route53 ... --profile zikvanderw` against the parent account's hosted zone. Simple, fits the existing pattern. Cost: every operator needs `zikvanderw` credentials locally, and every `deploy-allX` / `destroy-all` carries a step that depends on parent-account access.

**B. Sub-zone delegation.** Once-in-parent-account work: create a `sandbox.zoning-info.com` (and `test-new.zoning-info.com`) hosted zone in ZI-Sandbox, then add NS delegation records in the parent zone pointing each sub-zone at its ZI-Sandbox NS servers. After that, all day-to-day record management happens entirely within ZI-Sandbox with the normal profile. ALB recreation, sandbox destroy/redeploy cycles, future test-new — none of it ever touches the parent account again.

I'd lean toward **B**. The setup is one-time per env, but every subsequent operation (and there will be many — ALBs come and go on `destroy-all` cycles) stays within ZI-Sandbox. Cleaner separation, aligns with AWS Org best practices, and the `zikvanderw` profile only needs to exist on whoever does the one-time setup. The "parent zone is sacred infrastructure, child zones are application-managed" pattern.

The downside of B is *only* the one-time delegation. If you'd rather not touch the parent zone twice (once for sandbox, once for test-new), there's a middle path: delegate a single `envs.zoning-info.com` sub-zone to ZI-Sandbox, then your URLs become `sandbox.envs.zoning-info.com` / `test-new.envs.zoning-info.com`. Less natural URLs, but only one parent-account touch ever. I'd skip that — two delegations is fine, and the cleaner URLs matter.

**Naming:** `test-new` is fine. `test2` reads like "we couldn't figure out version control." `test-new` at least signals "this is the not-the-old-one version." Bonus: when the eventual cleanup happens and old `test` retires, you could rename to plain `test`, but renaming a zone is annoying — just live with `test-new`.

**Things that change downstream of the DNS:**

1. **Drupal trusted_host_patterns** — settings.php's pattern list is currently `^.+\.elb\.amazonaws\.com$` (and the literal `DRUPAL_SITE_NAME` if set). The new hostname needs to be in there. Easiest: set `DRUPAL_SITE_NAME=sandbox.zoning-info.com` in the env config (`/etc/worxco/envs/sandbox`). install-drupal.sh already escapes it into the pattern array.

2. **Nginx vhost** — listens on `_default_server` so the Host header doesn't gate it. No change needed there.

3. **smoke-test-drupal** — currently curls the ALB with the Drupal Host header. Should curl `sandbox.zoning-info.com` once DNS is up. Maybe parameterize: `make smoke-test-drupal HOST=sandbox.zoning-info.com`.

4. **HTTPS** — flagging this now: a real subdomain on a real org domain *will* attract people pointing browsers at `https://sandbox.zoning-info.com` and getting cert errors. Phase 3 (ACM + ALB HTTPS listener) is officially "later" per docs, but a public-looking URL changes the social pressure on that timeline. Worth deciding whether `sandbox.` going live triggers a Phase 3 bump-up. Even a self-issued cert would beat a NET::ERR_CERT_AUTHORITY_INVALID.

5. **make destroy-all** — if we publish DNS, destroy-all needs to also unpublish, otherwise the next deploy creates a new ALB and the old DNS points at a tombstone. Trivial addition.
