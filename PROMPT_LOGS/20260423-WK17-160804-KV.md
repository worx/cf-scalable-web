There it is — no `AWS_DEFAULT_REGION`. The CLI has credentials from the role but doesn't know what region to talk to. Try this on the box right now:

```bash
AWS_DEFAULT_REGION=us-east-1 aws sts get-caller-identity
```

That should return the instance role ARN, proving credentials work without any keys.

## 2. Public IP

You're right that SSM doesn't need a public IP. **But** the default VPC has no NAT gateway — outbound internet (apt-get, git clone, npm install, AWS API calls) routes through the IGW, which requires a public IP for the return path. Without the public IP, the instance can't reach anything.

Options:
- **Keep public IP, remove SSH port entirely** — SSM-only access, nothing listening on the public IP. This is the cheapest and simplest hardening.
- **Add NAT gateway** — Allows private-only IP with outbound internet, but costs ~$32/month. Overkill for one instance.

I'd recommend option 1: keep the public IP for outbound, but close port 22 entirely since SSM is the primary access method. The public IP with no open inbound ports is just a return address for outbound traffic — not a security risk.

## 3. Client Access

Since the instance role provides credentials via IMDS, anyone who SSMs into the box automatically inherits `AdministratorAccess` to that AWS account. No keys to share, no keys to rotate, no keys to accidentally commit to git. The role is tied to the instance, not to a person.

Want me to fix the template now? The changes would be:
1. Add `AWS_DEFAULT_REGION=us-east-1` to the environment profile
2. Remove the SSH security group rule (SSM only)
3. Remove the key pair creation (no SSH = no need for keys)
