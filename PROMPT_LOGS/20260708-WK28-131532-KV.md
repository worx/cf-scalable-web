## What "76 fell through ECASE" means

`ECASE` is Common Lisp's "switch statement that errors if no case matches." qmynd is walking the source database schema, encountered a **collation ID of 76 on some column**, and doesn't have that ID in its lookup table. The list of "Wanted one of" values is qmynd's supported collation IDs — and 76 is conspicuously absent (67-72 are there, then it jumps to 77).

qmynd was written against an older MySQL collation table and hasn't been updated for some of the newer or less common collation IDs that Aurora MySQL 8 uses.

## Diagnostic — find out what 76 is and where it's used

```bash
sudo mysql zinew <<'SQL'
-- What is collation ID 76?
SELECT id, collation_name, character_set_name, is_default
  FROM information_schema.collations
 WHERE id = 76;

-- Which columns in zinew use it?
SELECT table_name, column_name, character_set_name, collation_name
  FROM information_schema.columns
 WHERE table_schema = 'zinew'
   AND collation_name IN (
     SELECT collation_name
       FROM information_schema.collations
      WHERE id = 76
   );
SQL
```

Paste the output — that'll tell us:
- What collation `76` actually is (probably some flavor of `utf8mb3_*` or an older regional collation)
- Which tables/columns are using it (likely just a few, unless somebody deliberately set the DB default to it)

## Anticipating the fix

Once we know which columns are affected, we convert them to a collation qmynd *does* support — likely `utf8mb4_unicode_ci` (ID 224) which is Drupal's standard and definitely in qmynd's supported list.

For a small handful of columns, the fix looks like:

```sql
ALTER TABLE tablename
  MODIFY column_name VARCHAR(N)
  CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

For "the whole DB defaulted to it," we'd do a bulk conversion:

```sql
ALTER DATABASE zinew CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
-- Then per table:
ALTER TABLE tablename CONVERT TO CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

Since we did the mysqldump-then-restore approach, this collation surgery happens **only on your local scratch MySQL on deploy-host**. The prod Aurora MySQL is completely untouched. If we mangle a collation conversion, we just re-restore from `zinew.sql` on FSx and try again. Zero prod risk.
