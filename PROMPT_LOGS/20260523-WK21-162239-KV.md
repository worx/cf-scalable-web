Background on the spelling-confusion first: **ACME and ACM are entirely different things.**

- **ACME** = Automated Certificate Management Environment. An IETF protocol (RFC 8555) for automated cert issuance. **Let's Encrypt** is the most famous CA that speaks ACME. **CertBot** is the most popular ACME *client*. Together they're the open-source / non-AWS path most operators reach for.
- **ACM** = AWS Certificate Manager. An AWS service that issues certs from **Amazon Trust Services** (Amazon's own CA, with root certs in every modern browser's trust store). It does NOT use the ACME protocol — it's AWS-proprietary plumbing.

Both produce **publicly-trusted TLS certs that any browser will accept**. Issuer chain is different, but to the end user there's no visible difference between `Issued by: Let's Encrypt R3` and `Issued by: Amazon RSA 2048 M02`.

**The three differences that actually matter for our case:**

| | ACME / Let's Encrypt | AWS ACM |
|---|----------------------|---------|
| Cert validity | 90 days | 13 months |
| Renewal mechanism | You run a CertBot timer / cron; misses = expired cert + outage | AWS auto-renews silently, ~60 days before expiry. Zero operator action. |
| Where you can use it | Any server, any filesystem | AWS services only (ALB, CloudFront, API GW) — cannot be exported |
| Private key access | You manage it on disk (chmod 600, in /etc/letsencrypt/) | AWS holds it; you never see it |
| Validation CNAME (DNS-01) | CertBot tells you the CNAME, you put it in DNS yourself | ACM writes the CNAME directly into Route 53 (when the zone is in the same account) |
| Cost (public cert on AWS) | $0 | $0 |

**What we gain by choosing ACM here:**

1. **Zero renewal infrastructure.** No CertBot daemon, no `/etc/letsencrypt`, no cron timer, no "did the renewal hook fire?" debugging at 3am. AWS rotates the cert in place; the ALB picks up the new one automatically. We literally never touch this again.
2. **Private key never on our filesystems.** The cert's private key lives inside ACM's HSM-backed storage. It can't accidentally leak in a backup, can't be committed to git, can't be stolen via a server compromise. This is a real security upgrade over the "key in /etc/letsencrypt/live/" model.
3. **DNS validation auto-handled.** Since both the cert (ACM) and the zone (Route 53 sub-zone in ZI-Sandbox) live in the same AWS account, CFN can request the cert AND ACM writes the validation CNAME into the sub-zone itself — no operator step. End-to-end automation.
4. **One less piece of infrastructure to harden.** No CertBot host = no host to keep patched, no port to expose, no permissions to audit.

**What we give up:**

1. **AWS lock-in for certs.** ACM certs CANNOT be exported. If you ever moved off AWS, you'd re-issue via Let's Encrypt or another ACME provider — a half-day of work, not a fundamental obstacle, but worth knowing.
2. **No transparency on private key.** For some compliance regimes ("we must hold our own keys"), this could matter. For ISO 27001, AWS's KMS / FIPS 140-2 Level 3 HSM storage is acceptable — actually preferred over filesystem storage in many cases.
3. **AWS-only certs.** If you needed to put the same cert on a non-AWS server (a third-party CDN, an on-premise box, etc.), ACM can't do that. Our architecture has no such need.

        HostedZoneId: !Ref EnvsSubZoneId   # passed in from SSM lookup
