docs(memory): TLS cert strategy by scale — ACM-on-ALB ladder

Prep memo before Phase 3 HTTPS work begins. Captures the design
conversation that resolved the "ACM is too limited for multi-tenant
hosting" concern: it isn't, up to several thousand tenants, given
quota raises and multi-ALB sharding. The published 25-cert-per-ALB
default is a soft limit; routine quota raises take it to 100+, and
500+ with justification.

Decision ladder embedded in the memo:
  1-500 tenants     → single ALB + ACM (with quota raise where needed)
  500-1500          → ~3-5 ALBs at 300-500 certs each, all ACM
  1500-10,000       → multi-ALB with tenant→ALB sharding
  10,000+           → re-evaluate NLB+nginx+ACME-client architecture

Cost framing: at 500 tenants/ALB, ALB-tier cost is ~$0.03/tenant/mo.
DIY ACME automation requires ~40-80h initial build + 10-30h/year
ongoing, equating to $1-3k/year — roughly 60-200 ALBs of headroom.

Also captures the hard constraint that ACM certs CANNOT deploy to
nginx-on-EC2 (private key never leaves ACM's HSM), so any future
architectural shift that moves TLS termination off ALB requires
re-deciding the cert strategy entirely.

Fallback architecture (NLB+nginx+ACME-client) sketched at the bottom
of the memo so future-us doesn't reinvent it when the threshold is
crossed.

refactor(fsx): sibling-subtree isolation — PHP-FPM cannot see nginx configs

Restructure FSx layout from "root-mount with nginx as a subdir" to
"sibling subtrees mounted independently." Audit/defense-in-depth fix.

Before:
  FSx volume: /fsx/ (root)
  PHP-FPM:    mount /fsx → /var/www   (sees ENTIRE FSx incl. nginx/)
  NGINX:      mount /fsx → /var/www + mount /fsx/nginx → /etc/nginx/shared
  Risk: a PHP file-read primitive can read every nginx vhost config via
  /var/www/nginx/sites-enabled/*.conf — leaking upstream pool names,
  fastcgi params, rate limits, internal hostnames.

After:
  FSx volume: /fsx/ (root, never mounted by runtime fleets)
            ├── www/    (Drupal: src, sites/, drupal-private/, etc.)
            └── nginx/  (per-env vhost configs)
  PHP-FPM:    mount /fsx/www → /var/www
              (/fsx/nginx is OUTSIDE PHP's filesystem namespace entirely)
  NGINX:      mount /fsx/www → /var/www
              mount /fsx/nginx → /etc/nginx/shared
  Deploy-host: both mounts (operator workbench)

Defense in depth: PHP-FPM workers cannot reach /fsx/nginx via any
filesystem path — no `..`, no symlink, no traversal. Mount-layer
isolation, not chmod-dependent.
