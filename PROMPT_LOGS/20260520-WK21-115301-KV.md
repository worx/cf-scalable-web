fix(deploy-host): wait for cloud-init + bootstrap marker before reporting ready

Bit us 2026-05-20: `make deploy-deploy-host` printed "✓ Deploy host
deployed" right after CFN reported CREATE_COMPLETE, but the EC2
instance's UserData was still running bootstrap.sh — apt installs,
composer install, php extensions, etc. — for another 10-15 minutes.
EC2 status checks still in initializing. No MOTD yet. Any downstream
command that tries to use the deploy-host (init-fsx-layout, deploy-allX
Phase 1) hits "SSM agent not online" or "use-env not in PATH" because
bootstrap hasn't installed those yet.

Saying "ready" when it's not is misleading and causes operators to
trust signals they shouldn't. Fix:

scripts/wait-deploy-host-ready.sh (new):
  Three-stage block that returns only when the host is genuinely ready:

  Stage 1 — SSM agent registration (~2-3 min)
    Polls aws ssm describe-instance-information until PingStatus=Online.
    5-min timeout (very generous; usually <3 min).

  Stage 2 — cloud-init reaches terminal state (~10-15 min)
    SSM-dispatch `cloud-init status --wait`. This blocks on the
    instance until cloud-init is fully done (succeeded, errored, or
    disabled). Covers the entire UserData run including bootstrap.sh.
    Local poll with 30s feedback intervals; 30-min hard cap.

  Stage 3 — bootstrap marker sanity check
    SSM-test that /etc/worxco/deploy-host-marker exists. bootstrap.sh
    writes this near its end; presence is our project-level "done"
    signal. Belt-and-braces with stage 2.

  Exit 0 only if all three pass. Non-zero with a clear error on any
  failure.

make wait-deploy-host-ready (new target):
  Operator-callable. Can also be run standalone to check status of a
  deploy-host that someone else's deploy-deploy-host launched.

deploy-deploy-host target:
  - Old: prints "✓ Deploy host deployed" right after CFN deploy.
  - New: prints "✓ CloudFormation stack created. Instance is launching;
         bootstrap.sh still running." right after CFN deploy.
         Then calls `make wait-deploy-host-ready` which blocks until
         genuinely ready.
         Then prints "✓ Deploy host fully bootstrapped and READY".

Two-stage messaging surfaces the truth: AWS API done vs. instance
actually-usable are different events. Operators see both stages now.

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
