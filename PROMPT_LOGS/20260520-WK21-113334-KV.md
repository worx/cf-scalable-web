```
feat(deploy-host): stateful peering-handoff in destroy/deploy targets

Bit us 2026-05-19: `destroy-deploy-host CONFIRMED=yes` started, CFN
flipped the stack to DELETE_IN_PROGRESS, then canceled it 1 second
later because cf-scalable-web-sandbox-deploy-peering was still
importing the cf-deploy-host-sg-id export. Stack rolled back to
CREATE_COMPLETE; the `wait stack-delete-complete` poller spun for
30+ min before we noticed.

Per the 2026-05-19 architectural-review conversation (Kurt's design
in TODO.md): the destroy/deploy lifecycle should handle the peering
dependency itself, using a persistent SSM marker to remember which
envs had peering at destroy time. Symmetric, restartable, multi-env safe.

Implementation (inline in destroy-deploy-host and deploy-deploy-host
in Makefile):

destroy-deploy-host:
  1. Pre-check: `aws cloudformation list-imports --export-name
     cf-deploy-host-sg-id` to find every peering stack importing our
     exports.
  2. For each importer (extract <env> from stack name pattern):
     - Write SSM marker /worxco/deploy-host/peering-restore-pending/<env>
       with a UTC timestamp value (overwrite=true so re-runs are safe).
     - Run `make destroy-peering ENV=<env>`. If any peering destroy
       fails, bail BEFORE touching cf-deploy-host (so the env's peering
       is still up and the operator can investigate).
  3. Proceed with delete-stack as before.
  4. End summary: list any pending markers so the operator can see
     what's queued for the next deploy-deploy-host.

deploy-deploy-host:
  1. Normal stack deploy (unchanged).
  2. After verify-deploy-host, list any pending markers under
     /worxco/deploy-host/peering-restore-pending/.
  3. For each: run `make deploy-peering ENV=<env>`. On success,
     delete the SSM marker. On failure, leave the marker (operator
     retries; or marker carries to the next deploy cycle).
  4. Per-env atomicity: env-A's failure does NOT block env-B's restore.

All four scenarios from the original design covered:
  (1) Fresh account with no peerings → no importers found, no markers,
      clean destroy + clean deploy. No-op for the new logic.
  (2) Full env with peering up → destroy auto-tears-down peering and
      marks for restore; re-deploy restores it and clears the marker.
  (3) User manually destroyed peering before destroy-deploy-host →
      no importers found at destroy time, no marker written; deploy
      does not auto-restore — operator can manually deploy-peering
      if wanted.
  (4) Mid-restore failure → marker stays put; operator retries
      (re-run deploy-deploy-host, or manually `make deploy-peering`
      + `aws ssm delete-parameter`).

Closes the P1 TODO entry; resolution recorded inline.

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
```
