fix(deploy-host): refresh-env-config now manages /etc/hosts FSx entries

VPC peering with AllowDnsResolutionFromRemoteVpc=true allows resolution
of Route 53 private hosted zones across the peering. FSx OpenZFS does
NOT use Route 53 private zones — its DNS records live in the home
VPC's internal resolver only and are not exposed across peering.

Symptom: nc -zv <fsx-dns> 2049 fails with NXDOMAIN from the deploy
host even with all peering options correctly set.

Workaround: maintain /etc/hosts entries on the deploy host mapping
each FSx hostname to its private IP. The IP is stable for SINGLE_AZ_1
deployments (one ENI, fixed allocation).

refresh-env-config now does this in a second pass after writing
/etc/worxco/envs/<env> files:

  1. For each known env (sandbox/staging/production):
     - Read /<env>/fsx/dns-name from SSM
     - Extract filesystem ID from the DNS name (fs-XXX prefix)
     - aws fsx describe-file-systems → NetworkInterfaceIds[0]
     - aws ec2 describe-network-interfaces → PrivateIpAddress
     - Append "IP HOSTNAME  # env" to a tmp block

  2. Atomically replace the marked block in /etc/hosts:
     # BEGIN worxco-fsx-hosts (managed by refresh-env-config — do not edit)
     10.200.x.y fs-08ad...amazonaws.com  # sandbox
     # END worxco-fsx-hosts

The block is rewritten in full each refresh, ensuring stale entries
from destroyed environments are removed automatically.

Memory updated with the gotcha and the solution.

For deploy hosts deployed BEFORE this commit, run manually:
  NIC=$(aws fsx describe-file-systems --file-system-ids fs-XXX \
    --query 'FileSystems[0].NetworkInterfaceIds[0]' --output text)
  IP=$(aws ec2 describe-network-interfaces --network-interface-ids $NIC \
    --query 'NetworkInterfaces[0].PrivateIpAddress' --output text)
  echo "$IP fs-XXX.fsx.us-east-1.amazonaws.com" | sudo tee -a /etc/hosts
