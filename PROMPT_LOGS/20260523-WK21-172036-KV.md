Noted — 6-7 min is the consistent number, not the 10-15 I estimated. Updating my mental model.

One subtlety worth knowing: the make target returns when the CFN stack reaches CREATE_COMPLETE (instance launched), but `bootstrap.sh` keeps running *inside* the instance for some additional time after that before writing `/etc/worxco/deploy-host-marker`. Downstream SSM-dispatching scripts (`init-fsx-layout`, `install-drupal-remote`, `unmount-deploy-host-fsx`) all have their own "wait up to 25 min for marker" loops, so this isn't a problem — just don't be surprised if a script's first output is "still bootstrapping... attempt N/50" for a couple of cycles before it gets past the marker check.

If you want a one-shot check that bootstrap is fully done before kicking off deploy-allX (purely for peace of mind, not required):

```bash
DEPLOY_ID=$(aws --profile ZI-Sandbox ec2 describe-instances \
  --filters "Name=tag:Name,Values=cf-deploy-host" "Name=instance-state-name,Values=running" \
  --query 'Reservations[0].Instances[0].InstanceId' --output text)
aws --profile ZI-Sandbox ssm send-command --instance-ids "$DEPLOY_ID" \
  --document-name AWS-RunShellScript \
  --parameters 'commands=["ls -la /etc/worxco/deploy-host-marker 2>&1"]' \
  --query 'Command.CommandId' --output text
# wait 5s, then:
aws --profile ZI-Sandbox ssm get-command-invocation \
  --command-id <CMD_ID> --instance-id "$DEPLOY_ID" \
  --query 'StandardOutputContent' --output text
```

But honestly, just fire off `make deploy-allX ENV=sandbox` — the orchestration handles the wait gracefully.
