fix(deploy-host): drush "just works" — env file includes DRUPAL_DB_PASS + zsh auto-sources

Two related fixes so operators can run drush from the deploy-host
without remembering setup magic:

1. refresh-env-config now fetches DRUPAL_DB_PASS from Secrets Manager
   (worxco/<env>/drupal/db-password) and writes it into the env file.
   Previously: file had DRUPAL_DB_HOST/PORT/NAME/USER but not PASS,
   so drush (or psql, or any tool that bootstraps Drupal via
   settings.php's getenv() calls) would silently fall back to
   defaults — settings.php has `getenv('DRUPAL_DB_HOST') ?: 'localhost'`,
   so `drush status` reports "Database: localhost" and connections
   fail or hit the wrong host.

   File perms tightened from 0644 to 0640 root:ubuntu since it now
   contains a secret. ubuntu user (the operator) is in the ubuntu
   group; nobody else has shell access on the deploy-host anyway,
   but explicit-is-better.

   Rotation note: after `aws secretsmanager update-secret` on the
   DB password, re-run `sudo refresh-env-config <env>` to refresh
   the cached value. Matches the documented "rotate the Drupal DB
   password" runbook in OPERATIONS.md.

2. bootstrap.sh's /etc/zsh/zshrc.d/worxco-prompt.zsh now auto-sources
   /etc/worxco/envs/<active-env> on every zsh startup (when
   /etc/worxco/current-env names a non-NONE env that has an env
   file). Previously, the env file was only sourced by `use-env`
   for the current shell — fresh SSH/SSM-session logins started in
   an empty-env shell. Operators saw `[env:sandbox]` in the prompt
   and reasonably assumed env vars were set. With this change, that
   assumption holds.

   No network calls per shell startup — the env file is a flat list
   of `export` lines pre-populated by refresh-env-config. Sourcing
   is microseconds.

After both fixes, the end-to-end flow that bit Kurt today works:

    $ ssh sandbox
    [zsh auto-sources /etc/worxco/envs/sandbox]
    $ cd /var/www/drupal
    $ vendor/bin/drush status     # shows the real RDS endpoint
    $ vendor/bin/drush watchdog:show
    $ vendor/bin/drush <whatever>

No `use-env sandbox` needed (the prompt indicator already implied
it was active; now the env vars actually match).

### 16:13:05 - Response

**It works.** drush correctly reports the real RDS endpoint and "Database: Connected" / "Drupal bootstrap: Successful".

My earlier `bash -c "..."` test failed because of shell-quoting weirdness — the `\"` escapes interacted badly with how SSM parsed the parameters. The base64-wrapped version (which is what install-drupal-remote and admin-login-url and our other production tools use) works cleanly.

For your live testing: **start a fresh SSM session (or `ssh sandbox`)** so the zsh auto-source fires on shell startup. Then:

```bash
cd /var/www/drupal
vendor/bin/drush status        # should show real RDS endpoint
vendor/bin/drush watchdog:show
vendor/bin/drush pm:list
# etc.
```

Everything just works — no `use-env` needed first, no manual `export DRUPAL_DB_PASS=...` dance. The zshrc.d auto-source picks up `/etc/worxco/current-env` and sources the matching env file (which now contains the DB password too).

**Rotation note added to the script comments**: when you rotate the DB password via Secrets Manager, run `sudo refresh-env-config sandbox` afterward to refresh the cached value in the env file. Same pattern as documented in `docs/OPERATIONS.md` for the password-rotation runbook.
