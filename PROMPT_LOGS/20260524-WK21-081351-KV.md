The most striking thing: **AMI builds (25-min long pole today) only depend on image-builder stack + S3 configs**, both of which can run in default VPC. Today we wait for storage to finish before triggering AMI builds. There's no reason for that.

## Concrete plan with estimated savings

| Today | After | Saves |
|-------|-------|-------|
| AMIs triggered in Phase 2 (after storage finishes) | AMIs triggered in Phase 0 (parallel with VPC) | ~5-10 min off long pole |
| Phase 1 sequential (VPC→IAM→storage→…) | VPC + IAM + image-builder + app-drupal in parallel | ~3-5 min |
| Storage / database / cache sequential after VPC | All three parallel after VPC | ~3-5 min |
| Compute waits for explicit storage completion | Compute waits on `wait-amis` + `aws cfn wait` for storage/db/cache | trivial code, same result |

**Estimated new wall clock: ~30-35 min** down from today's ~45-50 min. Mostly because the AMI long pole gets stacked entirely under the VPC+storage critical path instead of after it.

## On wiring dependencies "after the fact"

CFN itself can't "deploy this stack and bind to a resource that appears later." BUT — we already use SSM as the late-binding layer. `cf-compute-alb` reads AMI IDs from `/<env>/ami/nginx`, RDS endpoint from `/<env>/rds/endpoint`, etc. As long as those SSM params exist when compute deploys, CFN doesn't care which stack wrote them or when.

So **the operator's `make` does the orchestration**, not CFN. `make` fires deploys in background (`&`), tracks PIDs, and `wait`s on each critical-path dependency. The pattern is what Phase 2 already does for AMI/DB/Cache; we'd just extend it to the whole flow.

**No Lambda-backed custom resources, no AWS Config, no fancy plumbing.** Just better orchestration in the Makefile.
