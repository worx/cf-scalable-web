### 14:39:16 - Response

Yes! `drush runserver` (or `drush rs`) is a really handy feature for exactly this — it spins up a development web server right inside your Drupal install. Glad you asked.

## What it actually does

Drush wraps PHP's built-in web server (`php -S`, available since PHP 5.4) with Drupal-aware routing. When you run:

```bash
cd /var/www/local/drupal
vendor/bin/drush runserver 0.0.0.0:8080
```

You get a working Drupal site at `http://<host>:8080` immediately — no Apache/Nginx/PHP-FPM setup needed. It serves the actual Drupal codebase, talks to the SQLite database, runs hooks, theme, the works. Single-threaded, no production hardening, but **perfect for "does my install actually work?" validation**.

The output looks like:
```
[notice] HTTP server listening on 0.0.0.0, port 8080, serving site Drupal Local Test, ...
```

It blocks the terminal — Ctrl-C to stop. Or background with `&`.

## Why the SSH tunnel is needed

The deploy-host's security group has **zero inbound rules** by design. Browser traffic from your Mac to `<deploy-host-ip>:8080` is blocked at the SG level. SSH-over-SSM works because SSM tunnels through AWS's control plane, not the network.

So we layer another tunnel inside the SSH-over-SSM connection:

**On the deploy-host** (in one terminal/SSM session):
```bash
cd /var/www/local/drupal
vendor/bin/drush runserver 0.0.0.0:8080
# Leave it running
```

**On your Mac** (in another terminal):
```bash
ssh -L 8080:localhost:8080 deploy-host
# This blocks too — leave it running
```

The `-L 8080:localhost:8080` flag means: "forward my Mac's port 8080 to the deploy-host's port 8080."

**Then in your Mac's browser**:
```
http://localhost:8080
```

The traffic flow:
```
Mac browser → Mac:8080 → SSH tunnel → SSM tunnel → deploy-host:8080 → drush runserver → Drupal
```

Login with `admin` / `admin` (the credentials from install-drupal-local.sh).

## Simpler alternatives for validation

If you just want to confirm the install worked, browser is overkill. Drush gives you everything via CLI:

```bash
cd /var/www/local/drupal

# Status report
vendor/bin/drush status

# List users
vendor/bin/drush user:list

# Drop into a SQLite shell (verify schema)
vendor/bin/drush sql:cli
# .tables  shows all the Drupal tables, then .quit

# Run any SQL
vendor/bin/drush sql:query "SELECT name FROM users_field_data WHERE uid > 0;"

# Check for any errors in Drupal's watchdog
vendor/bin/drush watchdog:show
```

For our iteration purposes, `drush status` + `drush user:list` is usually enough to confirm the install is healthy. Browser test is the ultimate "yes it really works" but only worth setting up the tunnel if you want to actually click around.

## Bonus tip: keep `drush runserver` alive between SSM sessions

If you Ctrl-C or your SSM session disconnects, runserver dies. To keep it running:

```bash
tmux new -s drupal     # start a tmux session
cd /var/www/local/drupal
vendor/bin/drush runserver 0.0.0.0:8080
# Ctrl-B then D to detach (server keeps running)

# Later, reattach:
tmux attach -t drupal
```

You already have tmux on the deploy-host — perfect use case for it.
