### 11:51:47 - Response

Hold up — your architecture is **better than what I was about to commit**. Let me re-think before we push anything.

## What you're proposing

| Host | FSx mount target | Drupal path | env-awareness |
|---|---|---|---|
| PHP fleet | `/var/www` | `/var/www/drupal/web` | implicit (each box is provisioned for one env) |
| nginx fleet | `/var/www` | `/var/www/drupal/web` | implicit |
| deploy-host (current) | `/var/www/sandbox` | `/var/www/sandbox/drupal/web` | explicit path prefix |
| deploy-host (proposed) | `/var/www` | `/var/www/drupal/web` | switchable via `use-env <name>` |

**Result:** every host sees Drupal at the SAME path, all the time. No env suffix in any path. The env-awareness lives in **mount metadata**, not in **filesystem structure**. Code (settings.php, drush, nginx vhost) stops needing to know which env it's in.

## I think this is right

The argument for it:
- `settings.php` paths can be plain `/var/www/drupal-private` — works on every host
- `drupal.conf` always says `root /var/www/drupal/web` — single template, no env baked in
- Multi-env in the future: each env's compute fleet is dedicated; the env-ness is in *which* FSx is mounted, not *where*
- "Two paths for the same data depending on host" is exactly the kind of asymmetry that bit us all week

What needs to change:
1. **Revert my mount-target edits** I was about to push (those changed PHP/nginx to `/var/www/$ENV` — wrong direction).
2. **Keep the failure-loudness fixes** I added (drop the `||echo WARNING` antipattern). Those are valuable independently.
3. **Deploy-host**: separate work, after you're back — change its FSx mount from `/var/www/sandbox` to `/var/www` and add `use-env <name>` (umount + remount + source the right `/etc/worxco/envs/<env>` file + record in `/etc/worxco/current-env`).

## On the deploy-host being dangerous

I agree 100%. Right operating posture:
- ASG `DesiredCapacity = 0` by default → instance off
- `make resume-deploy-host` scales to 1 when you need it
- `use-env <name>` (when added) gives explicit context selection at start of session
- `make pause-deploy-host` back to 0 when done

That model gives you:
- Almost zero attack surface 99% of the time
- Explicit "I am operating on env X" intent every session
- No persistent ambient ops capability
- Cost: deploy-host runs only when needed
