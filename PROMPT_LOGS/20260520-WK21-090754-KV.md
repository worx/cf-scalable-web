Let me think this through carefully and check for holes.

**What I hear:**
- Environment isolation is enforced by AWS resources (one FSx + one RDS + one Valkey per env), NOT by directory structure under those resources.
- Inside FSx, the directory layout uses `sites/<slug>/...` for multi-tenancy; the env never appears in a path.
- FSx is logically two mount points: one data mount (`/fsx` → `/var/www`), one config mount (`/fsx/nginx` → `/etc/nginx/shared`).
- Single-tenant (`/var/www/drupal/web/`) and multi-tenant (`/var/www/sites/<slug>/drupal/web/`) coexist — nginx vhosts point at whichever docroot.

That model is **internally consistent with what we have today** (post-cutover commit ee319d5) — we already dropped the env from the path. Multi-tenancy is the additive step of inserting `sites/<slug>/` between `/var/www/` and `drupal/`.

**Holes / decisions I'd want to surface before writing it up:**

1. **Per-request site config delivery to PHP-FPM** — biggest open question. Right now PHP-FPM workers get DRUPAL_DB_HOST/USER/PASS/NAME/SITE_NAME via `env[]` in www.conf, which is **global per box**. In multi-tenant, every site has different DB credentials (different DB name at minimum; possibly different DB user + password if isolation matters). The env-vars-in-www.conf model only works for one site per pool. Options:
   - **(a)** nginx passes site identifiers as `fastcgi_param` per vhost; settings.php uses the param to look up site-specific config (e.g., from a JSON file on FSx, or from SSM by site slug).
   - **(b)** Separate PHP-FPM pools per site (each with its own env[] block). More config but full isolation.
   - **(c)** Shared DB across all sites in an env, with Drupal's native multisite picking the right tables by `cookie_domain` / `sites.php` mapping. Simplest, but tight coupling.

   This decision shapes everything downstream — worth deciding before writing the layout doc.

2. **One codebase per env, or one codebase per site?** Your naming (`/var/www/sites/<slug>/drupal/`) implies **separate composer install per site** — each site gets its own `drupal/`, `vendor/`, etc. The alternative is **Drupal's native multisite**: one `/var/www/drupal/`, with each site as a subfolder under Drupal's own `web/sites/<sitename>/`. Tradeoff:
   - Separate installs: independent Drupal versions, independent upgrade schedules, 5x disk, can't share `vendor/`.
   - Shared codebase: efficient, but all sites locked to the same Drupal version simultaneously.
   - The path you proposed implies separate installs — confirming because this is a big architectural lever.

3. **Default nginx vhost** — when a request comes in with an unknown Host header (or just the ALB DNS), what happens? Today we have only one vhost so anything goes there. With N vhosts, we need an explicit `default_server` that returns 404 (or a friendly "site not found") — otherwise nginx picks the alphabetically-first vhost and serves it to anything, which is confusing.

4. **Backup granularity** — with `sites/<slug>/` folders, FSx backups can be restored per-site (good). RDS backups are still per-env (one snapshot covers all sites' DBs in that env). Worth documenting so operators know "I can roll back one site's files but not just one site's database without per-DB dumps."

5. **Cron / background jobs per site** — Drupal cron is per-site (`drush --uri=site.com cron`). In multi-tenant, the deploy-host (or a dedicated cron runner) needs to iterate every site. Probably a `make run-cron ENV=<env>` that loops the sites list. Out of scope for the FSx-layout doc but related.

6. **`sites/default/` for the single-tenant case** — does the single-site case live at `/var/www/drupal/web/` (today's layout) OR under `/var/www/sites/default/drupal/web/` (consistent multi-tenant layout with N=1)? The latter is cleaner — one layout, no special-case for single-site — but it'd require migrating today's running install. Worth deciding.
