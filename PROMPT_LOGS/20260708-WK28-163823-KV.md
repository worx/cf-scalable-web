### 16:39:25 - Response

That's an SBCL (Common Lisp runtime) **heap exhaustion**, not a system-memory issue. Different beast.

## What's actually happening

SBCL — the Lisp runtime that pgloader is compiled against — has a **fixed-size heap set at process start**. It does NOT grow into swap on its own. Once SBCL uses up its allocated dynamic space, garbage collection tries to reclaim; if GC can't free enough contiguous space, it aborts with "Heap exhausted" and drops you into LDB (the low-level debugger).

Your 1.8 GB of swap wasn't touched because SBCL's heap is a virtual-memory region sized once at startup. If pgloader was launched with SBCL's default heap (1024 MB on most builds), 1 GB is all it ever gets — regardless of how much RAM/swap the OS has.

## Step 1 — get out of LDB

From another shell (or `Ctrl+C` a few times if it's your foreground):

```bash
sudo pkill -9 -f pgloader
sudo pkill -9 -f sbcl
```

Confirm nothing's left:
```bash
ps aux | grep -E 'pgloader|sbcl' | grep -v grep
```

Since we used `include drop`, the partial import will re-do from scratch on the next run — no cleanup needed on the PostgreSQL side.

## Step 2 — give SBCL a bigger heap

Edit `scripts/tt.sh`, change:

```bash
pgloader --no-ssl-cert-verification /tmp/zinew.load 2>&1 | sudo tee /tmp/zinew-pgloader.log
```

to:

```bash
pgloader --dynamic-space-size 4096 --no-ssl-cert-verification /tmp/zinew.load 2>&1 | sudo tee /tmp/zinew-pgloader.log
```

`4096` = 4 GB max heap. On a 1.8 GB RAM box, SBCL will happily allocate that as virtual memory and page through swap as needed — slower than pure RAM, but it'll finish.

## Step 3 — also shrink batch size so memory pressure is less brutal

While we're editing, also change the WITH clause in the here-doc:

```
WITH include drop, create tables, create indexes, reset sequences, foreign keys,
     downcase identifiers, batch rows = 1000, prefetch rows = 1000
```

to:

```
WITH include drop, create tables, create indexes, reset sequences, foreign keys,
     downcase identifiers, batch rows = 250, prefetch rows = 250,
     workers = 2, concurrency = 1
```

That does three things:

- **`batch rows = 250, prefetch rows = 250`** — smaller batches, less per-row memory pressure at any moment
- **`workers = 2`** (down from pgloader's default 4) — fewer concurrent per-table workers holding rows in memory
- **`concurrency = 1`** — one table at a time instead of multiple concurrent tables

Slower wall-clock, dramatically lower peak memory. On a small instance this trade-off is right.

## Step 4 — add more swap (belt-and-suspenders)

You've got 2 GB swap right now. Since we're asking SBCL for 4 GB heap on 1.8 GB RAM, we want at least ~4 GB of swap available so paging doesn't hit its own wall. Add a 4 GB swap file:

```bash
sudo fallocate -l 4G /swapfile2
sudo chmod 600 /swapfile2
sudo mkswap /swapfile2
sudo swapon /swapfile2

# Confirm
free -h
```

You should now see ~6 GB total swap. When the migration's done, `sudo swapoff /swapfile2 && sudo rm /swapfile2` cleans it up.

## A word on speed expectations

With the smaller batches + workers=2 + heavy swap use, this run will be noticeably slower than the first attempt — probably 2-4× slower. For a 2.4 GB import, expect maybe 45-90 minutes wall clock. If that's too slow to tolerate, the alternative is **temporarily bumping deploy-host to a bigger instance type** (say `t4g.medium` = 4 GB RAM or `t4g.large` = 8 GB) for the migration, then downsizing after. That'd be a CloudFormation update-stack, ~5 minutes of downtime.
