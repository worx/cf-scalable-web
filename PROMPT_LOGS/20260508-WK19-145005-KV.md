**Message:**
```
feat(make): CONFIRMED=yes override for destroy-deploy-host and destroy-all

The destroy targets prompt 'Type yes to confirm' interactively, which
is great for safety on accidental invocations but annoying when you're
chaining destroy/redeploy commands and walking away. Stack four
commands, come back to find #2 silently waiting on a prompt.

Existing pattern: destroy-all already passes CONFIRMED=yes to its
child targets (destroy-storage, destroy-database, etc.) to suppress
their per-stack prompts. Extending the same pattern upward:

  destroy-deploy-host  CONFIRMED=yes  → skip the 'Type yes' prompt
  destroy-all          CONFIRMED=yes  → skip the COMPLETE TEARDOWN prompt
                                         (already passed to children)

Without CONFIRMED, both still prompt as before — safety preserved
for unaware/accidental invocations. Help text now mentions the
override so it's discoverable.

Iteration loop becomes:
  make destroy-peering ENV=sandbox
  make destroy-deploy-host CONFIRMED=yes   # was blocking on prompt
  make deploy-deploy-host
  make deploy-peering ENV=sandbox

Walk away with confidence.

Note: make remove-drupal ENV=production still requires the explicit
'WIPE PRODUCTION' confirmation — that's deliberate, production-data
destruction warrants a stronger affordance than CONFIRMED=yes.

Co-Authored-By: Claude Opus 4.6 (1M context) <noreply@anthropic.com>
```

## Iteration loop now walk-away friendly

```bash
make destroy-peering ENV=sandbox
make destroy-deploy-host CONFIRMED=yes
make deploy-deploy-host
make deploy-peering ENV=sandbox
```

Stack them in one terminal, go grab coffee. ~10-15 min later, all done.

For nuke-everything cycles:
```bash
make destroy-all ENV=sandbox CONFIRMED=yes
make deploy-allX ENV=sandbox
```

The `remove-drupal ENV=production` "WIPE PRODUCTION" prompt is intentionally NOT overridable — production data destruction warrants a stronger affordance. CONFIRMED=yes only bypasses the lighter "Type yes" confirmations.

Once your current redeploy completes successfully, that's our checkpoint to move to **Phase C step 3**.
