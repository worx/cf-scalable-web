### 17:26:54 - Git Commit

**Commit:** `e638d585c329`

**Message:**
```
feat(destroy-all): unmount FSx from deploy-host before destroy-storage

destroy-all leaves the deploy-host running by design — it's the operator
workbench, not part of the workload. But this had a hidden coupling: when
destroy-storage tore down FSx with the deploy-host still holding kernel
NFS mounts, the mounts went permanently stale. On the deploy-host's next
stop+start (or its next use-env invocation), stat() on the mount path
hung in D-state forever, wedging the next deploy-allX in init-fsx-layout.

This adds a `unmount-deploy-host-fsx` step in destroy-all, between
destroy-database and destroy-storage. The new script SSM-dispatches
`use-env none` to the deploy-host, which:
  - umounts /var/www and /etc/nginx/shared cleanly (FSx still alive at
    this point, so no stale-mount risk)
  - removes the worxco fstab block, so reboots can't re-establish the
    soon-to-be-dead mount

Past incident: 2026-05-22, init-fsx-layout hung for 45+ min after a
destroy-then-redeploy cycle because the deploy-host carried stale NFS
mounts to yesterday's destroyed FSx, and `mountpoint -q` blocked in
stat() on the dead server. (The hardening commit b79cdeb made use-env
itself stale-mount-tolerant; this commit prevents the stale mount from
existing in the first place.)

Design choices:
- Idempotent: if no deploy-host is running, exits 0 with a notice
- Tolerant: warnings instead of hard fails — destroy-all must not halt
  on a deploy-host cleanup issue. If use-env none doesn't return
  Success, prints the manual cleanup commands and continues.
- Standalone target too: `make unmount-deploy-host-fsx` for manual use
  outside the destroy-all flow.

Files Modified:
- scripts/unmount-deploy-host-fsx.sh (created)
- Makefile (new target + wired into destroy-all between database and storage)

Tests: bash -n passes; `make -n destroy-all CONFIRMED=yes` shows
       unmount-deploy-host-fsx running immediately before destroy-storage.

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
```

Then your `make destroy-all ENV=sandbox` will pick up the new `unmount-deploy-host-fsx` step that runs between `destroy-database` and `destroy-storage`. With it in place:

1. Compute, image-builder, cache, database stacks destroyed
2. **NEW:** `use-env none` SSM-dispatched to deploy-host → umounts FSx cleanly while it's still alive + scrubs fstab
3. Storage stack destroyed (FSx torn down with no client attached)
4. IAM, peering, VPC destroyed

Then on your follow-up `make deploy-allX ENV=sandbox`, init-fsx-layout will run against a deploy-host with **no carry-over state**, and use-env will mount the freshly-created FSx without anything to fight.

**Note about the use-env on the deploy-host itself:** It's still running the OLD pre-hardening version (the new code is in the repo but the deploy-host has the version baked into bootstrap.sh from before commit `b79cdeb`). That doesn't matter for *this* destroy-rebuild cycle because:
- destroy-all's new step calls `use-env none` while FSx is healthy → no stale mounts → no hang
- deploy-allX's init-fsx-layout will use-env against a fresh FSx with no stale state → no hang either

The hardened use-env only matters if you ever hit a stale-mount edge case the new pre-destroy step missed. Belt + suspenders. The deploy-host's installed version will refresh whenever you next rebuild the deploy-host AMI / instance.

Good luck with the rebuild.
