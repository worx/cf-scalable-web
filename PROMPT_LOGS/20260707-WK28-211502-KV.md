## First-time MySQL server install & init (Ubuntu 24.04)

Modern Ubuntu handles the "init the data directory + set up systemd unit + start service" automatically as part of `apt install`. You don't need the old `mysql_install_db` / `mysqld --initialize` dance anymore.

```bash
sudo apt-get update
sudo apt-get install -y mysql-server

# Verify it's running
sudo systemctl status mysql --no-pager
# Should show: Active: active (running)

# Verify /var/lib/mysql exists and is populated
sudo ls -la /var/lib/mysql | head -10
```

**Root auth on Ubuntu MySQL 8:** the `root` account uses socket auth (`auth_socket` plugin), so `sudo mysql` gets you in with no password. Do not run `mysql_secure_installation` for a scratch DB — the defaults are fine.

## Create scratch DB + user + restore

```bash
# Create the DB, matching Aurora's typical charset/collation for Drupal
sudo mysql <<'SQL'
CREATE DATABASE zinew
  CHARACTER SET utf8mb4
  COLLATE utf8mb4_unicode_ci;

CREATE USER 'worxco'@'localhost' IDENTIFIED BY 'localscratchpass';
GRANT ALL PRIVILEGES ON zinew.* TO 'worxco'@'localhost';
FLUSH PRIVILEGES;
SQL

# Restore. Pipe the dump straight into mysql to avoid double-disk-use.
# Adjust path to wherever your mysqldump wrote it.
time mysql -u worxco -p'localscratchpass' zinew < /var/www/zinew.sql
# For a 2.4G dump on decent I/O, expect 3-10 min.
```

Watch for these expected-but-cosmetic warnings during restore (Aurora mysqldump quirks):

| Warning | Meaning | Action |
|---------|---------|--------|
| `Unknown table 'column_statistics' in information_schema` | You didn't use `--column-statistics=0` on the dump. Harmless if it shows up during restore. | Ignore |
| `access denied; you need (at least one of) the SUPER, SYSTEM_VARIABLES_ADMIN privilege(s) for this operation` on `SET @@GLOBAL.GTID_PURGED` | You didn't use `--set-gtid-purged=OFF` on the dump. GTID lines fail but everything else restores. | Usually ignorable; if it aborts the load, we'll sed those lines out |
| `Definer 'worxco'@'somehost' does not exist` on views/procedures/triggers | Definer references a user that doesn't exist locally. Drupal 8+ rarely has views/procs, so this often doesn't come up. | If it appears, we'll sed the DEFINER clauses out |

If it aborts partway with any of those, tell me the exact error and I'll give you the sed one-liner to sanitize the dump. Quick preview of the common fixes:

```bash
# If GTID_PURGED aborts:
sed -i '/^SET @@GLOBAL\.GTID_PURGED/d' /var/www/zinew.sql

# If DEFINER clauses fail:
sed -i 's/DEFINER=`[^`]*`@`[^`]*`//g' /var/www/zinew.sql
```

Then re-run the restore.
