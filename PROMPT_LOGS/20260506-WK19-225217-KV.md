```
fix(peering): Enable cross-VPC DNS resolution after peering creation

VPC peering by default does NOT enable cross-VPC DNS resolution.
Instances in a peered VPC can ROUTE to the other VPC's resources but
cannot RESOLVE its private DNS names.

RDS and ElastiCache mask this because their hostnames are AWS-public
DNS that resolve to private IPs from any VPC via split-horizon. FSx
is different: its hostname is resolvable only from within its home
VPC. Symptom from the deploy-host:

  $ nc -zv fs-08ad1afb8b729b58f.fsx.us-east-1.amazonaws.com 2049
  nc: getaddrinfo for host "...": Name or service not known

The fix is to enable AllowDnsResolutionFromRemoteVpc on both sides
of the peering connection. AWS::EC2::VPCPeeringConnection does not
expose this option in CloudFormation, so it must be set via API
after the stack creates the connection.

make deploy-peering now:
  1. Deploys cf-deploy-peering.yaml (CFN)
  2. Reads PeeringConnectionId from stack outputs
  3. Calls aws ec2 modify-vpc-peering-connection-options with both
     RequesterPeeringConnectionOptions.AllowDnsResolutionFromRemoteVpc=true
     and AccepterPeeringConnectionOptions.AllowDnsResolutionFromRemoteVpc=true

This is idempotent — safe to re-run on existing peerings.

For peerings created BEFORE this commit, run the AWS CLI manually once:
  aws ec2 modify-vpc-peering-connection-options \
    --vpc-peering-connection-id pcx-xxxxxxxx \
    --requester-peering-connection-options AllowDnsResolutionFromRemoteVpc=true \
    --accepter-peering-connection-options AllowDnsResolutionFromRemoteVpc=true \
    --region us-east-1

Memory updated with the gotcha.
```
