Functionally it works as-is — nginx's master process runs as `root`, and `root` reads everything regardless of ownership. The config file is **only read by the master at startup/reload**, not by the worker processes (which run as `www-data`).

But for hygiene and consistency with the rest of `/etc/nginx/` (which is `root:root 644` everywhere), I'd normalize it. Since it sits on FSx and UIDs pass through NFS, using `root:root` is also more portable across the fleet:

```bash
sudo chown root:root /var/www/nginx/sites-enabled/drupal.conf
sudo chmod 644 /var/www/nginx/sites-enabled/drupal.conf
ls -l /var/www/nginx/sites-enabled/drupal.conf
```

No reload needed for the chown — nginx already has the file in memory from the previous reload.

The `644` mode is fine here because the vhost contains **no secrets** (unlike `www.conf` on the PHP boxes which we deliberately kept 640 because it embeds the DB password and Valkey AUTH token).

Same advice for the parent directory if you didn't already: `sudo chown root:root /var/www/nginx /var/www/nginx/sites-enabled` and `sudo chmod 755` on both. Matches what the AMI build creates for the local fallback dir.

Now try the curl — should be the real test:

```bash
ALB=$(aws elbv2 describe-load-balancers \
  --query 'LoadBalancers[?contains(LoadBalancerName,`sandbox`)].DNSName' --output text)
echo "ALB: $ALB"
curl -i "http://$ALB/" | head -30
```

You should see either Drupal's homepage HTML or a redirect to `/user/login`. If you get a 502, the PHP-FPM hop is the next thing to chase.
