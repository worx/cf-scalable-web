Ah — this is a legitimate design gap I missed when writing the orchestration helper. Let me explain, then give you the quick fix.

## What's happening

The `_ssm-run-jumpbox.sh` helper runs this:

```bash
aws --profile ZI-Sandbox cloudformation describe-stacks --stack-name cf-scalable-web-sandbox-migration-bucket ...
```

The `--profile ZI-Sandbox` flag looks up credentials in `~/.aws/credentials`. On your **Mac**, that profile exists (we've been using it all day). On **deploy-host**, that profile probably doesn't exist — deploy-host normally uses its **instance role** for sandbox-account operations (no profile needed because it IS in the sandbox account).

So the aws call silently fails (profile not found → empty output), the helper sees empty output and reports "stack not deployed" — but really it's "we couldn't authenticate to check."

**The bucket absolutely exists.** You've been using it all day. This is a credential-lookup issue on deploy-host, not a missing stack.

## The 2-minute fix — teach deploy-host that `ZI-Sandbox` profile = "use instance role"

On deploy-host:

```bash
# See what profiles are currently configured
cat ~/.aws/config 2>/dev/null
```

Add a profile block that points to the instance metadata service (no keys needed since deploy-host's instance role already has S3 + CFN access in sandbox):

```bash
mkdir -p ~/.aws
cat >> ~/.aws/config <<'EOF'

[profile ZI-Sandbox]
credential_source = Ec2InstanceMetadata
region = us-east-1
EOF
```

Verify it works:

```bash
aws --profile ZI-Sandbox sts get-caller-identity
# should show an ARN under the SANDBOX account
```

Then retry your `dump-all`:

```bash
cd migration
make dump-all CONFIRMED=yes | tee /tmp/dump-all-$(date +%Y%m%d).log
```

## Why this works

`credential_source = Ec2InstanceMetadata` tells the AWS CLI: "when using this profile, don't look for access keys — call IMDS on the running EC2 to get temporary credentials from the attached instance role." Deploy-host's instance role already has S3 read + CFN read for the migration bucket stack (since that's how the sandbox-side migration workflows have worked all along).

Zero new credentials, zero copies of anything sensitive.

## Also — you'll likely need the same treatment for `ZoningInfoAdmin`

Deploy-host is in sandbox account, but for prod-side operations (the SSM send-command to the jumpbox), the helper uses `--profile ZoningInfoAdmin`. That profile HAS to have real keys (or an assume-role config) because deploy-host's instance role can't act in prod.

You mentioned copying that profile to deploy-host during the original migration work — worth confirming:

```bash
aws --profile ZoningInfoAdmin sts get-caller-identity
# should return the PROD account ID (978068244875)
```

If that works, you're good. If it doesn't, we need to fix that one too before dump-all can dispatch SSM to the jumpbox.

## Longer-term design fix (for a future commit)

The right architectural fix is to have `_ssm-run-jumpbox.sh` accept env-var overrides so the profile lookup is configurable per host:

```bash
SANDBOX_PROFILE="${SANDBOX_PROFILE:-ZI-Sandbox}"
PROD_PROFILE="${PROD_PROFILE:-ZoningInfoAdmin}"
```

Then on deploy-host you could set `SANDBOX_PROFILE=""` in `.env` and the helper would skip the `--profile` flag (using the instance role directly). But that's a nice-to-have refactor for later; the credential-source approach above unblocks you tonight with zero code changes.

Try the `~/.aws/config` update and run `sts get-caller-identity` for both profiles. Report back what you see and we'll take it from there.
