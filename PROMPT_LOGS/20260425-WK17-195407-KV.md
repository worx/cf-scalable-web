This is a really well-thought-out idea. Yes, it's doable — and you're right that it needs to be a **hybrid: a hook for the automatic time tracking + a skill for the intelligent parts** (summarization, close-day, post). Let me lay out what I'm thinking and then ask my questions.

## Architecture: Hook + Skill Hybrid

### The Hook (automatic, fires every interaction)

The existing `prompt-logger.sh` already fires on `UserPromptSubmit` and `Stop`. We'd add a **companion hook** (or extend the existing one) that:

1. **On UserPromptSubmit**: Check the last Stop timestamp. If <10 min ago → extend the current time block. If >10 min → close the previous block (end time = last Stop + 10 min) and start a new block.
2. **On Stop**: Record the response timestamp.

This is pure bash — timestamps and simple math. No Claude intelligence needed here. The hook writes to a small JSON state file per project.

### The Skill (manual triggers, needs Claude's brain)

Two commands:

- **`/timeclose`** (or whatever we name it) — "Close the day":
  1. Reads the day's time blocks from the state file
  2. Reads that day's PROMPT_LOGS to understand *what was done*
  3. Uses Claude to generate bullet-point summaries for each time block
  4. Writes a finalized timesheet entry (Markdown or JSON) to a Claude-managed timelog directory

- **`/timepost`** — "Post to intermediate file":
  1. Reads all un-posted timesheets
  2. Formats them into Time_Billing's `pending_billing.json` structure (`staff`, `project`, `date`, `times[]`, `notes`)
  3. Writes to a **separate** intermediate file (not `~/Downloads/pending_billing.json` that the Apple Notes pipeline uses)
  4. Marks those entries as posted

### Data Flow

```
Interaction happens
    → hook records start/stop timestamps to state file
    → prompt-logger.sh records what happened (as it already does)

/timeclose (end of day)
    → Claude reads prompt logs + time blocks
    → generates bullet summaries
    → writes finalized timesheet to local store

/timepost (when ready to bill)
    → reads finalized timesheets
    → writes Time_Billing-compatible JSON
    → you move that into the Windows pipeline
```
