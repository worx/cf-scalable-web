fix(use-env): handle stale NFS mounts without hanging

Symptom: After destroy-all (which leaves the deploy-host running by
design) followed by a fresh deploy-allX, init-fsx-layout hung for
45+ min. Root cause: yesterday's `use-env` wrote fstab entries pointing
at the (now-destroyed) FSx. On today's deploy-host start, the kernel
re-established those NFS mounts to a server that no longer exists.
`mountpoint -q` calls stat() on the target path, which dispatches an
NFS request to the dead server, which never replies — the process
sits in D-state forever and `kill -9` doesn't free it.

Two changes:

1. Mount detection now reads /proc/mounts directly via awk, instead of
   calling `mountpoint -q`. /proc/mounts is an in-kernel structure
   lookup; no stat(), no network I/O, can't hang.

2. Unmount uses `umount -f -l` (force + lazy) instead of plain umount.
   Lazy (-l) detaches the mount from the namespace immediately, even
   if there are in-flight kernel references to the gone NFS server.
   Force (-f) hints to the NFS client to give up retransmissions.
   Both flags are safe for healthy mounts (no data loss; in-flight
   ops still complete), and the combination resolves the stale-mount
   case cleanly. Non-zero return is downgraded to a WARN so use-env
   continues to the re-mount step regardless.

Files Modified:
- scripts/deploy-host/use-env (mount detection + unmount hardening,
  applied to both the no-arg status block and the main switch path)

Tests: bash -n passes. Real validation: re-run init-fsx-layout against
       the previously-hung deploy-host after it's been freshly cleared.
Docs: extensive inline comments explain the two failure modes the
      hardening addresses.
