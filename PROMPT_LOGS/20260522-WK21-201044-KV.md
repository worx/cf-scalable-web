Three coordinated changes:

1. deploy-allX no longer runs install-drupal or smoke-test. After Phase 4

   the operator wants to know which layer broke. Step order chosen so a

   incident 2026-05-22: the check falsely reported pdo_pgsql missing

**New workflow:**

```bash
make deploy-allX ENV=sandbox            # infra only (~30-50 min)
make install-drupal-full ENV=sandbox    # Drupal + DNS + tests (~5-7 min)
```

**`install-drupal-full` step order is deliberate** — failure at step N narrows the problem to layer N or later:

1. `install-drupal-remote` — Drupal exists on FSx?
2. `smoke-test-drupal` — ALB+nginx+PHP-FPM+Drupal serve (forged Host header)?
3. `publish-dns` — Route 53 alias UPSERTed?
4. `smoke-test-public` — full public DNS path works?

Strict mode — any failure aborts. Unlike deploy-allX's old best-effort Phase 5/6, this target's contract is "set up the test site end-to-end" so the operator should know which layer broke.

**On the deploy-host question:** if you're already SSH'd in (via SSM), use `make install-drupal ENV=sandbox` directly — no `-remote`, no SSM round-trip. Note that `install-drupal-full` *should* be run from your Mac, not the deploy-host: steps 2 and 4 (smoke tests) hit URLs from outside the VPC, and step 3 (publish-dns) runs against Route 53 which is region-global — all of those work from your Mac and are appropriately "end-to-end" tests from there.

**On the tag** — sensible plan. Once your next cycle validates everything (deploy-allX clean, install-drupal-full clean end-to-end), I'd suggest:
