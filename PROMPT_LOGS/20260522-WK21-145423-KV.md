### 14:57:14 - Response

Pushed as `bfcfebe`.

**Once your deploy-allX completes, the new flow is:**

```bash
make deploy-allX ENV=sandbox      # infra + best-effort Drupal
make publish-dns ENV=sandbox      # NEW — reads site-name from SSM, finds sub-zone, UPSERTs alias
make smoke-test-public ENV=sandbox # validates the whole DNS path
```

And before next destroy-all:

```bash
make unpublish-dns ENV=sandbox    # leaves the sub-zone clean
make destroy-all
```

**Why this design holds up:**

- **Zero hardcoding** — works for `sandbox.envs.zoning-info.com`, would work just as well for `prod.envs.zoning-info.com`, or for someone forking this project and using `dev.envs.example.com`. The sub-zone name is *derived* from the site-name (`<host>.<sub-zone>`).
- **Refuses non-routable TLDs** explicitly. If you (or someone else) leaves `DrupalSiteName: "drupal-sandbox.test"` in the param file by mistake, `publish-dns` errors with a clear hint to update the JSON, instead of silently writing a record that'll never resolve.
- **Idempotent** — `publish-dns` uses UPSERT (always safe to re-run); `unpublish-dns` is a noop if no record exists.
- **Survives destroy-all + redeploy** without remembering anything: the next `publish-dns` reads the new ALB's DNS from the rebuilt CFN stack and updates the alias in place.

The one-time sub-zone delegation in `docs/DNS-SETUP.md` is still the prerequisite — these scripts only manage records *within* a zone that already exists in this AWS account.
