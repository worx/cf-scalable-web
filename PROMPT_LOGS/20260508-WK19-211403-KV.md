fix(php-fpm): mutate www.conf in place — pool.d include= is invalid

The drop-in pattern (configure-php.sh writing pool.d/extra/{cache-session,
drupal-env}.conf, [www] picking them up via include=...) was the wrong
architecture. PHP-FPM's `include=` directive is only honored at the GLOBAL
level (php-fpm.conf), not inside a [pool] section. Putting it inside [www]
caused FPM to fail config-load with "Unable to include www.conf" — every
fresh AMI boot failed health checks and the ASG cycled.

New approach: configure-php.sh writes env[...]/php_value[...] directives
directly into pool.d/www.conf between BEGIN/END markers (sed-strip + append,
idempotent across reboots). Locks www.conf to 0640 root:www-data because
the embedded DB password and Valkey AUTH token shouldn't be world-readable.

No RecipeVersion bump needed — components are unchanged; the canonical
S3 source for configure-php.sh and www.conf is what flipped, so a fresh
sync-configs + build-amis is enough.

Files:
- image-builder/configs/configure-php.sh — strip-and-append marker block in www.conf
- image-builder/configs/php/{7.4,8.3}/www.conf — remove the broken include= line
- docs/memory/gotchas.md — document the include=-in-pool-section trap
