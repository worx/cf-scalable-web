Now the HTTPS conversation. **Recommendation: ACM wildcard cert for `*.envs.zoning-info.com`, validated via DNS in the envs sub-zone we just delegated.**

**Why wildcard, why now:**

- One cert covers every env you'll ever stand up under the envs sub-zone — sandbox, test-new, future test-after-that-one, prod-rehearsal, etc.
- DNS-01 validation writes a CNAME into the envs sub-zone, which lives in ZI-Sandbox — no parent-account access for cert provisioning or renewal
- ACM auto-renews; you set it up once, it stays valid forever
- The validation is the dependency that ties HTTPS to the DNS work — sequencing-wise, DNS sub-zone must be live before ACM can validate

**What HTTPS adds (the actual implementation work, ~half a day):**

1. **ACM cert request** (one-time): wildcard for `*.envs.zoning-info.com`. ACM writes the validation CNAME into the envs sub-zone. Validation completes in ~5 minutes once the CNAME is in place.

2. **ALB HTTPS listener** (new CFN resource): port 443, with the ACM cert ARN attached, forwarding to the same target groups as the port 80 listener.

3. **Optional but recommended:** port 80 listener rule to redirect HTTP → HTTPS. Single-line CFN addition. Now every request is HTTPS.

4. **Security group:** ALB security group opens 443 inbound 0.0.0.0/0 (same posture as the existing 80).

5. **Drupal-side settings.php:**
   - `$settings['reverse_proxy'] = TRUE`
   - `$settings['reverse_proxy_addresses'] = [...]` — the ALB's possible source IPs. ALB IPs rotate; easiest is to trust the VPC CIDR since traffic from ALB to NGINX is always intra-VPC.
   - Drupal then reads `X-Forwarded-Proto` from the ALB and generates `https://...` URLs in form actions, link rels, etc.

6. **Nginx config:** pass `X-Forwarded-Proto` through to PHP-FPM as a fastcgi_param. Already typically handled by the standard fastcgi_params, but worth verifying our vhost forwards it.

**Three gotchas worth knowing:**

- **HTTP→HTTPS redirect breaks `make smoke-test-drupal` until updated.** Currently curls port 80; needs to follow the redirect (curl `-L`) or hit 443 directly.
- **Mixed-content warnings in Drupal admin pages** if any module hardcodes `http://` somewhere. Usually solved by `drush php:eval "\\Drupal::service('config.factory')->getEditable('system.site')->...` — Drupal doesn't store the scheme; the issue is usually a `<base href>` or hardcoded asset URL in a custom module/theme.
- **ALB does NOT do client-cert auth or mTLS by default.** If you ever need that, ALB does support it (relatively recently), but it's a separate listener rule and not the path we'd take for a sandbox site.
