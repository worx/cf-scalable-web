### 15:11:39 - Response

Deploy worked. And your `/root/.aws/` finding is worth talking through — because in this case, **NOT backing it up is actually the right call**, and I'd argue we should keep it out of the backup list even when we discover it.

## Why `/root/.aws/` shouldn't be in the backup

That folder on the jumpbox contains only two kinds of content, both AWS-managed:

| Path | What it is | Backup? |
|------|------------|---------|
| `/root/.aws/cli/cache/*.json` | Cached temporary credentials from IMDS (instance-role → STS) | **No** — ephemeral, ~1-hour expiration, regenerated on next AWS CLI call |
| `/root/.aws/config`, `/root/.aws/credentials` | Long-lived AWS creds/config | **No** — jumpbox uses IAM instance role, not stored creds. If either exists, it's a security anti-pattern we should scrub |

The `.aws/cli/cache/` gets populated the first time any process on the box calls AWS. If we restored those from a backup:
- Best case: they've expired → next call regenerates → no harm, no benefit
- Worst case: they haven't expired → a new instance temporarily uses old-instance credentials → weird logs, potentially failed operations
- Also: creates yet another copy of temporary-but-still-valid credentials in S3, which is a security downside for zero upside

The fact that our new jumpbox auto-generated its own `/root/.aws/` when the AWS CLI first ran is exactly what SHOULD happen. Amazon didn't put it there — the CLI put it there when UserData ran `aws s3 cp` for the first time.

**So your "oh shit" wasn't actually an "oh shit" at all — it was the system doing exactly the right thing.** Nice catch though; that's the kind of paranoid inspection that keeps clients' data safe.

## `.bashrc` / `.profile` — agreed, leave for now

Same reasoning as your call. On a fresh Ubuntu AMI, these are stock. Backing them up captures Ubuntu's defaults, which just get re-installed by the AMI anyway. Only worth adding to `BACKUP_PATHS` if you customize them (aliases, PATH additions, prompt tweaks). If that day comes, one line in `backup-state.sh`:

```bash
BACKUP_PATHS=(
  "/root/.ssh"
  "/root/.env"
  "/root/bin"
  "/root/.bashrc"       # ← add here
  "/root/.profile"      # ← and here
  ...
)
```

Redeploy the backup script (just re-run `make backup-jumpbox` — it uploads the current version to S3 automatically), and the next backup captures the new paths. No CFN redeploy needed. No instance touched.

## The bigger pattern you're establishing

You've just done the second half of "build a safety net, then use it and refine as you find gaps." The gap-finding is the interesting part — you'll surface a few more items over the coming weeks. Every one is:

1. Notice something's missing
2. Ask "should this be backed up?" — sometimes yes, sometimes no (like `.aws/`)
3. If yes, one-line edit to `BACKUP_PATHS`
4. `make backup-jumpbox` refreshes both the script in S3 AND the backup content
5. Next EventBridge nightly (or any manual backup) catches it going forward

Zero CFN churn, zero infrastructure risk, zero downtime for those refinements. That's exactly what the architecture is for.
