### 10:28:27 - Response

**Critical finding — and this might actually be THE bug all along.**

```
FATAL: password authentication failed for user "drupal_user"
```

The password in Secrets Manager is `X~pz\`*5Jdn6SjU<KbEMDokw,C8B>,x2f`. Look at character 3: that's a **literal backtick**. When `install-drupal.sh` originally ran with this value, its psql heredoc would have shell-interpolated the backtick as command substitution:

```bash
$CMD <<-EOSQL
    CREATE USER drupal_user WITH PASSWORD '$DRUPAL_DB_PW';
EOSQL
```

In an unquoted heredoc, `` `*5Jdn6SjU<KbEMDokw,C8B>,x2f` `` becomes a command substitution that tries to execute `*5Jdn6SjU<KbEMDokw,C8B>,x2f` (some glob expansion + redirection), produces garbage or empty output, and **the postgres user ends up with a truncated/mangled password** that doesn't match anything anyone has on file.

That's also why our FPM-served `_test.php` was returning 502 / connection-reset yesterday: workers were spawning, applying `getenv('DRUPAL_DB_PASS')`, getting the secret value, calling pg_connect → instant authentication failure → PHP fatal in some path → worker dies / FastCGI closes the connection mid-response.

**This is fixable in 60 seconds without redeploying anything**:

```bash
# On the deploy-host (uses master credentials via the psql-env helper)

# 1. Generate a new password with the SAFE charset (matches the tightened
#    ExcludeCharacters we committed last night)
NEW_PASS=$(aws secretsmanager get-random-password --password-length 32 \
  --exclude-characters '"@/\\'"'"'#%&+:;=?[]~$`<>*|^!(){}' \
  --region us-east-1 --query RandomPassword --output text)
echo "New password (32 chars, alphanum + safe-punct only): $NEW_PASS"
echo "$NEW_PASS" | grep -oE '[~$`<>*|^!(){}@/\\#%&+:;=?\[\]"'"'"']' && echo "BAD" || echo "CLEAN"

# 2. Update the postgres user FIRST (so RDS has the new password before
#    anything reads the new secret value)
psql-env <<EOSQL
ALTER USER drupal_user WITH PASSWORD '$NEW_PASS';
EOSQL

# 3. Update the secret to match
aws secretsmanager update-secret \
  --secret-id worxco/sandbox/drupal/db-password \
  --secret-string "$NEW_PASS" \
  --region us-east-1 > /dev/null
echo "Secret updated"

# 4. Clean up the local var so the new password isn't lingering in your shell
unset NEW_PASS
```

Then **re-test from the same PHP box** with the same shell (yours has `DRUPAL_DB_PASS` still pointing at the OLD secret value, so re-export from the secret first):

```bash
export DRUPAL_DB_PASS=$(aws secretsmanager get-secret-value \
  --secret-id worxco/sandbox/drupal/db-password \
  --region us-east-1 --query SecretString --output text)
echo "Now PASS_LEN=${#DRUPAL_DB_PASS}, starts with '${DRUPAL_DB_PASS:0:2}'"
php /var/www/drupal/web/_test.php
```

You should now see `PG_OK: [drupal, drupal_user, <timestamp>]`.

If THAT works, the next test is the real one — without doing any other deploys yet, retry the FPM-served curl through the ALB. **PHP-FPM workers are still in memory with the OLD (broken) password baked into their env via the marker block.** They won't see the new value until FPM restarts (and reads the new value via configure-php.sh).

