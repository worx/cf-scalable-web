## Autovacuum — PostgreSQL is very hands-off

Yes, PostgreSQL has **autovacuum built-in and enabled by default**. On RDS it's on out of the box, and it handles both:

- **VACUUM** — reclaims space from deleted/updated rows (PostgreSQL keeps old row versions for MVCC; autovacuum cleans them up)
- **ANALYZE** — refreshes query-planner statistics

Autovacuum wakes up every minute (`autovacuum_naptime=60`) and checks each table against thresholds. Once a table has had ~20% of its rows modified since the last vacuum, autovacuum kicks in on that table.

**You almost never need to manually vacuum in normal operation.** The one situation where a manual `ANALYZE` is genuinely useful is right after a bulk load like the one we just did — autovacuum's next natural trigger might be days away, so we jumped the gun with our manual command to make the query planner smart immediately. After that, you can basically forget PostgreSQL exists at the maintenance layer.

Verify autovacuum is on:

```bash
PGPASSWORD="$DRUPAL_DB_PASS" psql \
  "host=$DRUPAL_DB_HOST port=$DRUPAL_DB_PORT dbname=$DRUPAL_DB_NAME user=$DRUPAL_DB_USER sslmode=require" \
  -c "SHOW autovacuum;"
# Expected: on
```

And see when autovacuum last touched each table (useful diagnostic later):

```sql
SELECT relname, last_autovacuum, last_autoanalyze
  FROM pg_stat_user_tables
 ORDER BY last_autovacuum DESC NULLS LAST
 LIMIT 20;
```

## The pg_ permission-denied warnings — safe to ignore

`drupal_user` doesn't own PostgreSQL's internal system catalogs, so it can't `ANALYZE` them. That's by design and identical to MySQL's `mysql` schema being off-limits to app users. RDS runs its own maintenance on those tables using the master user (`dbadmin` in your config). The warnings just mean "we skipped those because you're not the owner" — every user table in `zinew` and `public` still got analyzed successfully.
