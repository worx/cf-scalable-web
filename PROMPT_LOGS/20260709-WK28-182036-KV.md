## But before you refresh — one gotcha to know about

PHP-FPM instances have their own **opcache** — in-memory bytecode cache separate from Drupal's DB cache. When we replaced the codebase on FSx, the running PHP-FPM instances still have the *old* bytecode from the install-drupal-full files cached in memory.

Depending on how the AMI was built, this either:

- **Self-heals** if opcache is configured with `opcache.validate_timestamps=1` and a low `revalidate_freq` — PHP notices the file mtimes changed and re-compiles automatically
- **Doesn't heal** if timestamps aren't validated (production-hardened config)

Fastest way to sidestep the guessing: **restart PHP-FPM on the pool**. From your Mac:

```bash
# Find the target pool — likely php83-asg for Drupal 11
make ssm-php   # or whichever target opens SSM into a PHP-FPM node
# If there's no direct target, find the instance and use aws ssm start-session directly
```

On a PHP-FPM node:

```bash
sudo systemctl restart php8.3-fpm
```

Repeat across each instance in the `php83-asg`. Or — since this is auto-scaling with a 7-day lifecycle (per the project CLAUDE.md), you could **trigger an instance refresh** on the ASG to spin up brand-new nodes:

```bash
aws --profile ZI-Sandbox autoscaling start-instance-refresh \
  --auto-scaling-group-name sandbox-php83-asg \
  --preferences MinHealthyPercentage=50,InstanceWarmup=180
```

That takes 5-10 min but gives you clean slate PHP-FPM instances with fresh opcache.

**Common issues you might hit and quick triage:**

| Symptom | Likely cause | Quick fix |
|---------|--------------|-----------|
| Blank white screen | PHP fatal, opcache stale | Check `/var/log/php8.3-fpm.log` on the node |
| 500 error | Missing module/dependency | Look at Drupal watchdog: `drush ws` |
| "The website encountered an unexpected error" | Cache leftover from install-drupal-full | `drush cr` again |
| Homepage renders but images broken | Expected for S3-hosted files; local files should work | Flag for follow-up |
| Redirect loop to `/user/login` or install page | Session data mismatch | `drush uli` to get a fresh admin login link |
| Slow first load, snappy after | Caches warming — normal after `drush cr` | Just wait |
