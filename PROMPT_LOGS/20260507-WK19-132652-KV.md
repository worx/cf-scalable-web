## SSH config tokens

- `%h` = the hostname after resolution (the `HostName` directive's value if set, otherwise the alias you typed)
- `%p` = the port (defaults to 22 unless overridden)
- `%u` = the user
- `%r` = the remote user

So in our `ProxyCommand`:
- `--target %h` becomes `--target i-0abc123def456` (the HostName)
- `'portNumber=%p'` becomes `'portNumber=22'`

SSM opens a tunnel from your local SSH client to the instance's port 22, and SSH runs through that tunnel.

## Multi-alias Host line

Yes — `Host deploy-host sandbox` makes BOTH aliases match the same block. You can `ssh deploy-host` or `ssh sandbox` and get the same behavior:

```
Host deploy-host sandbox
  HostName i-0abc123def456
  User ubuntu
  IdentityFile ~/.ssh/id_ed25519
  StrictHostKeyChecking accept-new
  ProxyCommand sh -c "AWS_PROFILE=ZI-Sandbox aws ssm start-session --target %h --document-name AWS-StartSSHSession --parameters 'portNumber=%p'"
```

## The replace-instance gotcha

The deploy-host's instance ID changes every time you `make deploy-deploy-host` (per yesterday's gotcha). So hardcoded `HostName` will go stale.

**Future enhancement** — dynamic lookup by tag, so you never edit the config. Replace `HostName` with a `ProxyCommand` that resolves it at connection time:

```
Host deploy-host sandbox
  User ubuntu
  IdentityFile ~/.ssh/id_ed25519
  StrictHostKeyChecking accept-new
  ProxyCommand bash -c 'I=$(AWS_PROFILE=ZI-Sandbox aws ec2 describe-instances --filters "Name=tag:Name,Values=cf-deploy-host" "Name=instance-state-name,Values=running" --query "Reservations[].Instances[].InstanceId" --output text); AWS_PROFILE=ZI-Sandbox aws ssm start-session --target $I --document-name AWS-StartSSHSession --parameters "portNumber=%p"'
```

That looks up the running deploy-host by Name tag every time you `ssh deploy-host`. Survives instance replacements with no config edits. Slightly slower (one extra API call), but you don't notice.

For now, hardcoding the ID is fine — just remember to update it after the next `make deploy-deploy-host`.
