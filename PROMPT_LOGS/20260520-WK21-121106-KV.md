feat(admin-ssh-key): registry + sync tooling for scp-over-SSM-proxy

Implements the P2 TODO captured in docs/memory/admin-access-policy.md:
a small registry of admin SSH public keys (in SSM) that's auto-synced
to the deploy-host's ubuntu authorized_keys. Enables scp/sftp/rsync
using the AWS-StartSSHSession SSM document — without ever opening
port 22 on any security group.

scripts/deploy-host/sync-admin-ssh-keys.sh (new):
  Runs on the deploy-host. Reads /worxco/admin/ssh-public-keys/* from
  SSM, writes a managed authorized_keys file (0600, ubuntu:ubuntu) with
  a generated header explaining the file's lifecycle. Idempotent.

scripts/admin-ssh-key.sh (new):
  Runs locally (Mac). Subcommands:
    add NAME FILE   — write SSM param + auto-sync to deploy-host
    remove NAME     — delete SSM param + auto-sync
    list            — show owner + key type + fingerprint
    sync            — force a fresh sync to deploy-host
  All sync ops dispatch via SSM to the deploy-host, which git-pulls the
  latest repo then runs sync-admin-ssh-keys.sh.

scripts/deploy-host/bootstrap.sh:
  Adds a new step near the end that runs sync-admin-ssh-keys.sh so
  freshly-deployed hosts come up with all currently-registered admin
  keys pre-installed (zero-touch for new operator boxes).

Makefile targets:
  admin-ssh-key-add NAME=<name> FILE=<path>
  admin-ssh-key-remove NAME=<name>
  admin-ssh-key-list
  admin-ssh-key-sync

Design choices noted in admin-access-policy.md memory:
  - Per-name SSM params (not a single StringList) — cleaner add/remove
    semantics, no comma-delimiter issues with SSH key comment fields.
  - Auto-sync on add/remove — UX is "one command, key is live."
    Operator never has to remember a separate sync step for the
    common case.
  - Public keys aren't secret, but the LIST of authorized admins IS
    sensitive — SSM param IAM policy should restrict write access to
    admin roles. Read access by the deploy-host instance role is fine
    (it needs to read them to write authorized_keys).

### 12:14:28 - Response

Pushed `77742f0..fb3da42`. **Workflow to test:**

**Add your key:**
```bash
make admin-ssh-key-add NAME=kurt FILE=~/.ssh/id_ed25519.pub
```

(Replace `id_ed25519.pub` with whichever pubkey file you want — could be `id_rsa.pub`, `~/.ssh/work.pub`, etc.) This:
1. Writes the key to SSM at `/worxco/admin/ssh-public-keys/kurt`
2. Auto-dispatches a sync to the running deploy-host
3. Within ~5 seconds, your key is in `/home/ubuntu/.ssh/authorized_keys`

**Verify:**
```bash
make admin-ssh-key-list
# Shows: kurt  ssh-ed25519  SHA256:... (your key's fingerprint)
```

**Test scp via SSM proxy:**

You need this in your local `~/.ssh/config` (one-time setup):

```
Host i-* mi-*
    ProxyCommand sh -c "aws ssm start-session --target %h --document-name AWS-StartSSHSession --parameters 'portNumber=%p'"
    User ubuntu
```

Then:
```bash
# Get the deploy-host instance ID
DEPLOY=$(aws ec2 describe-instances --filters Name=tag:Name,Values=cf-deploy-host \
  Name=instance-state-name,Values=running \
  --query 'Reservations[0].Instances[0].InstanceId' --output text)

# Test SSH (should land you in zsh on the deploy-host, MOTD and all)
ssh $DEPLOY

# Test scp
echo "hello from kurt's mac" > /tmp/test.txt
scp /tmp/test.txt $DEPLOY:/tmp/

# Verify via SSM session: cat /tmp/test.txt
```

Port 22 stays closed on all SGs the whole time — the SSH handshake travels through the SSM tunnel, NOT through a public TCP/22 connection. AWS Session Manager plugin (which you already have via `aws ssm start-session`) handles the proxy.

**To propagate updates** (already auto-handled): every `admin-ssh-key-add` and `admin-ssh-key-remove` triggers a sync to the live deploy-host. If you ever want to force a re-sync (e.g., suspect drift), `make admin-ssh-key-sync`.
