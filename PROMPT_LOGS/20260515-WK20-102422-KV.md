### 10:25:07 - Response

That's **expected behavior** — and not the cause of any problem.

PHP has three separate execution modes, each with its own environment:

| Mode | Config files | Env vars source |
|---|---|---|
| **PHP CLI** (`php script.php`) | `/etc/php/8.3/cli/php.ini` | Inherits your shell's environment |
| **PHP-FPM workers** (FastCGI requests) | `/etc/php/8.3/fpm/php.ini` + `pool.d/www.conf` | The `env[KEY]=value` lines in www.conf (because `clear_env=yes` strips parent env, then injects ours) |
| **PHP module under Apache** (mod_php) | apache2 config | Apache process env |

Our marker block lives in `pool.d/www.conf` → only PHP-FPM workers see those env vars. When you run `php /var/www/drupal/web/_test.php` from a shell, you're using PHP CLI, which has no concept of pool configs. CLI's `getenv('DRUPAL_DB_HOST')` returns empty because your shell never had those exported.

This is *intentional* — you don't want random shell users (or stray cron jobs) to silently inherit DB credentials.

**The CLI test IS still useful, though** — just set the env yourself first and re-run, to prove the PHP→RDS path works independent of FPM:

```bash
export DRUPAL_DB_HOST=sandbox-postgres.c256s2w8gam2.us-east-1.rds.amazonaws.com
export DRUPAL_DB_PORT=5432
export DRUPAL_DB_NAME=drupal
export DRUPAL_DB_USER=drupal_user
export DRUPAL_DB_PASS=$(aws secretsmanager get-secret-value \
  --secret-id worxco/sandbox/drupal/db-password \
  --region us-east-1 --query SecretString --output text)
export DRUPAL_SITE_NAME=drupal-sandbox.test
echo "Password length: ${#DRUPAL_DB_PASS}"
php /var/www/drupal/web/_test.php
```

Expected output:
```
PHP_OK
DRUPAL_DB_HOST=sandbox-postgres.c256s2w8gam2.us-east-1.rds.amazonaws.com
DRUPAL_DB_USER=drupal_user
...
PG_OK: ["drupal","drupal_user","2026-05-15 ..."]
```

If you get `PG_OK` here, that confirms:
- DB credentials currently in Secrets Manager are valid
- PHP can reach RDS from this box
- No SG / VPC / endpoint issue between PHP boxes and RDS

If you get `PG_FAIL` instead, that's a separate breakage we should chase before the deploy (probably means the existing postgres user's password got out of sync with what's in Secrets Manager, or the secret value has the `X~` characters that some part of the chain mangles).

**And once you do `unset DRUPAL_DB_PASS` after the test**, you're back to a clean shell — no creds lingering.

This is genuinely an orthogonal test from the 502 mystery, but useful baseline for ruling out "PHP can't reach RDS at all."
