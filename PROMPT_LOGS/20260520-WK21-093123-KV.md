**On secret propagation specifically (your callout):** The mechanism is in FSX-LAYOUT.md under "Secret propagation — how a config change reaches running workers." The short version:

- Drop `env[]` per-site config from `www.conf` entirely (was the boot-time approach; doesn't scale to multi-tenant)
- nginx passes `SITE_SLUG` as a `fastcgi_param` based on which vhost matched
- `settings.php` reads `/var/www/sites/$SITE_SLUG/site-meta.yml` at request time (with APCu cache, 60s TTL)
- Secret references in `site-meta.yml` → settings.php fetches actual values from Secrets Manager (also APCu-cached)
- Site add: write new `site-meta.yml` + new vhost → `make reload-nginx` (graceful). No PHP-FPM restart.
- Secret rotation: ALTER USER on RDS + update Secrets Manager + bump cache-stamp in `site-meta.yml` → workers see new value within ~60s (or immediately if cache-stamp is bumped). No PHP-FPM restart.
- The only thing that still needs `make restart-php-fpm` is **PHP code changes** (OPcache holds compiled PHP).

There's a doc-table in multi-tenancy.md ("Per-site config delivery and propagation") that summarizes which operation needs which restart — most need none.
