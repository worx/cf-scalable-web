### 12:37:15 - Response

**Root cause:**

The Drupal install itself succeeded — site got installed, settings.php got swapped to the env-var-driven version, file perms got set. The failure was in the very next step (uninstall Update Manager).

After `step "Replace settings.php with env-var-driven config"`, settings.php no longer carries hardcoded DB credentials — it calls `getenv('DRUPAL_DB_HOST')` etc. The earlier drush call worked because `drush site:install` was given `--db-url=...` on the command line (bypassing settings.php). But `drush pm:uninstall` has no such flag and must bootstrap through settings.php — which means the env vars must be in drush's environment.

The script exported those vars at line 565-569, but that was below the pm:uninstall call. So drush bootstrapped with empty `getenv()` returns, fell back to the `'localhost'` / `'drupal_user'` / `''` defaults in settings.php, couldn't connect to the DB, and reported the misleading-sounding "Command pm:uninstall was not found." (It really meant "I can't load any command list because I can't reach the DB.")

The compounding factor: that drush call uniquely used `sudo -E HOME=/root` while every other drush call in the script runs as the script's own root user — making this the *only* call where env propagation was even relevant.

**Fix:**
1. Hoisted the env-var export block up to immediately after the settings.php swap, where it's needed.
2. Dropped the `sudo -E HOME=/root` from the pm:uninstall — the script already runs as root via SSM dispatch, matching how every other drush call in this script works.
3. Removed the now-redundant duplicate export block from the verify step.

**Recovery for the current sandbox:**

Drupal is actually fully installed — the failure was post-install cleanup. You can finish the job by re-running just the install on the deploy-host. Since the script is idempotent for re-runs (`composer create-project` skips if `/var/www/drupal` exists, `drush site:install` will re-prompt about dropping tables), the cleanest path is:

```bash
make install-drupal-remote ENV=sandbox
```

The fixed script will re-run all steps; the only one that actually changes state at this point is the pm:uninstall, which will now succeed (and is itself idempotent — handles the "already not enabled" case from the previous fix).

If you'd prefer to skip the full re-run, you can run just the pm:uninstall by SSM'ing into the deploy-host and running:

```bash
sudo bash -c '
  export DRUPAL_DB_HOST=$(grep ^DRUPAL_DB_HOST /etc/worxco/envs/sandbox | cut -d= -f2- | tr -d \")
  source /etc/worxco/envs/sandbox
  cd /var/www/drupal
  vendor/bin/drush pm:uninstall update -y
'
```

But the full `make install-drupal-remote` is the cleanest test that the fix actually fixes it.
