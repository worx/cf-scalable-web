### 15:13:18 - Response

Your reasoning is solid — hardcoded paths in a self-contained script is the right call at this scale. Two observations that reinforce it:

## The `BACKUP_PATHS` array IS already config-as-data

In `scripts/migration/scripts/jumpbox/backup-state.sh`, the paths are in an array at the top:

```bash
BACKUP_PATHS=(
  "/root/.ssh"
  "/root/.env"
  ...
)
```

Adding a path = one line. Removing one = deleting one line. That's basically what a config file gives you, but without the parse-and-validate boilerplate. For arrays of file paths, bash arrays are perfectly good config.

The pattern I've been using is: **shared logic in `_common.sh`, script-specific data (paths, destinations) inline at the top of each script.** That's the "config lives with the code that uses it" pattern. For 2-3 backup scripts, it's clearer than an external config; for 20 backup scripts, you'd want the external config. We're firmly at 2.

## The real shared abstraction is `_common.sh`

The heavy lifting (log_init, log_upload_and_exit, colors, confirm, dry-run, mask_secret) is already extracted. What's LEFT in each script is exactly what SHOULD be per-script:
- What paths to back up
- What S3 key to write to
- What the confirmation message says

Duplication there is intentional — those are the "what this script does" bits, not the "how to do it safely" bits. Clean separation.

## If we ever get to a 3rd/4th similar backup script

Then it's worth extracting. Rule of thumb: two copies is fine ("we're not sure yet"); three is a smell; four is a refactor. We're at two after adding deploy-host — perfect spot to pause and evaluate.

## Before I start building — one question I raised you didn't answer

**Shared migration bucket vs new bucket for deploy-host state?**

| Option | Pros | Cons |
|--------|------|------|
| **Shared** — use existing `sandbox-migration-kv-worxco` under a new `config/deploy-host-*` prefix | Zero infrastructure add; one place to look; existing lifecycle rules already right | Bucket has a "TEMPORARY" tag; getting deleted with `make destroy-migration-bucket` would also lose deploy-host backups |
| **Separate** — new bucket `sandbox-deploy-host-backups` (or similar) | Deploy-host backups outlive migration lifecycle; cleaner separation of concerns | New CFN template, new stack, new bucket to manage |

Deploy-host is meant to be long-lived (unlike jumpbox which is per-migration). If migration bucket gets destroyed but deploy-host state was in there too, that's a bad coupling.

**My recommendation:** separate bucket. New CFN template (`cf-deploy-host-backups.yaml`), new stack, in sandbox account. ~30 lines of CFN. Lives at the parent-project level (not under `migration/`) since deploy-host is core infrastructure.
