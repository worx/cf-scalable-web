**Message:**
```
fix(wait-amis): bypass AWS CLI per-page JMESPath via jq on aggregated JSON

Past incident 2026-05-24: wait-amis ran for 40+ minutes when all 3
pipelines had been AVAILABLE for ~30 of those minutes. Loop never
exited until the hard timeout, blocking deploy-allXX from continuing
to the compute phase.

Root cause: AWS CLI's auto-pagination applies the --query JMESPath
expression to EACH PAGE of API results, then concatenates the per-page
outputs. With 7 historical images in the pipeline split across pages,
the query `reverse(sort_by(imageSummaryList, &dateCreated))[0].state.status`
returned ONE status per page — concatenated into a multi-line string
like "AVAILABLE\nBUILDING".

That multi-line value matched neither the `AVAILABLE)` arm nor the
FAILED-class arms of the case statement, so it fell through to `*)
ALL_DONE=0`. The loop kept polling indefinitely (until the 40-min cap)
even though all pipelines had genuinely succeeded.

Verified: dumping the raw bytes via xxd showed `AVAILABLE\x0aBUILDING\x0a`
for php-fpm-74 and php-fpm-83 (which had 7+ historical images each).
nginx, with fewer images, fit on one page and returned a single value.

Fix: use --output json + jq instead of --query --output text. jq
operates on the post-pagination AGGREGATED list (since AWS CLI joins
the pages into one JSON list before output), so sort_by + index work
correctly on the actual full result set.

  Before:
    aws imagebuilder list-image-pipeline-images ... \
      --query "reverse(sort_by(imageSummaryList, &dateCreated))[0].state.status" \
      --output text

  After:
    aws imagebuilder list-image-pipeline-images ... --output json \
      | jq -r '.imageSummaryList | sort_by(.dateCreated) | reverse | .[0].state.status // "UNKNOWN"'

Other --query patterns in the Makefile/scripts that use [0]-indexing
on Stacks, Reservations, single-object responses are NOT affected —
those queries operate on single-object responses where pagination
isn't a factor. Only sort+index over list responses is vulnerable.

Captured the past incident in an inline comment so future-us
recognizes the symptom (loop hangs at "still running" past timeout
even though pipelines look fine in the AWS console).

Files Modified:
- Makefile (wait-amis target)

Tests: make -n wait-amis parses; manually verified the new jq query
       returns the correct single status (BUILDING) for the
       currently-in-flight pipeline that was confusing the old query.
```

```bash
# Get the latest AMI ID for each pipeline:
for PIPELINE in nginx php-fpm-74 php-fpm-83; do
  case $PIPELINE in
    nginx) SHORT=nginx ;;
    php-fpm-74) SHORT=php74 ;;
    php-fpm-83) SHORT=php83 ;;
```
