**Commit:** `bf7df7daadf3`

**Message:**
```
fix(refresh-env-config): Use 'export' so vars propagate to child processes

Symptom: drush status reported DB hostname=localhost even after
'source /etc/worxco/envs/sandbox; export DRUPAL_DB_PASS=...'. Every
DB field in drush status matched the settings.php fallback default,
meaning getenv() returned empty for ALL of them.

Cause: the env file used 'KEY=value' lines (without 'export'). When
sourced, the variables exist in the parent shell but are NOT
exported, so they don't propagate to child processes. drush spawns
PHP, PHP can't see them via getenv(), settings.php falls back to
'localhost', Drupal can't connect, bootstrap fails.

Fix: every line in /etc/worxco/envs/<env> now uses 'export KEY=value'
syntax. Sourcing once exports everything; drush, future PHP-FPM
pool config, and any other downstream consumer all see the values.

The actual config values (DRUPAL_DB_PASS, etc.) the user needs to
manually export still come from Secrets Manager since secrets shouldn't
live in cached files. This fix only affects the non-secret endpoint
config.

Memory updated with this gotcha (the 'source without export' pattern
is one Bash veterans know but easy to forget).

To take effect on a running deploy host:
  cd ~/projects/cf-scalable-web && git pull
  sudo install -m 0755 scripts/deploy-host/refresh-env-config /usr/local/sbin/
  sudo refresh-env-config sandbox
  source /etc/worxco/envs/sandbox
  env | grep DRUPAL_DB_HOST     # should show the RDS endpoint, exported

Co-Authored-By: Claude Opus 4.6 (1M context) <noreply@anthropic.com>
```
