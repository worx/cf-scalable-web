fix(install-drupal): salt.txt needs root:www-data 0640, not root:root 0600

Today's deploy-allX otherwise succeeded but HTTP 500'd in Phase 6
smoke test. drush watchdog showed the same error as yesterday:
"RuntimeException: Missing $settings['hash_salt']".

settings.php paths were correct (yesterday's commit 417f9f0 fixed that).
salt.txt existed at the right path. But: PHP-FPM workers run as
www-data; the file was `-rw------- root:root`, i.e. mode 0600 owned
by root with group root. www-data has no read access. settings.php's
`is_readable($_salt)` returns false → `$settings['hash_salt'] = ''`
→ Drupal throws RuntimeException → HTTP 500.

Root cause in install-drupal.sh: the salt-generation step had
`chmod 600 $SALT_FILE` (owner-only) AND no chown. The file was
created via shell redirect as the script's effective user (root
when SSM-dispatched), inheriting root's primary group (also root).
Net result: root:root 0600 — PHP-FPM shut out.

Fix: chown root:www-data + chmod 0640. The 0640 mode keeps salt.txt
unreadable to non-www-data system users (still a secret) but
readable to PHP-FPM workers (which is the whole point).

This matches what docs/FSX-LAYOUT.md "File ownership and permissions"
table already documents — the doc was right, the install script was
the outlier. Comment added in install-drupal.sh pointing at the spec
so future readers see both files agree.

Diagnostic note for the gotchas pile: `sudo -u www-data cat $file`
RETURNS data even when the file is unreadable — the "Permission
denied" stderr message is itself ~57 bytes, easily mistaken for a
short file's content. The trustworthy test is from a PHP process:
`sudo -u www-data php -r '... is_readable(...) ...'` because it
doesn't conflate error-text with content.
