feat(migration): full workflow colocated under migration/ with backup/restore

Consolidates the one-time prod → sandbox Drupal migration into a self-
contained `migration/` subdirectory with its own Makefile, CFN templates,
parameters, and scripts. When a client migration is complete, the entire
folder can be deleted (`rm -rf migration/`) without touching the parent
project's Makefile or infrastructure.

## Architectural changes

- **Full colocation** — moved all migration-related files from the parent
  project's `cloudformation/`, `cloudformation/parameters/`, and (new)
  `scripts/migration/` into a single `migration/` subfolder. The parent
  Makefile now has a 3-line hint block pointing operators to
  `cd migration && make help` instead of exposing 10+ migration targets.

- **Cross-account SSM orchestration** — introduces
  `migration/scripts/_ssm-run-jumpbox.sh` as a reusable helper that:
    1. Uploads a specified script + `_common.sh` to sandbox S3
    2. Uses `aws ssm send-command` to run it on the prod jumpbox
    3. Polls for completion and streams output back
  All `make dump-*` and `make backup-jumpbox` targets are 2-line
  wrappers around this helper.

- **Shared shell helpers** — `migration/scripts/_common.sh` provides
  the primitives used by every runtime script:
  `confirm_or_exit`, `is_dry_run`, `run_or_echo`, `log_init`,
  `log_upload_and_exit`, `require_env`, `log_*`, `mask_secret`,
  plus ANSI color codes matching the parent Makefile.

## Backup/restore infrastructure (new)

Protects jumpbox operator-managed state (SSH keys, /etc/worxco, etc.)
against instance replacement events — including AMI-driven ones we
already got burned by twice this session.

- `migration/scripts/jumpbox/backup-state.sh` — tar of state paths →
  sandbox S3 under `config/jumpbox-latest.tar.gz` (never expires) and
  `config/jumpbox-archive/jumpbox-<UTC>.tar.gz` (90-day retention).

- `migration/scripts/jumpbox/restore-state.sh` — reverse flow with a
  MANDATORY pre-restore snapshot to `/tmp/pre-restore-backup-<UTC>.tar.gz`
  so any restore has a locally-preserved undo path.

- **UserData auto-restore** in `cf-migration-jumpbox.yaml` — a fresh
  instance boots, checks for a prior state backup, and restores it
  automatically. Non-fatal if no backup exists (fresh migration).

- **EventBridge Scheduler nightly backup** — `AWS::Scheduler::Schedule`
  resource with `ScheduleExpressionTimezone: America/Chicago` fires
  at 02:00 CST/CDT (DST-aware, no drift). Uses the same download-
  from-S3-and-run pattern as `make backup-jumpbox`.

- **`make destroy-jumpbox` takes a final backup** before delete-stack,
  with `SKIP_FINAL_BACKUP=yes` escape hatch.

## Migration data workflow (jumpbox side, invoked via SSM)

Four dump scripts + one convenience target for fire-and-forget:

- `dump-mysql.sh` — mysqldump prod → S3 stream (Secrets Manager-backed
  credentials, mode-600 defaults-file, no password on ps output)
- `dump-codebase.sh` — tar Drupal codebase (excludes settings.php,
  files/, private/, .git, .ddev)
- `dump-files.sh` — tar sites/default/files/
- `dump-private.sh` — tar private/
- `dump-all` — sequential mysql → codebase → files → private, aborts
  on any failure. ~30-45 min end-to-end for typical client data volumes.

## Migration data workflow (deploy-host side)

- `restore-mysql.sh` — S3 dump → local MariaDB scratch DB. Includes
  the `utf8mb4_0900_ai_ci` → `utf8mb4_unicode_ci` sed (MariaDB doesn't
  know MySQL 8's default collation) and `mysql_native_password` user
  setup for pgloader/qmynd compatibility.

## S3 bucket organization + lifecycle rules

Bucket layout under prefix-scoped lifecycle:
```
s3://sandbox-migration-kv-worxco/
├── scripts/                (no expiration — runtime dependency
│                            for scheduler + UserData)
├── dumps/                  (30 days — migration data ferry)
├── logs/                   (30 days — script audit trail)
└── config/
    ├── jumpbox-latest.tar.gz     (no expiration — auto-restore
    │                              source of truth)
    └── jumpbox-archive/          (90 days — state backup history)
```

Refactor from the previous "expire everything at 30 days" blanket rule
that would have silently killed both the auto-restore and scheduled
backup flows.
