### 16:09:47 - Response

**Found the bug.** Classic NLB issue:

- **`preserve_client_ip.enabled = true`** on the php target groups
- Both nginx (client) and PHP (target) are in the **same VPC**
- When nginx → NLB → PHP, the source IP is preserved as nginx's IP
- The PHP target replies **directly to nginx** (asymmetric routing — bypasses NLB)
- nginx's TCP stack sees an unsolicited response not matching the outbound flow → RST mid-response

This is the "NLB hairpin / preserve_client_ip same-VPC" symptom AWS warns about. TCP-only health checks from the NLB-managed IPs still work (different code path), which is why target health stays green.

**Fix:** disable `preserve_client_ip` on the PHP target groups. Then connections appear to come from the NLB's ENIs, the response flows back through the NLB, and the path is symmetric.

```bash
# Set the attribute on both PHP TGs
for TGN in sandbox-php74-tg sandbox-php83-tg; do
  TG=$(aws elbv2 describe-target-groups --names "$TGN" --query 'TargetGroups[0].TargetGroupArn' --output text)
  aws elbv2 modify-target-group-attributes --target-group-arn "$TG" \
    --attributes Key=preserve_client_ip.enabled,Value=false
  echo "$TGN: $(aws elbv2 describe-target-group-attributes --target-group-arn $TG --query 'Attributes[?Key==`preserve_client_ip.enabled`].Value' --output text)"
done

# Then retry
ALB=sandbox-alb-95496877.us-east-1.elb.amazonaws.com
curl -s -H "Host: drupal-sandbox.test" "http://$ALB/_test.php" --max-time 10
```

You should see `PHP_OK` + env vars + `PG_OK` this time.
