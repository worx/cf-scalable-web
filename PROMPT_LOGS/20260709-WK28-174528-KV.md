## Better long-term chmod pattern

For future reference, the `chmod 644` sweep I gave you was too heavy. The correct pattern for a modern Composer-managed Drupal is:

```bash
# Directories: 755
sudo find /var/www/drupal -type d -exec chmod 755 {} \;

# Files: 644 EXCEPT:
sudo find /var/www/drupal -type f -exec chmod 644 {} \;

# Restore execute on scripts detected by shebang
sudo find /var/www/drupal -type f -exec grep -l '^#!' {} \; 2>/dev/null | xargs -r sudo chmod +x

# vendor/bin/* is always executable
sudo chmod +x /var/www/drupal/vendor/bin/*
```

The `grep -l '^#!'` finds any file that starts with a shebang line (`#!/usr/bin/env php`, `#!/bin/bash`, etc.) and restores +x on those. That catches drush, phpunit, phpstan, and every other tool composer might install without you having to know their names.

## Note on where drush "belongs" in this architecture

Deploy-host CAN run drush because FSx is mounted here — so you're using it as a convenient admin box. That's fine and often useful for quick checks like `drush status`.

For actual site cache rebuilds after changes (`drush cr`), you'd normally do it on a PHP-FPM instance because:

- Opcache lives per-instance in memory — a `cr` on deploy-host clears Drupal's DB-cached data but NOT the opcode caches on the PHP nodes
- User uploads and other things behave more consistently when done from the actual runtime environment

But for `drush status`, `drush pm:list`, `drush uli`, etc — deploy-host is fine.
