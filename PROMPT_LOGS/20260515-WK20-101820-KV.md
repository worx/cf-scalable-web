### 10:18:41 - Response

Quick test from inside the nginx box:

```bash
# 1. Sanity — does DNS resolve to NLB ENI IPs?
getent hosts sandbox-nlb-fadbc87b8e311ff8.elb.us-east-1.amazonaws.com
# Should print two lines (one per AZ — 10.200.6.139 and 10.200.7.15 in our case)

# 2. TCP probe via nc (the cleanest test)
nc -zv sandbox-nlb-fadbc87b8e311ff8.elb.us-east-1.amazonaws.com 9083
# Expected: "Connection to ... 9083 port [tcp/*] succeeded!"

# Same for 9074
nc -zv sandbox-nlb-fadbc87b8e311ff8.elb.us-east-1.amazonaws.com 9074

# 3. If nc isn't installed, bash has a builtin:
timeout 3 bash -c '</dev/tcp/sandbox-nlb-fadbc87b8e311ff8.elb.us-east-1.amazonaws.com/9083' \
  && echo TCP_OPEN || echo TCP_FAIL
```

**What each result means:**

| Outcome | Meaning |
|---|---|
| ✅ `succeeded!` / `TCP_OPEN` | nginx box CAN reach NLB at 9083. The NLB IS forwarding to PHP-FPM on a healthy target (otherwise the listener would refuse). This tells you the L4 path is clean — any 502 from here is at the FastCGI/application layer above. |
| ❌ `Connection refused` (fast) | NLB listener exists but no healthy targets behind it. Check target group health. |
| ❌ `Connection timed out` (slow, ~3s) | SG or NACL is blocking, OR the NLB DNS isn't resolving to a routable IP. Re-check DNS first. |
| ❌ `no route to host` | Routing table issue (very unlikely inside one VPC). |

**One quick caveat for interpreting "succeeded":** `nc -z` and bash `/dev/tcp` only do a TCP three-way handshake then close. They don't speak FastCGI. So a "succeeded" means TCP works but doesn't prove the FastCGI roundtrip works. The latter is what was breaking yesterday — TCP was always fine, FastCGI was the layer that got RST.

You can also run the same `nc -zv` against the **actual NLB ENI IPs** to bypass DNS:
```bash
nc -zv 10.200.6.139 9083
nc -zv 10.200.7.15 9083
```

If you see different behavior between the two ENIs (one AZ works, the other doesn't), that points at cross-zone routing or an SG that's AZ-specific.
