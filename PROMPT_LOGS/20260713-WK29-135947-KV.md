## Recommended sequence

Since you've just made changes to `.env/migrate` and confirmed state on the current instance, take one belt-and-suspenders backup first so the new instance auto-restores your CURRENT state (not the state that was on the original pre-replacement jumpbox):

```bash
cd migration

make backup-jumpbox CONFIRMED=yes

# 2. Deploy — will replace instance, but state carries forward automatically
make deploy-jumpbox

make ssm-jumpbox
```

## What CFN will do this time

Both the parameter type change AND the AMI value change will register in the change-set:
- `Jumpbox` EC2: **Replacement: True** (not "Conditional" — this is a real replacement because ImageId is definitely changing to `ami-01ebd2f86ac3bd98c`)
- New UserData already includes auto-restore (deployed in the previous cycle)
- New instance boots → runs UserData → auto-restore pulls latest backup → state returns
- `NightlyBackupSchedule` updates to point at new instance ID (transparent — same LogicalId `${Jumpbox}` resolves to whatever the current instance ID is)
- `SchedulerExecutionRole` unchanged

**End state:** new instance on latest Ubuntu AMI, all state intact, scheduler wired to the new instance, deprecation reminders stopped, and any future AMI updates follow the same automatic pattern.
