## What happened

Our template has:
```yaml
LatestAmiId:
  Type: AWS::SSM::Parameter::Value<AWS::EC2::Image::Id>
  Default: /aws/service/canonical/ubuntu/server/24.04/stable/current/arm64/hvm/ebs-gp3/ami-id
```

That SSM parameter always resolves to **whatever Ubuntu's latest 24.04 ARM64 AMI is at the moment of deploy**. When we deployed the jumpbox originally, it got the AMI ID that was current then. Today it's different (Ubuntu publishes new AMIs regularly). CFN sees `ImageId` changed → mandatory replacement.

The IAM Role change itself is `Replacement: False` (as I predicted), but the AMI drift piggy-backs onto the same deploy and drags the instance replacement in with it.

This is a **classic gotcha** in AWS: never trust `latest` SSM parameters for anything you don't want to replace. My mistake for not flagging this earlier.

## Recommended fix — bypass CFN entirely for now

Since we only need to add ONE inline policy to the IAM role, we can do it directly via `aws iam put-role-policy` — completely bypassing CFN. This:
- Adds the policy to the running role immediately
- Doesn't touch the EC2 instance
- Preserves everything on the root volume
- Takes 3 seconds

The tradeoff: CFN's "known state" of the stack will drift from reality. Next time we deploy the stack for real (with a pinned AMI), CFN will "reconcile" and might try to add the policy again (which is idempotent) or remove it (if we forget to update the template). We'll fix that properly in a follow-up.

## Proposed sequence

**Right now:**

```bash
# 1. Attach the ReadMigrationSecret policy imperatively (no CFN, no instance change)
SECRET_ARN=$(aws --profile ZoningInfoAdmin cloudformation describe-stacks \
    --stack-name cf-migration-secret --region us-east-1 \
    --query "Stacks[0].Outputs[?OutputKey=='MigrationSecretArn'].OutputValue" \
    --output text)

aws --profile ZoningInfoAdmin iam put-role-policy \
    --role-name cf-migration-jumpbox-role \
    --policy-name ReadMigrationSecret \
    --policy-document "$(cat <<EOF
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "GetMigrationSecretValue",
      "Effect": "Allow",
      "Action": [
        "secretsmanager:GetSecretValue",
        "secretsmanager:DescribeSecret"
      ],
      "Resource": "$SECRET_ARN"
    }
  ]
}
EOF
)"

# 2. Verify the policy is attached
aws --profile ZoningInfoAdmin iam list-role-policies \
    --role-name cf-migration-jumpbox-role \
    --query 'PolicyNames' --output table
```
