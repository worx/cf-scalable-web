Yes, this makes sense. Let me sketch what I'm hearing, propose an architecture, and surface the questions that need answers before we build.

## What I Hear

A new optional CloudFormation stack (call it `cf-app-drupal.yaml` and `cf-app-wordpress.yaml`) that:
- Runs **after** the infrastructure is up
- Installs Drupal/WordPress onto FSx (so it persists across PHP instance lifecycles)
- Initializes the database in RDS
- Wires up Valkey for sessions/cache
- Writes settings.php / wp-config.php pointing to all AWS endpoints via SSM/Secrets
- Leaves you with a working site reachable via ALB + /etc/hosts

The whole point: prove the infrastructure works end-to-end without bringing your own production app.

## Where the Install Should Run

I see three viable approaches. My recommendation in italic:

| Approach | Pros | Cons |
|----------|------|------|
| One-shot installer EC2 in compute subnet | Clean, full toolchain (composer, drush, git, unzip), writes FSx then terminates | Adds a new instance type to manage |
| SSM Run Command on an existing PHP instance | No new resources | PHP instances are immutable serving infrastructure — installing onto them feels wrong |
| *Custom Resource Lambda invoking SSM* | *CloudFormation-native, automatic on stack create* | *Lambda + composer is painful, prefer running an actual EC2* |

I'd go with a **one-shot installer EC2** — small (t4g.small), launched by the stack, runs userdata that does everything, then a CloudFormation `WaitCondition` lets the stack succeed when install completes. The instance can either auto-terminate or stay around for `drush` operations (a "drupal-cli host" similar to deploy-host).

Actually — since you already have a deploy-host concept, the cleanest answer might be: **the installer is the deploy-host's job**. Add a `make install-drupal` Make target. Deploy-host already has SSH access to GitHub for fetching code, and (once VPC peering is done — which is on the TODO list) it'd have access to FSx/RDS. This would tie nicely to existing infrastructure.

But VPC peering isn't done yet. So for v1, a one-shot installer EC2 inside the project VPC is probably right.
