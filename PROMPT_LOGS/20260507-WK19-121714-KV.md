### 12:18:42 - Response

Good morning. Two strategic options for today, plus your memory/storage question:

## On memory/storage

t4g.micro = 1GB RAM, 20GB disk. Existing UserData adds 2GB swap, so 3GB total memory.

- **Disk**: ~200MB total for PHP + composer + drush + their deps. Well within 20GB.
- **Memory**: Composer with `drush/drush` is the riskiest step. It does a dependency tree solve which can briefly hit 500MB-1GB RSS. With swap it works but may thrash and be slow.

If memory was the cause, we'd see `Killed` or OOM messages in the log. We didn't yesterday — we saw a clean `set -e` exit at "HOME not set." Doesn't mean memory can't bite us tomorrow, but it's not yesterday's bug.

**Easy hedge**: bump deploy-host to `t4g.small` (2GB RAM, ~$6/mo vs ~$3/mo). Removes memory entirely from the suspect list. Worth doing for a non-trivial cost increase.

## On the debugging approach — two options

**Option A — tactical (what you suggested)**: Add `set -x` plus echo markers and resource snapshots throughout existing UserData. Quick to implement, gives full trace in `/var/log/deploy-host-bootstrap.log` next deploy.

**Option B — strategic**: Move the whole "after-AWS-CLI" portion of UserData into `scripts/deploy-host/bootstrap.sh` in the repo. UserData becomes minimal: install git, clone repo, then `bash scripts/deploy-host/bootstrap.sh`. Massive wins:

- **Debuggable**: `bootstrap.sh` is a real bash script. Can SSM into the box and re-run sections manually to find failures without another destroy/deploy cycle.
- **Iterable**: Edit `bootstrap.sh`, run it. No CloudFormation. Minutes vs ~10 minutes per attempt.
- **Visible**: Each step gets proper logging without CFN heredoc escaping pain.
- **Smaller UserData**: Drops well below the 16K limit forever, no future surprises.
- **Bonus**: `bootstrap.sh` can be run on a *running* deploy-host to update tools without replacing the instance.

This is the same logic that drove yesterday's "move helpers into scripts/deploy-host/" refactor. Same principle, applied bigger.

## My recommendation

**Do Option B, plus heavy `set -x` and step logging in `bootstrap.sh`**. The strategic refactor turns "wait 10 minutes for CFN" into "edit script, ssh in, re-run, see error in 30 seconds." Debugging becomes lightning fast.

Want me to do that? Or do you prefer the tactical patch? Either way, also bump to `t4g.small` while we're at it?
