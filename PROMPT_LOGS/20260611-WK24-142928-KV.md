## Proposed jumpbox CFN template

Lives at `cloudformation/migration/cf-migration-jumpbox.yaml`, deployed via dedicated make targets that hard-code `--profile ZoningInfoAdmin` so there's zero risk of accidentally deploying it into sandbox.

**Resources:**

1. **`AWS::EC2::Instance`** — t4g.micro, latest Ubuntu 24.04 ARM64 (find via SSM Parameter `/aws/service/canonical/ubuntu/server/24.04/stable/current/arm64/hvm/ebs-gp3/ami-id`), 8 GB gp3 root, in `subnet-85e938dc`
2. **`AWS::IAM::Role`** + instance profile, with `AmazonSSMManagedInstanceCore` (mandatory) and cross-account write to a designated sandbox S3 bucket (for ferrying dumps)
3. **`AWS::EC2::SecurityGroup`** with **no `SecurityGroupIngress`** at all. Outbound = AWS default (all open) so SSM Agent + S3 + RDS + SSH all flow naturally.
4. **UserData** that does:
   - `apt update && apt install -y mysql-client-core-8.0 awscli`
   - SSM Agent is already in Ubuntu — no install needed
5. Parameters: `KeyPairName` (optional, for emergency direct SSH if SSM ever fails), `MigrationS3Bucket` (the sandbox bucket we'll dump to)

**Stack name:** `cf-migration-jumpbox`. Standalone — not part of `deploy-all*`.

**Make targets:**
```
make deploy-migration-jumpbox     # uses --profile ZoningInfoAdmin
make ssm-migration-jumpbox        # opens SSM session — quick way to get a shell
make destroy-migration-jumpbox    # when migration is done
```

## One operator-manual step we can't avoid

The jumpbox will need to SSH into D8-Pls for rsync. That means D8-Pls needs to authorize the jumpbox's SSH key. Path of least friction:

1. After jumpbox boots: SSM into it, generate an ED25519 key pair, print the public key
2. **You** add that pubkey to D8-Pls's `~/.ssh/authorized_keys` via your existing SSH access (the `23.123.213.179/32` Hudson rule or the Worxco prefix list — whichever covers your current IP)
3. Jumpbox can now rsync

There's no way to skip this without giving D8-Pls an SSM-enabled IAM role too — which is a bigger change to prod than we want for a migration aid.
