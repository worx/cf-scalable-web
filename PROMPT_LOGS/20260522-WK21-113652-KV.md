feat(deploy-host): system-wide /etc/tmux.conf

Operators run long-lived tmux sessions on the deploy-host (e.g.
make serve-drupal-local). Default tmux is unfriendly on a Mac:
emacs-keys in copy-mode (no Ctrl-u/Ctrl-d or /?n vi-style search),
and a 2000-line scrollback that gets blown out by a single deploy-allX.

This config sets:
- mouse on (trackpad scroll works without entering copy-mode)
- history-limit 50000 (full deploy-allX runs stay in buffer)
- mode-keys vi (Ctrl-u/d half-page, Ctrl-b/f full-page, / and ? search)
- escape-time 10ms (vim/nvim feel snappier)
- 1-indexed windows/panes
- Status bar shows the active Worxco env on the right (mirrors zsh RPROMPT,
  so the env stays visible inside long-running panes where the shell
  prompt isn't on screen)
- bind r reloads the config in-session

System-wide at /etc/tmux.conf rather than per-user — applies to any
operator who SSMs in, including future users.

docs(dns): sub-zone delegation runbook + per-env publishing

Operator-facing manual runbook for giving each env a real public URL
(e.g. sandbox.envs.zoning-info.com) instead of the raw ALB DNS name.

Pattern A (recommended): one-time delegation of envs.zoning-info.com
from the parent zone (in the org-parent account, via zikvanderw
profile) to a sub-zone in the workload account. After that, all
day-to-day operations stay in the workload account.

Pattern B (alternative): per-env sub-zones (sandbox.zoning-info.com,
test-new.zoning-info.com, etc.). Documented for the rare case where
URLs need to live directly under the apex.

Key callouts in the doc:
- Two different hosted zone IDs are involved (envs sub-zone vs ALB
  regional constant). Confusing them gives a misleading AccessDenied —
  this gotcha + symptom now in a prominent warning at the top of the
  publishing section.

  cf-app-drupal stack from the DrupalSiteName param). PHP-FPM workers
  pick it up at boot via configure-php.sh, so changes require an ASG
  instance refresh on the running fleet (sample commands included).

Not automated by CFN because:
- The parent zone lives in a different AWS account from the workloads
- Different deployments will use entirely different domains
- One-time setup steps are operator decisions, not stack inputs
