### 10:14:00 - Response

Good question — and a perceptive observation. The two approaches are functionally equivalent, but ours uses an extra layer of indirection:

**Yours (production):**
```nginx
location ~ \.php$ {
  fastcgi_pass Aegir-FPM-Scaling-0639abcb64cf61e3.elb.us-east-1.amazonaws.com:9083;
}
```

**Ours:**
```nginx
# /etc/nginx/conf.d/upstream-php.conf  (generated at boot by configure-nginx.sh)
upstream php83 {
  server sandbox-nlb-fadbc87b8e311ff8.elb.us-east-1.amazonaws.com:9083;
  keepalive 16;
}

# /etc/nginx/shared/sites-enabled/drupal.conf
location ~ \.php$ {
  fastcgi_pass php83;
}
```

`php83` is an **nginx-internal alias** for an upstream block, *not* a DNS name — that's why `ping php83` fails outside nginx. Inside nginx, when nginx sees `fastcgi_pass php83;`, it looks up the upstream block, picks a server (we only have one), and connects. The actual hostname it connects to is the same NLB DNS your production uses.

**Why we did it this way:**
1. The `configure-nginx.sh` boot script reads the NLB endpoint from SSM (`/sandbox/nlb/endpoint`) and writes the upstream once. Then every site's `drupal.conf` (and future site vhosts) just says `fastcgi_pass php83;` — no DNS name embedded in each vhost.
2. `keepalive 16` declares connection pooling — worth ~the performance of one connect+TLS handshake per request.
3. Easier multi-site: when you add a second site, you don't have to copy/paste the NLB DNS into each vhost; you just reference `php83` or `php74`.

**However** — two things worth being aware of:

1. The `keepalive 16` directive only takes effect if the location block also has `fastcgi_keep_conn on;` (which our drupal.conf does NOT). So keepalive is currently dead code for us. Not harmful, but you could either add `fastcgi_keep_conn on;` to enable it, or drop the `keepalive 16` line. (Doesn't explain the 502s — we'd see connection-pool errors, not RSTs.)

2. **Last night's manipulation might have left the upstream pointing at a specific PHP box IP**, not the NLB. I had you run a `sed -i` that replaced `server sandbox-nlb-...:9083` with `server <PHP_IP>:9000` to bypass the NLB, then tried to restore by calling configure-nginx.sh.
