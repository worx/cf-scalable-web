Good question — and understanding this matters for how you maintain the script long-term.

## Short answer

`backup-state.sh` is **transient** on the jumpbox. It gets downloaded fresh from S3 every time it needs to run (either by `make backup-jumpbox` or by the nightly EventBridge scheduler). There's no persistent file that "lives" on the jumpbox.

## The three copies

| Location | Role | Persistence |
|----------|------|-------------|
| **`migration/scripts/jumpbox/backup-state.sh`** (your Mac / repo) | **Source of truth** — where you edit | Version-controlled in Git |
| **`s3://sandbox-migration-kv-worxco/scripts/jumpbox/backup-state.sh`** | **Distribution copy** — what actually runs at 2 AM | Persists in S3 (7-day lifecycle on the `scripts/` prefix means it's refreshed frequently) |
| **`/opt/migration/scripts/jumpbox/backup-state.sh`** (on jumpbox) | **Transient landing zone** — downloaded, executed, then just sits there | Overwritten every invocation; wiped on instance replacement |

## Flow for each trigger

**`make backup-jumpbox`** (on-demand):
1. `_ssm-run-jumpbox.sh` uploads `_common.sh` + `backup-state.sh` from your local repo → S3 (`scripts/` prefix)
2. Sends SSM command telling jumpbox to download those from S3 and run
3. Jumpbox downloads to `/opt/migration/scripts/`, chmods +x, runs it

**EventBridge Scheduler at 2 AM** (nightly):
1. Assumes execution role
2. Sends SSM command to jumpbox — the same download-and-run sequence from CFN template's `Input` block
3. Jumpbox downloads whatever's currently in S3 → runs it

**Notice: the scheduler doesn't touch your local repo.** It reads from S3.

## The one gotcha to remember

**If you edit `backup-state.sh` locally and don't run `make backup-jumpbox` afterward, the S3 copy is stale — and the next 2 AM scheduled run will execute the OLD version.**

Simple fix: after any edit to `backup-state.sh` (or `_common.sh`), run `make backup-jumpbox CONFIRMED=yes` once. It uploads the fresh version to S3 AND takes a backup with the new code. Both concerns handled in one shot.

## Why we set it up this way

This pattern is deliberate. The alternatives all have downsides:

| Alternative | Downside |
|-------------|----------|
| Embed the script in CFN template Input | CFN redeploy for every script change; template becomes bloated and diff-noisy |
| Bake into UserData | Same — UserData changes trigger replacement risk |
| Git-clone on jumpbox | Adds Git dependency + auth for private repos; more moving parts |
| Bake into a custom AMI | Requires an AMI build pipeline for every script change |

**Download-from-S3 gives us:**
- Fast iteration (edit → `make backup-jumpbox` → done, no CFN)
- Small CFN template
- Scripts are always fetched fresh — no drift between "what's in S3" and "what runs"
- No AMI churn
- Same pattern for on-demand and scheduled — one code path

## If you want to be extra defensive

Optional future improvement: add a lightweight `make sync-scripts` target that just uploads scripts to S3 without running a backup. Useful if you edit the script but don't want to trigger a manual backup right now. ~5 lines of Makefile.

But honestly, running `make backup-jumpbox CONFIRMED=yes` after each edit is a fine workflow. It's fast (~15 seconds), and the fresh backup is always useful anyway.

## Bottom line

- The script "lives" in your local repo (source of truth)
- S3 is the runtime distribution point (scheduler reads here)
- Jumpbox has a transient copy that doesn't matter across reboots
- Anytime you edit, `make backup-jumpbox` keeps everything in sync
