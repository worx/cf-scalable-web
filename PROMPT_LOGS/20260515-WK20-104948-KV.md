Postgres still rejects the new password — that confirms **`ALTER USER` didn't take effect on the postgres side.** Either it wasn't run yet, or it ran but didn't commit.

Run this on the **deploy-host** and paste the output. The key thing is the `SELECT` at the end will fail if the user doesn't actually have the password you just set — useful test in one shot:

```bash
# On the deploy-host
NEW_PASS=$(aws secretsmanager get-secret-value \
  --secret-id worxco/sandbox/drupal/db-password \
  --region us-east-1 --query SecretString --output text)
echo "Setting drupal_user password to: $NEW_PASS"

# Run ALTER USER and immediately verify by trying to connect AS drupal_user
psql-env <<EOSQL
ALTER USER drupal_user WITH PASSWORD '$NEW_PASS';
\du drupal_user
EOSQL

echo ""
echo "=== Now try connecting as drupal_user with that password ==="
PGPASSWORD="$NEW_PASS" psql \
  "host=sandbox-postgres.c256s2w8gam2.us-east-1.rds.amazonaws.com port=5432 dbname=drupal user=drupal_user sslmode=require" \
  -c "SELECT current_user, current_database(), now();"

unset NEW_PASS
```

What we're looking for:

1. **`ALTER ROLE`** in the first block's output — confirms the SQL ran. If you see an error instead, the SQL failed.
2. **`\du drupal_user`** — shows the user exists.
3. **The standalone `psql` connection** — proves the postgres user now accepts the new password. If THIS works, your PHP box should work too.

If the `psql` test fails with "password authentication failed" too, then the `ALTER USER` either silently failed or there's something weirder (different database, role inheritance, etc.). If it works here but PHP still fails, then there's some encoding issue between the secret and how PHP retrieves it (unlikely but possible).
