### 14:19:37 - Response

Yes — this is exactly the right approach. Brilliant call.

Let me think through what it gives us and what to build:

## What local prototyping validates

The Drupal-install-specific logic — which is where 90% of bugs live — is testable locally without touching FSx, RDS, or the cloud round trip:

- ✅ Composer dependency resolution (`composer create-project drupal/recommended-project`)
- ✅ Drush `site:install` flow and arguments
- ✅ settings.php template generation
- ✅ File permissions, ownership, marker files
- ✅ Idempotency / remove-and-reinstall cycle
- ✅ `make` target dispatch (local vs SSM)

## What we still test in cloud later

Cleanly separated and small surface area:

- FSx mount path (already validated by `mount-env`)
- RDS connection from PHP-FPM (boot script env vars)
- ALB → NGINX → NLB → PHP-FPM request path
- TrustedHostPatterns / hostname routing

## On the volume question

**No need to expand or add a volume**. Current 19GB disk has 13GB free. Drupal codebase + vendor dir + SQLite DB ≈ 300MB. Tons of headroom.

## Proposed approach — two scripts, shared core

Better than mode flags, IMO. Two distinct scripts let each be perfectly tailored:

```
scripts/deploy-host/
├── bootstrap.sh                     (existing)
├── install-drupal-local.sh          NEW — SQLite, local /var/www/local/drupal
├── remove-drupal-local.sh           NEW — wipe local install
├── install-drupal.sh                LATER — full cloud (RDS + FSx)
└── remove-drupal.sh                 LATER — drop DB + wipe FSx
```

The local pair becomes our iteration playground. Once the install/remove cycle is dialed in (5-10 minute iterations vs 75-minute deploys), we port the validated logic to the cloud variants — at that point most of the work is just swapping connection params and changing the install path.

## Concrete first steps

1. **Tonight (small change)**: add `sqlite3` + `php8.3-sqlite3` to `bootstrap.sh` so they're there on the next bootstrap. For *today's* deploy-host, just `sudo apt install -y sqlite3 php8.3-sqlite3` to add them in place.

2. **Build `install-drupal-local.sh`** — does the full Drupal 11 install via composer + drush against SQLite at `/var/www/local/drupal`. Generates a settings.php (without the env-var dynamics yet — those come in the cloud variant). Drops a `.installed` marker.

3. **Build `remove-drupal-local.sh`** — `rm -rf /var/www/local/drupal`. Idempotent. Done.

4. **Build `make install-drupal-local` / `make remove-drupal-local`** — dispatch via SSM if running from Mac, run directly if on deploy-host.

5. **Iterate**: `make install-drupal-local` → see what breaks → fix → `make remove-drupal-local && make install-drupal-local`. Cycle in minutes.

6. **When dialed in**: build cloud variants by porting validated logic.

## Bonus benefit

Local Drupal also gives us a sandbox for module testing, theme development, etc. — independent of the production-ish infrastructure. Worth keeping around even after cloud install works.

Want me to start with steps 1-4? Should be straightforward given everything we already have. Then you can play with `make install-drupal-local` while `deploy-allX` runs (or doesn't run — entirely your call) in parallel.
