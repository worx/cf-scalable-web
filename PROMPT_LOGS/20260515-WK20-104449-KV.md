### 10:45:31 - Response

Looking at your output: PASS_LEN=32 in the test means `$DRUPAL_DB_PASS` in your shell *was* set to **something** 32 chars long. But your shell probably still has the **old** `X~pz...` exported from your earlier test — `export` persists across commands in the same shell, and you only refreshed the secret in Secrets Manager, not in your environment.

More important question though: **did you run the `ALTER USER ... WITH PASSWORD ...` step on postgres?** Secrets Manager has the new value `zMb9NjXhDpG,G6etCpf7A_qtNMBlg,yk` but PostgreSQL has no idea about that — secrets and DBs aren't linked. Updating one without the other is exactly the state we're trying to fix.

**Step-by-step, this time on the deploy-host (where `psql-env` works with master creds):**

```bash
# On the deploy-host (NOT the PHP box)

# Read the CURRENT secret value (already the new one)
NEW_PASS=$(aws secretsmanager get-secret-value \
  --secret-id worxco/sandbox/drupal/db-password \
  --region us-east-1 --query SecretString --output text)
echo "Will set drupal_user to: $NEW_PASS"

# Apply it to postgres
psql-env <<EOSQL
ALTER USER drupal_user WITH PASSWORD '$NEW_PASS';
SELECT current_user, now();
EOSQL

unset NEW_PASS
```

Then **back on the PHP box, refresh the shell env to the current secret value**:

```bash
# Re-pull fresh — overwrites whatever was in $DRUPAL_DB_PASS
export DRUPAL_DB_PASS=$(aws secretsmanager get-secret-value \
  --secret-id worxco/sandbox/drupal/db-password \
  --region us-east-1 --query SecretString --output text)

# Sanity check — first 2 chars should be 'zM' now, not 'X~'
echo "Now starts with: ${DRUPAL_DB_PASS:0:2} (length ${#DRUPAL_DB_PASS})"

php /var/www/drupal/web/_test.php
```

Expected: `PG_OK: ["drupal","drupal_user","2026-05-15 ..."]`

The second error you saw — `no pg_hba.conf entry for host "10.200.9.212"... no encryption` — is a red herring. It's PHP's pg_connect retrying without TLS after the first attempt failed auth, and RDS refusing non-TLS connections. Once the password matches, neither error will appear.

Run those two blocks (deploy-host ALTER USER, then PHP box re-test) and paste the new output.
