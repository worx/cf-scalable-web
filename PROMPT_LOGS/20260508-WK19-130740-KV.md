### 13:08:18 - Response

Good instinct, but I'd hold off until *after* `install-drupal` runs. Here's the reasoning:

## Right now your deploy-host is in a known-good state

- All bootstrap-installed tools work (already validated)
- The `sudo install` you just did is harmless — it copied newer versions of two scripts directly from the repo. No state to "lose."
- FSx is mounted, peering works, /etc/hosts has the FSx entry, RDS connection is verified.

That's exactly the surface `install-drupal` will exercise. Doing destroy/deploy now adds 10-15 minutes (destroy-peering, destroy-deploy-host, deploy-deploy-host, deploy-peering, mount-env, refresh-env-config) without testing the thing we actually came to test.

## The smarter validation

Run `install-drupal ENV=sandbox` first. Two outcomes:

| Outcome | Next step |
|---------|-----------|
| **Install succeeds** | Now you have a working Drupal install AND a known issue (helpers don't auto-update on git pull). Destroy/redeploy the deploy-host as final clean-state validation. The fresh bootstrap will install the now-current helper scripts on its own — no `sudo install` needed. The Drupal install itself survives (it lives on FSx, not on the deploy-host). |
| **Install fails** | We learn something specific. Fix it, push, manually `install` the updated script, retry. Once it works, *then* destroy/redeploy. |

This way:
- Forward momentum on the actual test
- Clean-state validation still happens, just in a more productive order
- The Drupal install itself is durable — destroy-deploy-host doesn't touch FSx or RDS

## On the helper-update gotcha

Your earlier point — needing `sudo install` after every `git pull` — is real friction. After install-drupal validates, I'll add `make install-helpers` so you can do `git pull && make install-helpers` to sync helper changes without going through bootstrap.sh. Tiny target, big win.

For now: `make install-drupal ENV=sandbox` and let's see what happens.
