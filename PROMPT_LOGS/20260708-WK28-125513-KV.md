Great — that confirms it's the URL parser mangling the encoded password. Not "broken out of INTO" exactly (pgloader's grammar doesn't support that in 3.6.x), but there are two clean paths. Let's try the simpler one first.

## The likely culprit

pgloader's URL parser in 3.6.x has a known quirk: it either (a) doesn't decode the `%XX` sequences at all, or (b) double-decodes them. Either way, `jq @uri`'s output ends up NOT matching the actual password string when it reaches PostgreSQL.

The clean fix is to **skip the URL-encoding entirely and put the raw password in the URL**. That works as long as the password doesn't contain the URL structural characters `@`, `:`, `/`, `#`, `?`. Most AWS-generated passwords are safe alphanumerics.

## Fix #1 — the simple one to try first

Edit `scripts/tt.sh`. Change this line:

```bash
SANDBOX_PW_ENC=$(printf '%s' "$DRUPAL_DB_PASS" | jq -sRr @uri)
```

to this:

```bash
SANDBOX_PW_ENC="$DRUPAL_DB_PASS"
```

(Leave the variable name alone — no need to rename. Or comment out the old line and add the new one right below.)

Save, rerun. Three possible outcomes:

| Result | Meaning | Next |
|--------|---------|------|
| pgloader connects and starts copying | Done — password had no URL-structural chars, and jq's over-encoding was the problem | Sit back and watch the migration |
| Same `password authentication failed` error | Password has some special char that IS getting mangled, or contains a URL structural char | Go to Fix #2 below |
| Different error (URL parse fails at pgloader startup) | Password contains `@`, `:`, `/`, or `#` — URL is now unparseable | Go to Fix #2 below |

## Fix #2 — if #1 doesn't work (env-var fallback)

pgloader 3.6.x lets you omit the user/password from the URL and pick them up from `PG*` environment variables. This bypasses URL parsing entirely for those fields.

Change the INTO line to:

```
INTO postgresql://${DRUPAL_DB_HOST}:${DRUPAL_DB_PORT}/${DRUPAL_DB_NAME}?sslmode=require
```

(no `user:password@` in there)

And add these exports before the `pgloader` invocation:

```bash
export PGUSER="$DRUPAL_DB_USER"
export PGPASSWORD="$DRUPAL_DB_PASS"

pgloader --no-ssl-cert-verification /tmp/zinew.load 2>&1 | tee /tmp/zinew-pgloader.log

unset PGUSER PGPASSWORD
```

The password never touches URL parsing — pgloader picks it up as a literal env var and passes it directly to the underlying cl-postgres client.

## Quick sanity check while you decide

Before either fix, this quick command tells us what kind of password we're dealing with — will help predict whether Fix #1 will work:

```bash
sudo bash -c 'source /etc/worxco/envs/sandbox && \
  echo "length: ${#DRUPAL_DB_PASS}"; \
  printf "%s" "$DRUPAL_DB_PASS" | grep -o "[@:/#?%%+&]" | sort -u | tr "\n" " "; echo'
```

That prints the password length and, on the second line, any URL-structural characters it contains (nothing printed = safe for raw-in-URL). If line 2 is empty, Fix #1 will almost certainly work. If it prints any characters, jump straight to Fix #2.
