## The honest answer about complexity

It's reasonable, but the orchestration code gets denser. You'll have ~10 background processes in flight, each writing to its own log file, with `wait $PID` barriers between dependency tiers. The Phase 2 in today's `deploy-allX` already does this for 3 parallel tracks — we'd extend the pattern to ~10 tracks.

**Three real complications worth flagging up front:**

1. **Visibility.** With 8-10 background processes, raw interleaved output is unreadable. Each track writes to its own tempfile; the orchestrator prints periodic heartbeat ("[8m 22s] vpc=done iam=done storage=running db=running cache=done images=running [3/8 done]"). We already do this in today's Phase 2 — just scaled up.

2. **Error propagation.** If VPC fails at T=2, AMI builds are still happily running and won't fail until T=28. Need a "watchdog" pattern: when any critical-path track fails, signal the others to abort. Cleanest way: use a shared sentinel file (`touch $TMPDIR/abort` from the failing track; other tracks check it periodically).
