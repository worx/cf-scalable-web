## About your one-liner

Your `make backup-jumpbox CONFIRMED=yes; make deploy-jumpbox; sleep 180; make ssm-jumpbox` sequence was a good instinct, but note that the `;` chain runs each step regardless of the previous exit code. Once the deploy failed with error 254, the `sleep 180 && make ssm-jumpbox` still tried to run — you'd have SSM'd into the OLD instance (not a rebuilt one) and everything would look fine, which could be misleading.

For a "stop on any failure" chain, use `&&`:

```bash
make backup-jumpbox CONFIRMED=yes && make deploy-jumpbox && sleep 180 && make ssm-jumpbox
```

But given the deploy takes variable time (sometimes UPDATE_COMPLETE is slower), you might prefer explicit waiting:

```bash
make backup-jumpbox CONFIRMED=yes && \
make deploy-jumpbox && \
aws --profile ZoningInfoAdmin cloudformation wait stack-update-complete \
    --stack-name cf-migration-jumpbox --region us-east-1 && \
sleep 60 && \
make ssm-jumpbox
```

That waits for the CFN update to actually complete before proceeding, then a small buffer for UserData to finish, then SSM in.
