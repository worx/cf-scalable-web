## Design Approach

Before I write code, here's the design I'm proposing — push back if anything looks wrong:

### Two-step stack approach

**Step 1**: Add route table exports to `cf-vpc.yaml` (small modification, no resource changes)

**Step 2**: New `cf-deploy-peering.yaml` with parameters for the default VPC bits (since they're not in any stack)

### Cross-VPC SG references — the gotcha

Within a VPC, you can write SG rules like `SourceSecurityGroupId: sg-12345`. **Across peered VPCs**, AWS does support this — but the SG rule on the project side needs the deploy-host SG ID (which we have from import). It works as long as peering exists.

### What `cf-deploy-peering.yaml` will contain

```yaml
Parameters:
  EnvironmentName:        # sandbox/staging/production
  DeployHostStackName:    # default: "cf-deploy-host"
  DefaultVpcId:           # the 172.31.x.x VPC
  DefaultVpcCidr:         # default: 172.31.0.0/16
  DefaultRouteTableId:    # main route table of default VPC

Resources:
  # The peering connection itself
  VPCPeeringConnection:
    Type: AWS::EC2::VPCPeeringConnection
    Properties:
      VpcId: !ImportValue '${EnvironmentName}-vpc-id'
      PeerVpcId: !Ref DefaultVpcId

  # Route in default VPC: project CIDR -> peering
  RouteFromDefault:
    Type: AWS::EC2::Route
    Properties:
      RouteTableId: !Ref DefaultRouteTableId
      DestinationCidrBlock: !ImportValue '${EnvironmentName}-vpc-cidr'
      VpcPeeringConnectionId: !Ref VPCPeeringConnection

  # Routes in project VPC route tables: 172.31.0.0/16 -> peering
  # (one per route table — public + 1-3 private depending on AZ count)
  RouteFromProjectPublic:
    Type: AWS::EC2::Route
    ...
  RouteFromProjectPrivate1:
    Type: AWS::EC2::Route
    ...

  # SG ingress rules on project-side SGs
  FSxIngressFromDeployHost:
    Type: AWS::EC2::SecurityGroupIngress
    Properties:
      GroupId: !ImportValue '${EnvironmentName}-fsx-sg-id'
      IpProtocol: tcp
      FromPort: 2049
      ToPort: 2049
      SourceSecurityGroupId: !ImportValue '${DeployHostStackName}-sg-id'
  FSxPortmapper:    # TCP 111

  ValkeyIngressFromDeployHost: # TCP 6379

Outputs:
  PeeringConnectionId:
  PeeringStatus:
```

## Three Questions Before I Write Code

**Q1: Default VPC parameters — pass them in, or look them up?**

Two options:
- **Pass in**: User runs `aws ec2 describe-vpcs --filters Name=isDefault,Values=true ...` once, puts results in parameter file. Simple, deterministic.
- **Look up**: Use a CloudFormation custom resource (Lambda) to discover the default VPC at deploy time. More magical, more code.

I lean **pass in**. The default VPC almost never changes for an account. Worth more setup once than carrying a Lambda forever.

**Q2: Route table exports — modify cf-vpc.yaml, or add via a new tiny stack?**

The peering needs to add routes to `PublicRouteTable` and `PrivateRouteTable1/2/3`. I see two paths:
- **Add exports** to existing `cf-vpc.yaml` (small change, requires re-deploying the VPC stack — should be a no-op, but always slightly risky)
- **Have peering stack look up route tables** via custom resource (avoids touching cf-vpc.yaml)

I lean **add exports**. It's the right home for them, and a "describe outputs" change to a deployed stack is genuinely safe (no resource updates).
