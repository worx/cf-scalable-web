### 16:19:28 - Response

SSM Session Manager doesn't trigger the MOTD system — that's a login shell feature (SSH, console login). SSM starts a non-login shell, so `/etc/update-motd.d/` scripts never run.

A few options:

1. **Add it to the shell profile** — Put a `cat /etc/motd` or run the script from `/etc/profile.d/` (which SSM *does* source)
2. **Add it to `.bashrc`** — The ubuntu user's `.bashrc` runs on every SSM session

The simplest fix is to add it to the existing `/etc/profile.d/deploy-host-env.sh` that we're already creating. Just append a call to the motd script at the end. But that would show it on every subshell too, which is noisy.

Better approach: add it to `/home/ubuntu/.bashrc` with a guard so it only runs on interactive sessions. Want me to update the template, or do you want to test it manually on the current instance first to see what works?
