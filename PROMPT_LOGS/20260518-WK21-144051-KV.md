```
feat(make): check-drift — flag CFN templates committed but not deployed

The 2026-05-13 → 2026-05-15 debug arc was driven in part by an undeployed
cf-iam.yaml edit that sat in git for 6 days while we chased the symptom
("empty DB password in www.conf") through multiple wrong-trees. This
target would have caught it on day one.

scripts/check-cfn-drift.sh compares per template:
  - last-commit timestamp from `git log -1 --format=%ct -- <template>`
  - deployed stack's LastUpdatedTime (or CreationTime) from
    `aws cloudformation describe-stacks`
Status per template:
  OK            committed time <= deployed time
  DRIFT         local newer than deployed (red — needs deploy)
  UNCOMMITTED   working tree differs from git (yellow — commit first)
  NEW           template in repo, no matching stack in AWS (cyan)
  MISSING       expected template not in repo

Exit codes are CI-friendly: 0 = nothing to do, 1 = drift or uncommitted.

First live run on sandbox found a real drift (cf-app-drupal.yaml — the
/var/www/drupal install-path SSM update from commit ee319d5 was never
deployed). The tool justified itself in its first execution.

Implementation notes:
  - bash 3.2 compatible (no associative arrays) so it works on macOS
    out of the box, not just on the deploy-host's bash 4+.
  - iso_to_epoch tries GNU `date -d` first, falls back to BSD `date -j`
    — same script runs on Mac and Linux.
  - Color sequences only when stdout is a TTY (log-friendly).
  - Stack-name mapping kept in sync with the *_STACK Makefile variables;
    deploy-host is special-cased (no env prefix).

Limitations (worth iterating later):
  - Doesn't check cloudformation/parameters/*.json — RecipeVersion bumps
    and similar parameter changes won't show as drift until a downstream
    template re-references them.
  - Doesn't detect a reverted-then-reapplied edit if the final state
    happens to match the deployed state. Use git history for that.

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
```
