## Suggested sequence when your current deploy finishes

**1. Wait for the current CFN deploy to complete.** Check with:

```bash
aws --profile ZoningInfoAdmin cloudformation describe-stacks \
    --stack-name cf-migration-jumpbox --region us-east-1 \
    --query 'Stacks[0].StackStatus' --output text
# Wait until it says UPDATE_COMPLETE
```

**2. Confirm the new jumpbox auto-restored your state.** SSM into it:

```bash
cd migration
make ssm-jumpbox
# Once in the SSM session:
sudo su - root
ls -la /root/.ssh/            # should show id_ed25519, id_ed25519.pub, etc.
cat /root/.ssh/id_ed25519 | head -2   # confirms it's your key, not a fresh one
tail -50 /var/log/cloud-init-output.log | grep -i restore   # UserData restore log
```

**3. Then run `make deploy-jumpbox` again** — this time using the SSM parameter default. Two possible outcomes:

- **Best case:** SSM param resolves to the SAME AMI that was just deployed → CFN sees no change → no-op. Deprecation reminders stop because you're on current AMI.
- **Also fine:** SSM param resolves to a NEWER AMI → CFN replaces the instance AGAIN → UserData auto-restore fires again → same state carries forward.

Either way, deprecation reminders stop and you're on a maintained pattern.

The system worked exactly the way it was designed to when I got this wrong. Backup was in place, auto-restore in the new UserData, S3 lifecycle already refactored.
