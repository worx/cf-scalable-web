### 14:43:08 - Response

Aurora MySQL 8's default collation is `utf8mb4_0900_ai_ci` — a MySQL 8.0-specific collation family that MariaDB never adopted. The dump file has this baked into every `CREATE TABLE` and probably several `SET NAMES` statements too.

## Diagnostic first — what utf8mb4_0900_* variants are in the dump?

The `0900_ai_ci` isn't the only one that might be there. Aurora could have `0900_as_ci`, `0900_as_cs`, `0900_bin`, etc. Let's see the full list:

```bash
grep -oh 'utf8mb4_[0-9a-z_]*' /var/www/mysql/zinew.sql | sort -u
```

## Fix — sed the dump to MariaDB-compatible collations

Once we know what's in there, we do an in-place sed with backup. Assuming the diagnostic shows the usual suspects, this is what it'll look like:

```bash
# Make a backup first (2.4G file, this'll take a bit on FSx)
sudo cp /var/www/mysql/zinew.sql /var/www/mysql/zinew.sql.bak

# Replace the MySQL-8-only collations with MariaDB-friendly equivalents
sudo sed -i \
  -e 's/utf8mb4_0900_ai_ci/utf8mb4_unicode_ci/g' \
  -e 's/utf8mb4_0900_as_ci/utf8mb4_unicode_ci/g' \
  -e 's/utf8mb4_0900_as_cs/utf8mb4_unicode_cs/g' \
  -e 's/utf8mb4_0900_bin/utf8mb4_bin/g' \
  /var/www/mysql/zinew.sql

# Verify no 0900 collations remain
grep -c 'utf8mb4_0900' /var/www/mysql/zinew.sql
# Should print: 0
```

## Verify and retry

```bash
# Drop and recreate the empty database (clean slate)
sudo mysql <<'SQL'
DROP DATABASE IF EXISTS zinew;
CREATE DATABASE zinew CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
GRANT ALL PRIVILEGES ON zinew.* TO 'worxco'@'127.0.0.1';

SQL

# Re-run the restore
time mysql -h 127.0.0.1 -u worxco -p'localscratchpass' zinew < /var/www/mysql/zinew.sql
```

## Why this substitution is safe

| MySQL 8 collation | Behavior | MariaDB replacement | Impact |
|---|---|---|---|
| `utf8mb4_0900_ai_ci` | Unicode 9.0, accent-insensitive, case-insensitive | `utf8mb4_unicode_ci` (Unicode 4.0, ai_ci) | Sort order for exotic characters may differ subtly; for Drupal English/Latin content, functionally identical |
| `utf8mb4_0900_bin` | Binary | `utf8mb4_bin` | Identical semantics |

For a Drupal migration this is completely fine — Drupal doesn't rely on the specific `_0900_` sort semantics. And remember, this collation only matters inside the *scratch* MariaDB; once pgloader is done, PostgreSQL has its own collation system entirely.

## Watch for other MySQL-8-only bits

If the restore fails again after the collation fix, it might be one of these Aurora-isms (all fixable with more sed):

- `DEFAULT_ENCRYPTION='N'` — table option unknown to MariaDB
- `ROW_FORMAT=DYNAMIC` — supported but sometimes flagged
- `/*!80000 ... */` conditional comments — MariaDB may misinterpret
- `SET @@SESSION.sql_require_primary_key` — MySQL-8-only
