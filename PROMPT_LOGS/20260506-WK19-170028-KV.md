**2. Local Mac vs deploy-host execution — your design is exactly right**

The cleanest pattern: a shell script `scripts/install-drupal.sh` that does all the actual work, designed to run *on the deploy-host*. Then the Make target detects context:

```makefile
install-drupal:
    @if [ -f /etc/worxco/deploy-host-marker ]; then \
        bash scripts/install-drupal.sh $(ENV); \
    else \
        # On Mac — verify deploy-host running, send via SSM
        aws ssm send-command --instance-ids ... --document AWS-RunShellScript ...
    fi
```

If deploy-host is stopped, the SSM command will fail cleanly with "instance not running." User then runs `make start-deploy-host` (you have this) and retries. **Question for you**: should `make install-drupal` auto-start the deploy-host if it's off, or just fail with a clear message? The marker file `/etc/worxco/deploy-host-marker` would be created in the deploy-host's UserData so the script reliably knows where it's running.

**3. Database privilege handling — solvable, no voodoo**

Here's the actual flow:

```bash
# Step 1: Fetch RDS master password from Secrets Manager
MASTER_PW=$(aws secretsmanager get-secret-value \
    --secret-id worxco/sandbox/rds/master-password \
    --query SecretString --output text)

# Step 2: Use master to create Drupal's database + restricted user (one-time)
PGPASSWORD="$MASTER_PW" psql -h $RDS_ENDPOINT -U postgres <<EOF
  CREATE DATABASE drupal_sandbox;
  CREATE USER drupal_user WITH PASSWORD '$DRUPAL_PW';
  GRANT ALL PRIVILEGES ON DATABASE drupal_sandbox TO drupal_user;
EOF

# Step 3: Drush install runs as drupal_user (limited to its own DB)
drush site:install standard \
    --db-url="pgsql://drupal_user:$DRUPAL_PW@$RDS_ENDPOINT/drupal_sandbox" \
    --account-name=admin --account-mail=$ADMIN_EMAIL ...
```

We already have the master password in Secrets Manager from the `cf-database` stack. The script fetches it briefly, runs three SQL statements, then never uses it again. Drupal's runtime credentials are scoped to its own database only. This is the standard pattern — no special privileges needed beyond what we already have.

**5. Secrets Manager rotation — let me give you the *why* now**

The short version:

| Reason | Explanation |
|--------|-------------|
| **Damage control** | If a credential leaks (compromised laptop, accidentally committed key, exfiltrated by malware), rotation limits how long the leaked credential remains usable. A 30-day rotation means the worst case is 30 days of unauthorized access, not "until someone notices." |
| **Compliance** | ISO 27001, SOC 2, PCI-DSS all require periodic credential rotation. Audits will ask. Some are explicit (PCI-DSS = 90 days for some credentials). |
| **Resilience testing** | If your code only works with one specific password forever, you have a single point of failure. Rotation forces your code to handle credential changes gracefully — which is also what protects you when something *forces* a rotation (incident response). |
| **Standard hygiene** | Same reason you change locks when an employee leaves. |

**Cost considerations**: Rotation isn't free. Each rotation invokes a Lambda that performs the rotation logic (typically ~$0.001 per invocation, negligible). The bigger consideration is **rotation can briefly interrupt connections** — your app must handle "wait, did that change?" gracefully. RDS rotation in particular needs careful handling because long-lived connections might fail mid-query during the rotation window.

For a sandbox/test environment, rotation is overkill. For production with sensitive data, it's table stakes.
