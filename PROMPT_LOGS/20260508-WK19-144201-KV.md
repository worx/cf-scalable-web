fix(bootstrap): Auto-mount step works with 'export KEY=value' env file format

The auto-mount step extracted FSX_DNS via:
  fsx_dns=$(grep '^FSX_DNS=' "$envfile" | cut -d= -f2-)

Two days ago we changed refresh-env-config to write 'export KEY=value'
lines (so 'source' alone exports vars to child processes). The grep
pattern '^FSX_DNS=' no longer matches the new lines (which start
with 'export '). With set -euo pipefail:

  - grep returns 1 (no match)
  - pipefail makes the pipeline return 1
  - set -e kills the script

Bootstrap died silently right before the MOTD step. Symptom on the
freshly deployed instance: /etc/motd does not exist. The bootstrap
log showed:

  fsx_dns=
  on_err
  === BOOTSTRAP FAILED
  === exit code: 1
  === last step: Auto-mount FSx for any deployed environments

Fix: replace grep+cut with sourcing the env file in a subshell. Works
for both 'KEY=value' and 'export KEY=value' formats. Also added
'|| true' as defense against future bash version changes where command
substitution failures under pipefail+set -e might propagate.

  fsx_dns=$(. "$envfile" 2>/dev/null; echo "${FSX_DNS:-}") || true

Memory gets the gotcha for future reference. The pattern of "grep
KEY=value lines" is fragile when the file format may change between
'KEY=value' and 'export KEY=value' — sourcing is robust.

Recovery on the running deploy host (no need for full destroy/deploy):
  cd ~/projects/cf-scalable-web && git pull
  sudo bash scripts/deploy-host/bootstrap.sh

bootstrap.sh is idempotent — re-running picks up where it failed
without harm. After this, /etc/motd will exist, FSx will be mounted,
and /etc/fstab will have the persistence entry.
