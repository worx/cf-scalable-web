feat(deploy-allX): parallelize Phase 2 + pre-create FSx layout (P0 phase-ordering fix)

Closes the P0 phase-ordering bugs from the 2026-05-19 destroy-deploy
debug arc. Two chicken-and-egg gone, ~45 min of wall clock reclaimed.

PROBLEM (recurring on every greenfield deploy-allX):
  - Phase 4 launched nginx ASG before /fsx/nginx existed on the
    freshly-created FSx. configure-nginx.sh's NFS submount failed,
    `set -e` killed the script BEFORE it wrote
    /etc/nginx/conf.d/upstream-php.conf. Result: nginx served only
    the catch-all `server _`, every site request 404'd.
  - Phase 4 also launched PHP ASGs before Phase 5 deployed RDS.
    configure-php.sh saw no /<env>/rds/endpoint in SSM, logged a
    warning, skipped writing the env[] block to www.conf. PHP-FPM
    workers booted with no DRUPAL_* env vars; every request 500'd.

FIX:
  - New scripts/init-fsx-layout.sh + `make init-fsx-layout ENV=<env>`
    target. SSM-dispatches to deploy-host: refresh-env-config → use-env
    → mkdir -p /var/www/nginx/sites-enabled /var/www/sites. Idempotent.
    Run before any compute deploy that targets a freshly-created FSx.

  - deploy-all Phase 1 now includes init-fsx-layout (after deploy-storage).
    Gracefully degrades if deploy-host isn't running yet (prints clear
    warning telling operator to run it manually + instance-refresh nginx
    once deploy-host is up).

  - deploy-allX restructured for both correctness and speed:
      Phase 1 (sequential, ~5 min): VPC, peering, IAM, storage, init-fsx-layout
      Phase 2 (PARALLEL, ~25 min driven by AMI long pole):
        (a) Image Builder + AMI builds
        (b) deploy-database
        (c) deploy-cache
      Phase 3 (sequential, ~5 min): update-ami-params + deploy-compute
      Phase 4 (~1 min): deploy-app-drupal
      Phase 5 (~5 min): install-drupal-remote
      Phase 6 (seconds): smoke-test-drupal
    Total: ~30 min vs. previous ~75 min.

  - By the time Phase 3 (compute) runs, all of /fsx/nginx existing,
    /<env>/rds/endpoint published, AMI IDs in SSM — every prerequisite
    nginx and PHP boot scripts touch. configure-nginx.sh and
    configure-php.sh remain `set -e`-strict; they fail fast and
    visibly if prereqs are ever missing again (right behavior for
    surfacing future regressions, not silently producing broken hosts).

  - deploy-allX no longer wraps deploy-all internally. Each is its own
    fully-explicit orchestrator. deploy-all is the "infrastructure only,
    no data layer, no Drupal" use case; deploy-allX is "zero to serving
    Drupal in one command."

DOCS:
  - docs/ARCHITECTURE.md "deploy-allX Phase Ordering" section rewritten
    with new 6-phase table + a before/after explainer of what changed.
  - TODO.md P0 entry marked resolved with date + summary.
