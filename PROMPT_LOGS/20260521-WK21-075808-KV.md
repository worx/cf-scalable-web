fix(use-env): create /fsx/nginx via /var/www mount BEFORE attempting submount

Bug I introduced yesterday in commit ec72b49 (two-mount design): when
use-env runs against a freshly-created FSx volume, the second mount
(/fsx/nginx → /etc/nginx/shared) fails because the NFS server hasn't
got /fsx/nginx yet. Symptom from this morning's deploy-allX:

  mount.nfs4: mounting fs-...:/fsx/nginx failed,
  reason given by server: No such file or directory

Worse: init-fsx-layout (whose JOB is to create /fsx/nginx) calls
`use-env` first to mount FSx — so use-env's failure prevents
init-fsx-layout from running. Chicken-and-egg.

Fix: between the two mounts, use-env now mkdir's /fsx/nginx via the
just-mounted /var/www access path. NFS server sees the directory
before the second mount attempt, so the submount succeeds.
Idempotent (mkdir -p), re-runs are no-ops.

This puts the "ensure my mounts can succeed" responsibility on
use-env itself — the right place. init-fsx-layout continues to do
its broader layout work (creating /var/www/sites etc.) but doesn't
have to bootstrap the nginx subdir as a precondition for its own
prerequisite.
