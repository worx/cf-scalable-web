### 14:28:07 - Response

You called it. Here's the actual architectural state I just confirmed:

**The `/health` endpoint *only* exists inside the Drupal vhost.**

- `compute-alb-sandbox.json` sets `HealthCheckPath: /health` (line 5)
- The baseline nginx config (`image-builder/configs/nginx/nginx.conf`) has **no** `/health` location, and **no `default_server`** at all — it just `include`s vhosts from `/etc/nginx/shared/sites-enabled/*.conf`
- That `/health` location is defined inside `drupal.conf` (in `publish-drupal-vhost.sh:79-83`), which only gets written to FSx by `install-drupal.sh`

So a freshly-booted nginx with no `drupal.conf` on FSx **literally has no http server listening on port 80**. Every ALB health check fails until install-drupal runs. Today's "ELB system health check failure" replacements in the ASG scaling history weren't a Drupal-install-timing race — they were the inevitable consequence of nginx having nothing to serve. The system has been brittle since the day `/health` was put inside `drupal.conf`.

**Two-part fix, both worth doing:**

**Part 1 — Move `/health` into the baseline nginx.conf (the actual decoupling).**

Add a `default_server` to `image-builder/configs/nginx/nginx.conf` that:
- Returns `200 OK` on `/health` unconditionally
- Returns `404` on everything else (or a "site not configured yet" placeholder)
- Has no PHP dependency, no FSx vhost dependency, no Drupal dependency

Then strip `default_server` from `drupal.conf`'s `listen 80` (it'll match by `server_name` instead). Both vhosts can serve `/health`; ALB hits the baseline's, real users hit Drupal's. Cost: AMI rebuild on next deploy (~25 min via `build-amis-if-needed`). After this, the nginx fleet is fully healthy from second one of boot, regardless of whether Drupal exists.

**Part 2 — Make Phase 5/6 of deploy-allX non-fatal.**

Restructure the end of `deploy-allX` so:
- Phases 1–4 (infra) determine deployment success/failure
- Phases 5–6 (install-drupal + smoke-test) run as a best-effort appendix
- If they fail, summary says "Infrastructure healthy at ENV=<env>. Drupal install needs attention — re-run with `make install-drupal-remote ENV=<env>`." Exit 0.

This matches your stated intent: Drupal is in `deploy-allX` purely for laziness, not as a hard dependency.

**Sequencing suggestion:**

Part 2 is a 10-minute Makefile edit; can be done immediately. Part 1 is a real nginx.conf change + AMI bump (RecipeVersion increment) and needs an AMI rebuild to take effect — first deploy-allX after the change picks it up. Both ship in the same commit, but the AMI rebuild is what actually fixes the underlying brittleness.

**One more thing I'd flag:** Even with Part 1 in place, `install-drupal.sh` currently writes `drupal.conf` with `listen 80 default_server`. Two `default_server` blocks for port 80 on the same nginx is a config error (`nginx -t` fails). So we'd also need to update `publish-drupal-vhost.sh` (in the same commit) to drop `default_server` and rely on `server_name` matching. Pure Host-header routing, which is what we actually want.

Want me to do both parts in a single commit?
