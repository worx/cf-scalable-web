### 11:12:02 - Response

# **DRUPAL IS RUNNING.**

That's a real Drupal exception, not an infrastructure error. The actual problem:

```
AssertionError: The file ... (/var/www/sandbox/drupal/web/core/modules/announcements_feed/announcements_feed.info.yml) do not exist.
```

Drupal's **compiled service container has cached `/var/www/sandbox/drupal/` as the app root**, but on FSx the files are actually at `/var/www/drupal/` (no `sandbox/` subdirectory). The container was probably compiled when Drupal was running from a different path, and the stale cache is now referencing a path that no longer exists.

Two ways to fix:

**Option A (quick — just rebuild the cache):**

```bash
# From the deploy-host
refresh-env-config sandbox     # picks up the new DB password
. /etc/worxco/envs/sandbox     # source it into THIS shell

cd /var/www/drupal              # actual install location
drush --root=/var/www/drupal/web cache:rebuild

# Confirm it's happy
drush --root=/var/www/drupal/web status
```

**Option B (cleaner — nuke the compiled cache directory):**

```bash
# On the deploy-host
sudo rm -rf /var/www/drupal/web/sites/default/files/php/*
# OR
sudo rm -rf /var/www/drupal/web/sites/default/files/php/container
```

Either way, the next request rebuilds the container against the current paths.

Then re-curl:
```bash
curl -sv -H "Host: drupal-sandbox.test" "http://sandbox-alb-95496877.us-east-1.elb.amazonaws.com/" --max-time 15 | head -40
```

You should get a real Drupal page (probably a redirect to `/user/login` or the install screen).

**Underlying mystery for follow-up** (not blocking — just worth understanding): why was the container compiled with `/var/www/sandbox/drupal/` as the root when nothing in the install path actually IS that? My best guess is `install-drupal.sh` originally invoked drush with `--root=/var/www/sandbox/drupal/web` (because SSM `/sandbox/drupal/install-path` returns `/var/www/sandbox/drupal`), but the actual `composer create-project` landed the codebase at `/var/www/drupal/`. A path mismatch between drush's `--root` flag and where the code actually lives would explain how the container got compiled with the wrong path baked in.

Worth fixing in install-drupal.sh so the next env install doesn't have this footgun. But for sandbox right now: clear the cache and test.
