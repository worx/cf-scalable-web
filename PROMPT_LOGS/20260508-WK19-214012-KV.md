```bash
# Drill into the in-flight build and show which phase/step it's on
for P in $(aws imagebuilder list-image-pipelines \
    --query "imagePipelineList[?contains(name,'php-fpm') && contains(name,'sandbox')].arn" \
    --output text); do
  NAME=$(echo "$P" | awk -F/ '{print $NF}')
  ARN=$(aws imagebuilder list-image-pipeline-images --image-pipeline-arn "$P" \
    --query 'reverse(sort_by(imageSummaryList, &dateCreated))[0].arn' --output text)
  echo "=== $NAME ==="
  aws imagebuilder get-image --image-build-version-arn "$ARN" \
    --query 'image.{State:state.status,Reason:state.reason,Build:imageScanningConfiguration,Started:dateCreated}' \
    --output json
done
```

Two practical concerns to rule in/out:

1. **Test phase can be slow** — once the build phase finishes, Image Builder runs the test phase (which `apt-get update` + verify each package is present). That alone is 2-3 min on top of build.
2. **Build instance type / region capacity** — sometimes capacity is constrained at off-peak hours and the launch step itself dawdles.

If you want to see actual stdout from the running components, the build-host writes logs to S3:
```bash
BUCKET=$(aws ssm get-parameter --name /sandbox/s3/image-builder-bucket --query Parameter.Value --output text)
aws s3 ls "s3://$BUCKET/build-logs/" --recursive | tail -20
```

I'd give it another 10 min before getting suspicious. If at 30 min it's still BUILDING, then something's wrong (most likely a hung apt-get or a bash script that's stuck in a loop).
