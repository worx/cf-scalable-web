### 12:40:03 - Response

Correct read. **The instance WILL be replaced** — and the culprit is exactly what got us with the jumpbox: `ImageId: RequiresRecreation: Always` with `ChangeSource: DirectModification`.

The SSM parameter `/aws/service/canonical/ubuntu/server/24.04/stable/current/arm64/hvm/ebs-gp3/ami-id` currently resolves to `ami-02c4144237becae44`, which differs from whatever AMI was current when your deploy-host was last built. Ubuntu ships new AMIs regularly. CFN sees `ImageId` changed → mandatory replacement.

## Your plan is exactly right

Backup first, then deploy, then verify auto-restore:

```bash
# 1. Manual backup while the current deploy-host is healthy
make backup-deploy-host

# 2. Confirm it landed in S3
make list-deploy-host-backups
# should show: config/deploy-host-latest.tar.gz  + 1 timestamped archive

# 3. NOW deploy — will replace the instance, but state auto-restores
make deploy-deploy-host

# 4. Wait ~3-5 min for CFN update + UserData completion
aws cloudformation wait stack-update-complete \
    --stack-name cf-deploy-host --region us-east-1

# 5. Verify auto-restore worked
make ssm-deploy-host
# once in the new instance:
sudo ls -la /root/.ssh/
sudo cat /home/ubuntu/.aws/config
sudo cat /home/ubuntu/.aws/credentials   # should be the STUB
sudo tail -100 /var/log/deploy-host-bootstrap.log | grep -i restore
```

That's the whole design working as intended — the replacement pain is fully absorbed by auto-restore.

## Heads-up for future `make deploy-deploy-host` calls

**This isn't a one-time issue.** As long as `UbuntuAMI` uses the SSM-parameter type, every future `make deploy-deploy-host` triggers replacement any time Ubuntu has published a newer AMI (which is roughly monthly).

You have two choices:

| Option | Trade-off |
|--------|-----------|
| **Keep SSM param** (current) — always fresh AMI | Every deploy potentially replaces the instance. Auto-restore covers it. Same choice we made for jumpbox. |
| **Pin AMI** — hardcode an AMI ID | Instance stays put on `make deploy-deploy-host`. You manually update the AMI ID when you want a newer one. Less magic, more control. |

For jumpbox (short-lived, per-client-migration ephemeral), we deliberately kept the SSM param + accepted the replacement risk because we'd wired up auto-restore to make it painless.

For deploy-host (long-lived operational infrastructure), **the same reasoning applies now that auto-restore is in place.** Security updates roll in automatically; state carries forward automatically. My recommendation: keep the SSM param. Same pattern as jumpbox.

But it's your call and easy to flip if you'd rather pin.

## After tonight — one more habit worth building

Just as `git status` before `git commit` becomes reflex, **change-set preview before `make deploy-deploy-host` should be reflex** (since we know this target might replace the instance). Consider a wrapper:

```make
preview-deploy-host:  ## Show what deploy-deploy-host WOULD do (no changes made)
	@aws cloudformation deploy \
		--stack-name $(DEPLOY_HOST_STACK) \
		--template-file cloudformation/cf-deploy-host.yaml \
		--capabilities CAPABILITY_NAMED_IAM \
		--no-execute-changeset 2>&1 | tail -5
	@echo "$(CYAN)Get the change set ID above, then run:$(NC)"
	@echo "$(CYAN)  aws cloudformation describe-change-set --change-set-name <arn> \\$(NC)"
	@echo "$(CYAN)      --query 'Changes[*].ResourceChange.{Action:Action,LogicalId:LogicalResourceId,Replacement:Replacement}'$(NC)"
```

Not needed tonight — you already did the manual preview. But worth adding as a follow-up.
