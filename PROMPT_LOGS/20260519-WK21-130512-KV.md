fix(install-drupal-remote): wait for bootstrap marker, drop SSM timeout cap

Two failures from the first deploy-allX run, now fixed:

1) executionTimeout=1800 was SHORTER than SSM's default (3600). The
   first run hit the 30-min cap mid-bootstrap and was killed. Removed
   the override entirely so we get the default 1-hour window.

2) The auto-bootstrap self-heal branch was the wrong call. Bootstrap
   is a 15-30 min apt + composer + extension install — it belongs in
   cf-deploy-host's UserData (one-time, at instance boot), not inside
   our SSM dispatch. Running it inline coupled two unrelated lifecycles
   and blew through the timeout. It also triggered git's "dubious
   ownership" check because bootstrap.sh ran some git ops as root in
   the ubuntu-owned repo.

The new design:
  - Wait for /etc/worxco/deploy-host-marker (written by bootstrap.sh
    near the end). Up to 25 min @ 30s poll. Also requires use-env in
    PATH as a belt-and-braces check (marker is written before the
    use-env install in current bootstrap.sh).
  - Sync to latest committed code (handles the case where the deploy-host
    was instantiated from an older commit than what's local).
  - `git config --system --add safe.directory` so git ops work regardless
    of which user (root vs ubuntu) invokes them.
  - refresh-env-config → use-env → make install-drupal as before.

Also improved the local poll loop:
  - 90 iterations × 30s = ~45 min wall clock (was 60 × 10s = 10 min)
  - If we hit the cap, exit 2 with a "still running, check via" message
    rather than claiming failure. The SSM command keeps running on the
    host regardless of our local view.

Operational change: install-drupal-remote.sh now ASSUMES the deploy-host
has already been deployed by cf-deploy-host (which auto-bootstraps via
UserData). It does not try to bootstrap pre-existing hosts. If you have
a deploy-host that's missing /etc/worxco/deploy-host-marker, run
bootstrap.sh interactively first.
