## 2. Documentation co-location

I tested — you can put additional files in a skill directory without breaking anything. Claude Code only reads `SKILL.md` and files explicitly referenced from it. A `docs/` folder or a `README.md` next to `SKILL.md` is invisible to the skill execution unless `SKILL.md` tells Claude to read it.

My recommendation for the documentation structure:

**For skills** (already have directories):
```
skills/tk-close/
├── SKILL.md          # Claude-facing instructions (what Claude reads)
├── README.md         # Human-facing documentation (what a staff member reads)
└── diagrams/         # Optional Mermaid diagrams
    └── workflow.mmd
```

**For hooks** (currently loose files — restructure into directories):
```
hooks/
├── prompt-logger/
│   ├── prompt-logger.sh
│   └── README.md
├── time-tracker/
│   ���── time-tracker.sh

```

This means updating the hook paths in `settings.json` from `hooks/time-tracker.sh` to `hooks/time-tracker/time-tracker.sh`, but that's a one-time change.

**The key principle**: everything needed to understand and hand off a skill or hook lives in its own directory. Someone receiving `tk-close/` gets SKILL.md (the logic), README.md (the explanation), and any diagrams. Self-contained.
