### 14:36:19 - Response

**Sub-zone exists. Real ID:** `Z03149452QE09GQX23F48` (from `envs.zoning-info.com.` in your ZI-Sandbox account).

Set:
```bash
ENVS_ZONE_ID=Z03149452QE09GQX23F48
```

and re-run the change-resource-record-sets command. It'll succeed this time.

**On your other question — variables not persisting:** your instinct was right. Shell variables don't persist across command runs unless `export`'ed (and even then, only into child processes of the shell that did the export). Re-defining `ENV`, `ALB_DNS`, `ALB_ZONE`, and `ENVS_ZONE_ID` in a fresh shell is exactly correct. The error wasn't from a missing variable — it was from a wrong value in one variable. The other three were defined and used correctly.

The doc should be louder about this. Want me to add a "find your envs sub-zone ID" snippet at the top of the publishing section, so future-you (or anyone else following this) doesn't repeat the mix-up?

The snippet would be a one-liner that does the listing for them:

```bash
ENVS_ZONE_ID=$(aws --profile ZI-Sandbox route53 list-hosted-zones \
  --query "HostedZones[?Name=='envs.zoning-info.com.'].Id | [0]" --output text | sed 's|/hostedzone/||')
echo "envs sub-zone: $ENVS_ZONE_ID"
```

Plus a callout box explaining the two-zone-IDs gotcha — that the alias record refers to two *different* hosted zone IDs (yours, and AWS's ALB-region constant), and confusing them gives a misleading AccessDenied.
