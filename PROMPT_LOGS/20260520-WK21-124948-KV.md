fix(bootstrap): MOTD double-print on SSH was pam_motd + zshrc.d, not zprofile

Previous fix theorized that the second MOTD print came from Ubuntu's
/etc/zsh/zprofile re-sourcing /etc/profile via `emulate sh -c '...'`.
That theory was wrong: live inspection showed /etc/zsh/zprofile is
just a comment header on this Ubuntu, doesn't source /etc/profile,
and `zsh -l -c 'true'` from the host produces zero output.

The actual cause: `ssh sandbox` goes through Ubuntu's sshd → PAM
stack, which includes pam_motd. pam_motd prints /etc/motd
automatically BEFORE the user's shell starts. Then sshd spawns the
shell, which runs /etc/zsh/zshrc.d/00-motd.zsh, which prints
/etc/motd AGAIN. Different access path triggers a different
mechanism — `aws ssm start-session` doesn't go through PAM (the SSM
agent forks the shell directly), so the previous single-MOTD
behavior was specific to that path.

Fix: skip our shell-side MOTD print when SSH_CONNECTION is set
(sshd sets this on every SSH login; absent in SSM Session Manager
sessions). One env-var check covers both cases:

  - ssh login: PAM prints once, our scripts skip → 1 MOTD total
  - SSM session: PAM not involved, our scripts print → 1 MOTD total

Applied symmetrically to both /etc/profile.d/00-worxco-motd.sh and
/etc/zsh/zshrc.d/00-motd.zsh.
