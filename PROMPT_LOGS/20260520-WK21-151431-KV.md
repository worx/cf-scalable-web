```
feat(make): admin-login-url — drush user:login via SSM; mention in endpoints

Adds the Drupal-native lockout-recovery path: `drush user:login` (aka
`drush uli`) generates a single-use 24-hour login URL that takes you
straight to the admin profile page, bypassing the password entirely.
Standard Drupal flow anyone who's run Drupal at the CLI level reaches
for instinctively.

New script scripts/admin-login-url.sh:
  SSM-dispatches to deploy-host, sources env, invokes
    drush user:login --uri=http://<ALB-DNS>
  with --uri pre-expanded at outer-script time so drush gets the
  literal URL (previously embedded $ALB_DNS as inner-shell var, which
  drush received as a literal string — fell back to URI=localhost).
  Refuses if deploy-host's current-env doesn't match the requested
  ENV (same pattern as clear-drupal-cache.sh).

  Extracts the URL via grep so the output is just the URL itself plus
  a tip — easy to paste into a browser.

New Make target: `admin-login-url ENV=<env>`.

`make endpoints` Drupal-admin block now mentions this as an
alternative to the password-fetch flow:

  Or (Drupal-native, no password needed):
    make admin-login-url ENV=sandbox
  (runs 'drush user:login' on the deploy-host via SSM; prints a
   single-use 24h URL that drops you straight onto the admin page.
   Standard Drupal lockout-recovery flow — no AWS knowledge needed.)

Two flows now coexist: AWS-style (fetch secret from Secrets Manager,
type into login form) and Drupal-style (request a one-time URL via
drush). Same Drupal admin account either way; the Drupal-style path
matches what Drupal engineers' muscle memory expects.

Drive-by fix in same commit: the endpoints "Or" message originally
used backticks around `drush user:login`. Make passes the recipe to
the shell, where backticks trigger command substitution — the shell
tried to execute drush locally on the Mac, the line silently broke,
and the whole "Or" block didn't render. Replaced with single quotes.
```
