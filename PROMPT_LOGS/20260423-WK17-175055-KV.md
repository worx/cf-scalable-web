### 17:51:10 - Response

Great questions. Let me answer each:

## 1. Auto-clone on first boot

Yes, absolutely. We know the repo URL, it's not going to change. The UserData can do `git clone`, `cp .env.example .env`, the whole setup. The deploy host would boot ready to `make deploy-all` with zero manual steps.

## 2. GitHub key granularity

GitHub has two types of SSH keys:

- **Account-level SSH keys** (Settings → SSH Keys) — access to every repo your account can see. This is what I suggested earlier. Too broad.
- **Deploy Keys** (per-repo, Settings → Deploy Keys) — access to **one specific repository**, read-only by default (you can optionally grant write). This is exactly what you want.

A deploy key scoped to `worx/cf-scalable-web` with read-only access means: can clone and pull that one repo, nothing else. Can't push, can't see other repos, can't do anything else on the account.

## 3. Automate with `gh` CLI

Partially. The `gh` CLI can add deploy keys:

```bash
gh repo deploy-key add key.pub --repo worx/cf-scalable-web --title "deploy-host" --read-only
```

But `gh` needs to authenticate first, which requires a PAT or browser OAuth. So the *initial setup* (adding the key to GitHub) is a one-time manual step from your laptop. After that, the key lives in Secrets Manager and the deploy host pulls it automatically forever.

The flow would be:
1. **One-time (your laptop)**: Generate key → store in Secrets Manager → add deploy key to repo via `gh` or GitHub UI
2. **Every boot (automatic)**: UserData pulls private key from Secrets Manager → writes to `~/.ssh` → configures git → clones repo → copies `.env`
