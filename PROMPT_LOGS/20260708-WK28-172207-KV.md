## tmux scrollback — you CAN scroll

```
Ctrl+b then [
```

That puts tmux into **copy mode**. Once you're in copy mode:

- Arrow keys / Page Up / Page Down to scroll
- `Ctrl+u` and `Ctrl+d` for half-page up/down
- `g` to jump to top of buffer, `G` to jump to bottom
- `/` to search forward, `?` to search backward, `n` and `N` to jump between matches
- `q` when done

That gets you back the full summary that scrolled off. It only holds whatever's in tmux's history (default ~2000 lines but usually enough).

## About the timing math

pgloader's "total time" is misleading — it's the **sum of per-phase durations**, not wall-clock. During index creation especially, pgloader runs indexes **in parallel** across worker threads on the PostgreSQL side, so the reported "1h 31m for indexes" is the sum of index rebuild times across parallel workers, not the actual clock time you experienced.

Your ~1h 50m wall-clock experience is the truth. The 2h 4m "total" is thread-time accumulation. So it's not lying, but it IS reporting a different metric than you're mentally comparing against.

## About the 12 million / 1.6 numbers

Those numbers you saw are almost certainly:
- **~12 million rows** copied across all tables (the big cumulative total)
- **~1.6 GB** transferred (bytes counter — pgloader measures COPY payload size)

For a 2.4 GB uncompressed MySQL dump, 1.6 GB on the wire to PostgreSQL makes sense — PostgreSQL's binary COPY format is somewhat more compact than MySQL's dump format, and the CAST rules eliminate some overhead (zero-dates → NULL saves bytes, etc).
