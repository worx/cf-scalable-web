### 14:12:32 - Response

Now we know exactly what's happening — and it's not something you can fix at the schema level.

## The actual problem

Look at the last part of your output:

```
information_schema    19
sys                    8
```

There are **27 columns in MySQL's own system schemas** (`information_schema` + `sys`) that use `utf8mb3_tolower_ci`. That collation is used internally in MySQL 8 for case-insensitive matching of identifier names (schema names, table names, column names).

You **can't change these** — they're part of MySQL 8's built-in metadata layer, read-only, baked into the server.

When pgloader queries `information_schema` to introspect your schema (which it must, to plan the migration), MySQL sends back result-set metadata that references collation ID 76 for those columns. qmynd (pgloader's Common Lisp MySQL driver) has an incomplete collation table and doesn't know about 76 → ECASE explodes.

## This is a well-known pgloader-vs-MySQL-8 incompatibility

pgloader 3.6.7's qmynd hasn't been updated to handle several collations MySQL 8 introduced or reassigned. There's no way to work around it at the SQL level — the qmynd driver source itself needs the collation table extended. Not a mistake on our part; a tool limitation.

## Your options

| Option | Effort | Reliability |
|--------|--------|-------------|
| **Switch local scratch DB to MariaDB** | ~10 min | High — MariaDB's information_schema doesn't use utf8mb3_tolower_ci |
| **Use Docker MySQL 5.7 as the scratch DB** | ~15 min (needs Docker) | High — MySQL 5.7 predates the collation |
| **Patch qmynd source and rebuild pgloader** | ~1-2 hours | Fragile, gets undone by apt updates |

**My recommendation: switch to MariaDB.** It speaks the MySQL wire protocol, mysqldump files load into it unchanged, qmynd is happy with it, and it's a first-class Ubuntu package. Same-ish thing, minus the incompatibility.

## MariaDB switch — the steps

```bash
# 1. Stop MySQL (we're throwing away the scratch data anyway — the dump is safe on FSx)
sudo systemctl stop mysql

# 2. Remove MySQL entirely
sudo apt remove --purge -y mysql-server mysql-server-8.0 mysql-server-core-8.0 mysql-client-core-8.0
sudo rm -rf /var/lib/mysql /etc/mysql
# also drop the pgloader-compat.cnf if any leftover
sudo rm -f /etc/mysql/mysql.conf.d/pgloader-compat.cnf

# 3. Install MariaDB
sudo apt update
sudo apt install -y mariadb-server mariadb-client

# 4. Verify (should say something like 10.11.x-MariaDB)
sudo mysql -e "SELECT VERSION();"

# 5. Create the scratch DB, user, and load the dump
sudo mysql <<'SQL'
CREATE DATABASE zinew CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER 'worxco'@'127.0.0.1' IDENTIFIED BY 'localscratchpass';
GRANT ALL PRIVILEGES ON zinew.* TO 'worxco'@'127.0.0.1';
FLUSH PRIVILEGES;
SQL

# 6. Restore
time mysql -h 127.0.0.1 -u worxco -p'localscratchpass' zinew < /var/www/zinew.sql
```

## What you don't need to redo on MariaDB

- **No `default_authentication_plugin` gymnastics** — MariaDB defaults to `mysql_native_password`, which qmynd handles fine
- **No collation surgery** — MariaDB's `information_schema` doesn't use `utf8mb3_tolower_ci`
- **No character_set_server change** — MariaDB defaults are qmynd-compatible

## Then rerun tt.sh

No change to the script needed — the FROM URL still points at `worxco@127.0.0.1:3306/zinew`, and MariaDB listens on the same port with the same protocol.

## Expected outcome

pgloader should get past the schema introspection phase and start actually copying tables. If it hits per-table CAST issues after that, we'll deal with them individually — but those would be data-conversion issues, not driver issues.
