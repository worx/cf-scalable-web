## 2 — Multi-tenancy reshapes the architecture

Confirming this changes a lot of what we've done so far. The current `install-drupal.sh` assumes "one site per environment" — `/var/www/sandbox/drupal/` with database `drupal` on RDS. Multi-tenant means:

| Aspect | Current (single site) | Multi-tenant target |
|--------|----------------------|---------------------|
| **Path** | `/var/www/<env>/drupal/` | `/var/www/<env>/sites/<site-slug>/drupal/` |
| **Database** | one `drupal` db per env | `drupal_<site-slug>` per site (all in the same RDS) |
| **DB user** | `drupal_user` | `drupal_<site-slug>_user` |
| **Hostname** | `drupal-sandbox.test` (one) | `client1.test`, `client2.com`, etc. (many) |
| **PHP version** | one version per env | per-site (you mentioned 4 versions in current prod) |
| **Secrets** | `worxco/<env>/drupal/*` | `worxco/<env>/sites/<site-slug>/*` |
| **NGINX** | one server block | `server_name`-based routing → upstream PHP-FPM port |
| **PHP-FPM** | one pool | per-version pool, possibly per-site config |

The good news: **the foundation we built supports this**. NLB with port-based PHP version routing, FSx as shared codebase storage, deploy-host as control plane — all multi-tenant-friendly. What needs to change is mostly the install/upgrade scripts and NGINX config.

The bad news: it's a real refactor of the Drupal management layer. Worth doing carefully when we get there.

### How the layered scripts evolve

The future multi-tenant install would look like:
```
make install-site SITE=client1 ENV=sandbox PHP=83 DOMAIN=client1.test
```

That'd mean re-tooling around a `site` parameter rather than (just) `env`. The cf-app-drupal stack becomes per-site rather than per-env.

## 3 — Custom modules per tenant

Combined with #2, we're really designing a **multi-site, layered-codebase** system. The standard pattern:

```
worxco-drupal-base/                    ← shared base (CodeCommit repo)
  composer.json                        ← Drupal core + standard modules
                                         every site uses (admin_toolbar,
                                         pathauto, redis, etc.)
  config/sync/                         ← shared base config
  web/modules/custom/worxco_*/         ← shared custom modules
  web/themes/custom/worxco_base/       ← shared base theme

worxco-drupal-site-client1/            ← per-site overlay (CodeCommit repo)
  composer.json                        ← requires worxco-drupal-base@^1.0
                                         + client1-specific modules
                                         + client1-specific theme
  config/sync/                         ← client1 config (overrides base)
  web/modules/custom/client1_*/        ← client1's custom modules
  web/themes/custom/client1_theme/     ← client1's theme

worxco-drupal-site-client2/            ← another per-site overlay
  ...
```

This gives each tenant:
- Auto-inherits base updates (security patches, new shared modules)
- Has freedom to add their own modules, themes, config
- Separate git history (clean audit trail per client)
- Separate deploy/promote lifecycle (one client's upgrade doesn't block another's)
