Two clues in those errors:

1. `password authentication failed for user "dbadmin"` — the password we're sending isn't what RDS expects
2. `no encryption` (in the second alternate-connection-mode failure) — the script is trying both TLS and non-TLS; TLS got auth failure, non-TLS got rejected entirely (RDS requires TLS, which is correct)

The **real** error is the first one. Most likely cause: the secret in Secrets Manager has a **JSON structure** (not raw password), so we're sending the whole JSON blob as the password.

Let me have you check what's actually in the secret without showing the actual password:

```bash
aws secretsmanager get-secret-value \
  --secret-id worxco/sandbox/rds/master-password \
  --query SecretString --output text | head -c 100
```

Then add `| jq .` if you see `{`:

```bash
aws secretsmanager get-secret-value \
  --secret-id worxco/sandbox/rds/master-password \
  --query SecretString --output text | jq 'keys'
```

I'm betting you'll see `["password","username"]` or similar — the AWS-managed-credentials format. That's RDS auto-rotating the password on its own and storing the result as JSON.

Two questions to answer:
1. **What does the secret look like?** (JSON keys, or plaintext starting with the password?)
2. **What does `cf-database.yaml` use?** Could you grep it for `ManageMasterUserPassword` and `MasterUserPassword`?

```bash

```

That'll tell us whether RDS is managing its own secret (in which case the secret format is RDS's standard JSON) or we created the secret ourselves separately.
