feat(make): pause-compute and resume-compute for cost optimization

For short pauses (overnight, between sessions), tearing down the
compute layer is overkill. Scaling the NGINX and PHP ASGs to 0 stops
EC2 hour charges while preserving the stack structure (ALB/NLB still
exist, no CFN churn).

  make pause-compute ENV=sandbox
    For each of <env>-nginx-asg, <env>-php74-asg, <env>-php83-asg:
      - Read current MinSize/MaxSize/DesiredCapacity
      - If non-zero (i.e., currently running), save them to SSM at
        /<env>/asg-pause-state/<asg-name> as 'MIN:MAX:DESIRED'
      - Scale ASG to 0/0/0
    Idempotent: re-running on already-paused ASGs is a no-op.
    Missing ASGs (e.g., compute layer destroyed) are skipped with
    a warning.

  make resume-compute ENV=sandbox
    For each ASG:
      - Read saved state from SSM
      - Restore min/max/desired to the saved values
      - Delete the SSM saved-state param
    Missing saved state = ASG not paused via this target = skip.

Cost win: ~$30-50/mo savings while paused, depending on ASG sizes.
ALB/NLB still cost (~$32/mo combined) — for bigger savings on longer
pauses, 'make destroy-compute ENV=... CONFIRMED=yes' is the next step.

Caveat documented in target comments: 'make deploy-compute' resets
ASG sizes from CFN parameters and would unpause without going through
resume-compute. Orphaned SSM state is harmless.

Help section updated with both targets and the cost note.

## Phase C step 4 — PHP-FPM boot script wiring

Now the real piece. Goal: when an HTTP request comes through ALB → NGINX → NLB → PHP-FPM, the PHP process should serve the Drupal install on FSx.

### What needs to happen on PHP-FPM instances at boot

1. **Mount FSx at `/var/www/<env>`** (same path the deploy-host uses)
2. **Read SSM params** under `/<env>/drupal/*` for db-name, db-user, site-name, install-path
3. **Read DB password from Secrets Manager** at `worxco/<env>/drupal/db-password`
4. **Write env vars into PHP-FPM pool config** (`/etc/php-fpm.d/drupal.conf` or similar):
   ```
   env[ENVIRONMENT_NAME] = sandbox
   env[DRUPAL_DB_HOST]   = sandbox-postgres.xxx.amazonaws.com
   env[DRUPAL_DB_PORT]   = 5432
   env[DRUPAL_DB_NAME]   = drupal
   env[DRUPAL_DB_USER]   = drupal_user
   env[DRUPAL_DB_PASS]   = <fetched value>
   env[DRUPAL_SITE_NAME] = drupal-sandbox.test
   ```
5. **Restart PHP-FPM** so the env vars take effect
6. **NGINX reverse proxy** (the NGINX layer) needs to know about `drupal-sandbox.test` as a server_name pointing to the right PHP-FPM port

### Where this code lives

Per the architecture-reference memory:
> Boot Script Flow (PHP instances)
> 1. UserData calls `/opt/worxco/configure-php.sh` (baked into AMI)
> 2. Script gets IMDS token → region → reads `/environment/name` from SSM
> 3. Mounts FSx: `mount -t nfs4 -o vers=4.1,port=2049 $FSX_DNS:/fsx /var/www`
> 4. Configures Valkey session handler from `/${ENV}/cache/endpoint`
> 5. Configures Postfix SES relay (if SES credentials exist)
> 6. Starts PHP-FPM and health-check NGINX (port 9100)

So `/opt/worxco/configure-php.sh` already exists in the PHP AMI. Steps 1-3 + 4 (Valkey) are already there. What's missing is the Drupal-specific env vars.
