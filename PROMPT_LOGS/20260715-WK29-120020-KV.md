### 12:02:29 - Response

Two great architectural questions. Let me think through both carefully — you may end up disagreeing with parts of my reasoning, which would be useful signal.

## Question 1 — One stack vs two

### The real trade-offs (my honest assessment)

**One stack (your proposal):**

| Pro | Con |
|-----|-----|
| One command to deploy everything | Destroying deploy-host stack destroys the bucket (unless you add `DeletionPolicy: Retain`) |
| Tight coupling reflected in code | `Retain` policy creates orphan buckets that make re-deployment awkward (bucket-already-exists errors on CREATE) |
| One place to look for both concerns | Every template update goes through the same CFN transaction; a bucket-only change also queues an EC2 diff-check |
| Less risk of "user forgot to deploy the second stack" | A rollback due to any resource failure affects both |
| Simpler for new users | Larger blast radius per stack update |

**Two stacks (current):**

| Pro | Con |
|-----|-----|
| Independent lifecycle — destroy deploy-host, keep backups | Two deploy commands, in order |
| Bucket lifecycle changes don't touch EC2 diff-checks | User has to remember the second stack exists |
| Rollback of one doesn't affect the other | Two places to look when modifying deploy-host behavior |
| Standard AWS pattern: stateful storage lives in its own stack from stateless compute | Some coupling is implicit rather than explicit |

### My honest recommendation

**Keep them separate, but paper over the "two commands" cost with a wrapper.** Both of your concerns are legitimate, but each has a fix that's lighter than merging the stacks:

**For "user forgets to deploy both":** add a wrapper target.

```make
deploy-deploy-host-init:  ## Full deploy-host bootstrap: bucket + host + first backup
	@$(MAKE) deploy-deploy-host-backups
	@$(MAKE) deploy-deploy-host
	@echo "$(CYAN)Wait for UserData to finish, then take first backup:$(NC)"
	@echo "$(CYAN)  make backup-deploy-host$(NC)"
```

One command does the whole bootstrap. Removes the "did I forget?" concern.

**For "two places to look":** add cross-references in each template's header comment.

```yaml
# NOTE: state backup lives in cf-deploy-host-backups stack (separate).
# When changing deploy-host state paths, also update:
#   scripts/deploy-host/backup-state.sh BACKUP_PATHS[]
```

**The killer argument for keeping them separate**, honestly: **the whole point of the backup bucket is to survive events that destroy the deploy-host.** If they're in the same stack, and something goes wrong with the stack update forcing a rollback that includes bucket deletion, you've lost the safety net you built the bucket to be.

Even with `DeletionPolicy: Retain`, you'd get orphan buckets on stack destroy, which is confusing operationally. The two-stack pattern models the actual lifecycle relationship: **the bucket is designed to outlive the instance.**

### On your "transitional consolidation" idea

If you decide to merge, your proposed path (deploy separately, back up, then consolidate — forcing a recreation with auto-restore) is a clean migration. Auto-restore was literally designed for this scenario. So merging is safe if you want to; I just don't think you should.

**Bottom line:** I recommend adding the wrapper target and keeping them separate. If you disagree, the merge is doable and safe.
