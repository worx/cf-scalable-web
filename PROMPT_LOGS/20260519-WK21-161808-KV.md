fix(drupal): write nginx vhost to FSx + reload nginx fleet after install

Phase 8 smoke test returned HTTP 404 from nginx because the project had
NO mechanism for creating the nginx vhost that maps the Drupal hostname
to /var/www/drupal/web. The base nginx config has

  include /etc/nginx/shared/sites-enabled/*.conf;

which on nginx instances is the FSx mount $FSX:/fsx/nginx. On a freshly-
created FSx (post destroy-all), that directory is empty — nginx has no
server block matching Host: drupal-sandbox.test, so the request falls
through to nginx's implicit catch-all and gets a 404.

(This worked on 2026-05-15 because the FSx then was preserved across the
debug sessions and someone had hand-written the vhost during diagnosis.
That vhost died with the destroy-all FSx wipe.)

Three changes to fix end-to-end:

1) install-drupal.sh now writes /var/www/nginx/sites-enabled/drupal.conf
   to FSx (visible on nginx hosts as /etc/nginx/shared/sites-enabled/...)
   with a standard Drupal 11 vhost: server_name=$SITE_NAME, root=$INSTALL_DIR/web,
   fastcgi_pass=php83, and the usual security location blocks (private
   files, dotted paths, hidden files).

2) New scripts/reload-nginx.sh + make reload-nginx ENV=<env> — SSM-dispatches
   `nginx -t && nginx -s reload` (graceful, no dropped connections) to
   every instance in <env>-nginx-asg. Same pattern as restart-php-fpm.sh.

3) install-drupal-remote.sh now invokes reload-nginx automatically after
   the install-drupal SSM dispatch succeeds. This makes Phase 7 of
   deploy-allX produce a serving Drupal without further operator action.

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
