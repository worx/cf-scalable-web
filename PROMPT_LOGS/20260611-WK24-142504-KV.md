### 14:26:32 - Response

**Sizing: t4g.micro, not nano.** Reasoning:

- Ubuntu 24.04 base footprint: ~250-300 MB
- SSM Agent: ~50-100 MB resident
- ssm-session-worker per active session: ~30-50 MB
- A `tar | aws s3 cp -` pipeline (for codebase) holds buffers in memory

That puts a single concurrent session in t4g.nano (512 MB) into ~80% utilization with zero headroom for surprises. **t4g.micro (1 GB) is ~$3/month more and gives you comfortable margin.** For a temporary migration aid, the saving on nano isn't worth the babysitting.

Per-hour cost on-demand: nano ≈ $0.0042/hr, micro ≈ $0.0084/hr. Even running it for a full 30 days = ~$6 vs $3. If you destroy after a week of migration work, it's pennies either way.

## How I'd provision it

Since it's prod infrastructure (touched via `ZoningInfoAdmin`), version-controlled and destroy-able, I'd recommend a small CFN template in this repo — but kept separate from the main lifecycle. Something like:

```
cloudformation/
└── migration/
    └── cf-migration-jumpbox.yaml    ← new, prod-account, temporary
parameters/
└── migration-jumpbox.json           ← new
```

With dedicated Make targets that explicitly use the prod profile, so there's no chance of accidentally deploying it into ZI-Sandbox:

```
make deploy-migration-jumpbox    # uses --profile ZoningInfoAdmin
make destroy-migration-jumpbox   # same
```

**What the jumpbox CFN template would create:**

| Resource | Why |
|----------|-----|
| `AWS::EC2::Instance` (t4g.micro, latest Ubuntu 24.04 ARM64) | the conduit |
| `AWS::IAM::Role` with `AmazonSSMManagedInstanceCore` + read access to migration S3 bucket | so SSM Agent connects + jumpbox can write dumps to S3 |
| `AWS::IAM::InstanceProfile` binding the role | required for EC2→IAM |
| `AWS::EC2::SecurityGroup` "migration-jumpbox-sg" | identity for SG rules |
| `AWS::EC2::SecurityGroupIngress` on prod RDS SG, port 3306, source = jumpbox SG | the actual access grant |
| `AWS::EC2::SecurityGroupIngress` on prod Drupal app SG, port 22, source = jumpbox SG | same for rsync |

**No public IP, no inbound SSH from anywhere.** SSM Agent handles everything via outbound TLS.

## What I need from you to write the template

Five things, all visible in the prod console / via `aws --profile ZoningInfoAdmin ec2 describe-*`:

1. **Prod VPC ID** (`vpc-...`)
2. **A private subnet ID** in that VPC where the jumpbox will live — needs to be in the same routing domain as the RDS and the Drupal app servers (likely an "app tier" subnet)
3. **Prod RDS security group ID** — the SG attached to `d8zdb-1.cluster-...`
4. **Drupal app server security group ID** — whatever SG is on the EC2 instances running Drupal
5. **Whether the prod VPC has a NAT Gateway / VPC endpoints** — SSM Agent needs to reach `ssm.us-east-1.amazonaws.com` and `ssmmessages.us-east-1.amazonaws.com`. If the subnet has no internet egress, we need VPC endpoints (3 of them: ssm, ec2messages, ssmmessages). Most prod VPCs have NAT already.

You can fish those out with:

```bash
# VPC + subnets
aws --profile ZoningInfoAdmin ec2 describe-vpcs --query 'Vpcs[].[VpcId,Tags[?Key==`Name`].Value|[0],CidrBlock]' --output table

# SGs filtered by name pattern (will need to adjust the pattern)
aws --profile ZoningInfoAdmin ec2 describe-security-groups --query 'SecurityGroups[?contains(GroupName,`db`)||contains(GroupName,`drupal`)||contains(GroupName,`app`)].[GroupId,GroupName,VpcId]' --output table

# Confirm RDS cluster's actual SG
aws --profile ZoningInfoAdmin rds describe-db-clusters --query 'DBClusters[?DBClusterIdentifier==`d8zdb-1`].VpcSecurityGroups' --output table

# Check for NAT / VPC endpoints
aws --profile ZoningInfoAdmin ec2 describe-nat-gateways --query 'NatGateways[].[NatGatewayId,State,SubnetId]' --output table
aws --profile ZoningInfoAdmin ec2 describe-vpc-endpoints --query 'VpcEndpoints[?contains(ServiceName,`ssm`) || contains(ServiceName,`messages`)].[ServiceName,VpcId]' --output table
```
