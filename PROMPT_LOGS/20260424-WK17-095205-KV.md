### 09:52:25 - Response

Good catch on the FSx mount. The boot script tried but failed — we saw that earlier in the console output: `WARNING: Failed to mount /var/www`. The VPC endpoints gave us SSM access, but the FSx mount needs NFS traffic between the PHP subnets (Tier 3) and the FSx in the data tier (Tier 4), which should be allowed by the security groups we set up. The issue might be DNS resolution for the FSx endpoint, or routing between tiers.

But let me address your bigger question first, because it affects how we fix this:

## Where to Install Drupal

**Use a PHP instance, not a separate box.** Here's why:

1. The PHP instances already have PHP, the extensions, and (once FSx mounts) access to `/var/www`
2. Composer's memory issue is solved with your 2GB swap — you have 1GB RAM + 2GB swap = 3GB total, which is enough for modern Composer
3. You can also tell Composer to use less memory: `COMPOSER_MEMORY_LIMIT=-1 composer create-project drupal/recommended-project /var/www/drupal`
4. A separate box means another AMI, another security group, another FSx mount config — unnecessary complexity

The workflow would be:
1. Fix the FSx mount (debug why it's failing)
2. SSM into a PHP 8.3 instance
3. Install Composer
4. Install Drupal into `/var/www`
5. Since FSx is shared, all PHP instances see the same files immediately

**But** — Composer isn't in the AMI. We'd need to either install it on the running instance (temporary, lost on replacement) or add it to the Image Builder component (permanent, available on every instance). Since you'll need Composer for updates and module installs ongoing, it should probably be in the AMI.

Before any of that though — we need to fix the FSx mount. Want me to debug why it's not mounting? I can SSM into one of the running PHP instances and check.
