feat(deploy-host): Cached endpoint config + refresh-env-config + show-env

Phase B follow-up adding zero-friction endpoint discovery for ad-hoc
shell use. Complements (does not replace) the live-query helpers from
the Phase B baseline.

New files (created by UserData on boot):

  /etc/worxco/envs/<env>
    Sourceable shell config with all endpoint variables for one env:
      DRUPAL_ENV, RDS_ENDPOINT, RDS_PORT, RDS_USER, RDS_DATABASE,
      FSX_DNS, VALKEY_HOST, VALKEY_PORT, ALB_DNS, DRUPAL_PATH
    No secrets — passwords stay in Secrets Manager and are fetched
    on demand by psql-env / valkey-env. Mode 0644, root-owned.

  /usr/local/sbin/refresh-env-config
    Regenerates /etc/worxco/envs/<env> from current SSM values.
    Takes one or more env names (default: sandbox staging production).
    Silently skips environments with no deployed infrastructure.
    Used after destroy/deploy cycles to refresh stale cache.

  /usr/local/bin/show-env
    Pretty-prints the cached config for an env. No sudo, no AWS calls,
    instant. Lists available cached envs when called with no args.

  /etc/sudoers.d/worxco-refresh-env-config
    NOPASSWD entry: ubuntu ALL=(root) NOPASSWD: /usr/local/sbin/refresh-env-config
    Mode 0440 as required by sudo. Lets ubuntu run sudo refresh-env-config
    without typing a password — frictionless ergonomic for the daily user.

UserData runs refresh-env-config once at boot (best-effort) so a fresh
deploy-host comes up with whatever envs are already deployed pre-cached.

MOTD updated to advertise the new commands. Memory updated with the
live-vs-cached distinction and example workflow.

Live vs cached distinction:
  info-env <env>   live SSM query (~3-5s, network)
  show-env <env>   cached file (instant)

Both stay — they serve different needs. Live for "what is true right
now", cached for "give me the endpoint values quickly so I can use them".
