## What pgloader-through-SSM-tunnel actually does to prod

pgloader against MySQL is **read-only** — it never issues writes. But "read-only" doesn't mean "zero impact":

- **Reads it does:** `SHOW CREATE TABLE`, `information_schema` queries, and `SELECT * FROM <table>` on every table.
- **Impact profile:** it's a full-table scan across every table, batched at 1000 rows. On a big table this is a sustained sequential I/O workload.
- **Locking:** InnoDB `SELECT` doesn't lock — MVCC snapshots handle read consistency. So no blocking of production writes.
- **Connections:** one long-lived MySQL connection.
- **Risk:** low for correctness (nothing gets written), moderate for performance if the DB is small/busy. On a client's live app, a sustained scan can noticeably raise CPU and increase read latency for other queries.

So the concern isn't "will pgloader corrupt prod." It's "will pgloader put a scan-shaped load on the client's box while it's serving real users." That's a legitimate concern.

---

## Your idea is the right one — with one refinement

**Yes — mysqldump → S3 → restore-locally → pgloader-against-local is the safe path.** And I'd recommend it not just because you're new to pgloader, but because it's the pattern anyone experienced with pgloader would use for a *first* migration attempt: it lets you iterate.

**One refinement I'd make:** point mysqldump at the **read replica endpoint** (`d8demo-cluster.cluster-**ro**-cmbywichztfj.us-east-1.rds.amazonaws.com`), not the writer. Aurora clusters have a separate read-only endpoint that hits the replica. This gives you:

- Zero impact on the primary write path (the read replica is designed to absorb read load)
- `--single-transaction` mysqldump = REPEATABLE READ snapshot = consistent point-in-time view, no locks
- If for some reason the dump misbehaves, the writer is completely untouched

That's the standard "safe dump from production" pattern for Aurora/RDS.

## Proposed shape of the safer flow

**Phase A — one clean touch of prod (on the jumpbox, ~2-10 min depending on DB size):**

```bash
# On jumpbox, single mysqldump from read replica → S3 (encrypted at rest)
mysqldump \
  --single-transaction \
  --routines --triggers --events \
  --set-gtid-purged=OFF \
  --column-statistics=0 \
  -h d8demo-cluster.cluster-ro-cmbywichztfj.us-east-1.rds.amazonaws.com \
  -u worxco -p"$PROD_PW" \
  zinew \
  | pigz \
  | aws s3 cp - s3://sandbox-migration-kv-worxco/zinew.sql.gz
```

After this runs, **prod is done.** Jumpbox can be shut down or destroyed. Everything downstream is sandbox-only.

**Phase B — install mysql-server on deploy-host, restore locally, run pgloader against the local copy (fully isolated, iterate as many times as you want):**

```bash
sudo apt-get install -y mysql-server
sudo mysql -e "CREATE DATABASE zinew_local CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"
# ... user setup ...
aws s3 cp s3://sandbox-migration-kv-worxco/zinew.sql.gz - | gunzip | mysql -u root zinew_local
```

Then pgloader points at `mysql://worxco:xxx@127.0.0.1:3306/zinew_local` → `postgresql://drupal_user:xxx@sandbox-rds/drupal`. Fully sandbox-internal. You can rerun pgloader 50 times, tune the CAST clauses, drop and rebuild — no prod involvement, no SSM tunnel, no risk.
