## Why deploy-host is harder than jumpbox (and worth doing anyway)

Deploy-host has more moving parts than jumpbox:

| State on deploy-host | Sensitivity | Ideal storage |
|---------------------|-------------|---------------|
| `~/.aws/config` (profile definitions) | Low вЂ” no secrets | S3 tar (fine) |
| `~/.aws/credentials` (ZoningInfoAdmin real keys) | **HIGH вЂ” long-lived cross-account creds** | Secrets Manager, not S3 |
| `/etc/worxco/envs/sandbox` (DB passwords, etc.) | **HIGH вЂ” RDS master creds** | Secrets Manager, or re-derive from CFN each boot |
| `/home/ubuntu/.ssh/` (SSH keys) | Moderate | S3 tar (encrypted-at-rest) вЂ” same as jumpbox |
| Git repo clone at `~/projects/cf-scalable-web/` | None вЂ” version controlled | Not backup material |
| FSx mounts (Drupal codebase at `/var/www/`) | Data вЂ” survives instance replacement naturally | FSx persists independent of EC2 |
| MariaDB scratch DB | Ephemeral (regenerable from S3 dump) | Not backup material |
| Installed packages (pgloader, mariadb-server, session-manager-plugin) | None вЂ” reinstallable via UserData | Bake into UserData/AMI |

**The complication:** the two most-critical items on deploy-host are AWS credentials and DB passwords вЂ” both of which we should NOT just tar to S3 (even encrypted). Those want dedicated secret storage.
