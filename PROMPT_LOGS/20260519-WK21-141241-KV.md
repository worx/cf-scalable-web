### 14:13:45 - Response

Worth a real conversation. Here's what's going on:

**The "new experience"** is AWS's reorganization of the SSM console (rolled out late 2023, evolving since). The substance behind the marketing is **Default Host Management Configuration (DHMC)** + **Quick Setup** automation. They're trying to solve the historical pain point that getting SSM working on a fleet required per-instance IAM roles with `AmazonSSMManagedInstanceCore`, agent installation, the right tags, etc. DHMC short-circuits all that — it uses **instance identity** (via IMDS) instead of per-instance IAM, so every EC2 in the account is automatically SSM-managed without you wiring anything up.

**Quick Setup** is a layer on top: pre-canned CloudFormation StackSets that turn on common SSM features account-wide:
- DHMC (instance identity registration)
- Patch Manager baseline + scan
- Inventory collection (writes to S3)
- State Manager associations for routine ops
- CloudWatch agent install + config

The two stacks you noticed are the implementation:
- **`AWS-QuickSetup-SSM-LocalDeploymentRolesStack`** — IAM roles QuickSetup needs to deploy its configurations *in this account*. Bootstrap layer.
- **`stackset-aws-quicksetup-ssm-rahmy-fa44ebf5-...`** — an instance of one of the StackSets (the `rahmy` is an internal QuickSetup configuration code; the long hex is a StackSet instance ID). To know exactly what it's doing, click into the stack → Template tab.

The new S3 bucket is almost certainly the **inventory data sink** — DHMC + Inventory collects EC2 inventory (installed apps, network config, etc.) and stores it there. Costs are tiny (KB-scale per instance per day) but it does exist.

**Cost outlook:** SSM is mostly free, including all the pieces Quick Setup turns on (DHMC, Patch scanning, State Manager, Inventory collection, Run Command, Session Manager). The only $ is the inventory S3 bucket — pennies a month at our scale.

**My recommendation: roll it back.** Reasons:

1. **We don't need it.** Our existing IAM roles already grant SSM access explicitly, and our compute instances are already SSM-managed via that pattern. DHMC is redundant — and dual-managed instances (IAM role *and* DHMC) can have surprising precedence behaviors when policies conflict.

2. **It introduces implicit behavior.** Patch Manager baseline can flag your AMIs as "non-compliant" without doing anything to them, which is mostly noise — but if anyone later flips on auto-remediation, you'd have surprise OS patches landing on production at AWS's chosen maintenance window. Inventory collection is benign but more moving parts.

3. **It conflicts with our explicit-everything philosophy.** Our entire project is "no hidden state, everything in CFN, all permissions narrow and named." Quick Setup adds StackSets we didn't write, doing things we have to read the templates to understand. That's the opposite direction.

4. **The one thing DHMC would save us** — per-instance SSM IAM boilerplate — isn't even painful in our setup (it's three lines in cf-iam.yaml). Net benefit: near-zero.

**To roll it back cleanly:**
- Open SSM console → Quick Setup → find the active configuration (the `rahmy` one) → Delete configuration. That removes the StackSet instance.
- Then delete `AWS-QuickSetup-SSM-LocalDeploymentRolesStack` (the bootstrap IAM stack) — only after the StackSet instance is gone.
- The S3 bucket may need manual deletion (StackSet deletion sometimes leaves it as a tombstone).

**One caveat for the future:** If you go down the "burn and recreate the AWS account" path from yesterday's destroy-all-residue memory, DHMC would actually be useful in fresh accounts because it gets SSM working without any IAM setup. So it's not "never useful" — just not aligned with how this project is currently designed.
