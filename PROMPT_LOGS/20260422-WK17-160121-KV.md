## tmux

## mosh

Here's the problem: **mosh won't work over SSM.** 

Mosh requires UDP ports 60000-61000 between your client and the server. SSM Session Manager tunnels everything over HTTPS (port 443) through the AWS API — there's no direct UDP path. Mosh fundamentally can't run inside an SSM session because it needs its own UDP channel.

Your options for persistent, resilient sessions:

| Method | Survives disconnect? | Requires SSH? | Requires open ports? |
|--------|---------------------|---------------|---------------------|
| SSM + tmux | Yes | No | No |
| SSH + tmux | Yes | Yes (key) | Port 22 |
| SSH + mosh | Yes (roaming) | Yes (initial) | Port 22 + UDP 60000-61000 |
| mosh over SSM | **Not possible** | — | — |

**My recommendation**: Stick with **SSM + tmux**. You get:
- No SSH keys to manage
- No ports to open (the bastion's security group doesn't even need port 22)
- Session logging via CloudTrail
- tmux handles the persistence across disconnects

The only thing you lose vs. mosh is the instant reconnect when switching networks (closing laptop on WiFi, reopening on different WiFi). But SSM sessions are pretty resilient — they'll hold for a few minutes of disconnect, and tmux preserves everything if the session does drop.

We can still install mosh on the instance for the case where you *do* want to SSH directly (maybe from the office with a stable IP), but it wouldn't be the primary access method.
