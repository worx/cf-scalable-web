### 13:22:04 - Response

Interesting — the picture is more subtle than we thought.

**Key observations from your output:**

- DB default: `utf8mb4_unicode_ci` ✅ (good)
- Table-level defaults using utf8mb3: **none** ✅ (good)
- The 4 columns found use `utf8mb3_general_ci` (**ID 33**) — and **ID 33 IS in qmynd's supported list** (it's in the "Wanted one of" set)

So those 4 report_type columns aren't the actual problem. Something else in the schema is using **`utf8mb3_tolower_ci` specifically** (ID 76 — the missing one) and we haven't found it yet.

The usual suspects for "collation hiding somewhere not in a column":

1. **Triggers** — they store their creation-time `collation_connection` per-trigger
2. **Views** — same
3. **Stored routines / functions** — same
4. **Events**

Aurora sometimes carries these over from legacy history even when the app (Drupal) doesn't create them.

## What to expect / how we fix it

| Where 76 turns up | Fix |
|-------------------|-----|
| Triggers | `DROP TRIGGER` on each — Drupal doesn't use MySQL-side triggers, they can go |
| Views | `DROP VIEW` on each — Drupal doesn't use MySQL views |
| Routines | `DROP PROCEDURE/FUNCTION` on each — same |
| Events | `DROP EVENT` on each — same |
| Nothing at all (all queries return empty) | It's coming from server/session default collation — different fix path, we'd inspect `SHOW VARIABLES LIKE '%collation%'` |

For Drupal specifically, all of these auxiliary objects are safe to drop — Drupal runs entirely on tables + indexes + foreign keys, never touches MySQL-side procs/triggers/views. And we're on the scratch DB, so if we drop the wrong thing, `mysql zinew < /var/www/zinew.sql` restores.
