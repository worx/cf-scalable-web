1. **MySQL server / session defaults** are set to `utf8mb3_tolower_ci`, and qmynd chokes on the connection-level metadata

## What I'm expecting

**Most likely scenario:** some MySQL server-level variable (probably `collation_server` or `character_set_server`) got set to something involving `utf8mb3_tolower_ci` when `mysql-server` was installed via apt — maybe locale-driven. qmynd then trips over it during handshake even though your `zinew` DB is fine.

If that's what we see, the fix is one drop-in config file, similar to what we did for `mysql_native_password`:

```bash
sudo tee -a /etc/mysql/mysql.conf.d/pgloader-compat.cnf > /dev/null <<'EOF'

# Force sane charset/collation defaults so qmynd's older collation
# table doesn't get tripped by unfamiliar IDs like 76 (utf8mb3_tolower_ci)
character-set-server = utf8mb4
collation-server = utf8mb4_unicode_ci
EOF

sudo systemctl restart mysql

# Verify
sudo mysql -e "SHOW VARIABLES LIKE 'collation_server';"
sudo mysql -e "SHOW VARIABLES LIKE 'character_set_server';"
```

**Less likely scenario:** it's actually somewhere in the `mysql` or `sys` schema (some column with legacy collation). We'd handle that differently — pgloader shouldn't normally introspect those, so this would be a weirder pgloader-config issue.

Run the two diagnostic queries and paste the output — the `SHOW VARIABLES` result will make the picture clear immediately.
