**You're right, and the case is sharper than I'd realized once I traced the actual mounts.**

Here's where `/fsx` (the FSx root) currently gets mounted at `/var/www`:

| Where | What mounts | Why |
|-------|-------------|-----|
| `image-builder/components/install-nginx.yaml:168` + `cf-image-builder.yaml:549` | nginx fleet mounts `/fsx` at `/var/www` AND `/fsx/nginx` at `/etc/nginx/shared` | nginx needs the Drupal tree to serve static files (CSS/JS/images via `root /var/www/drupal/web;` in drupal.conf), AND it needs the vhost configs |
| `image-builder/configs/configure-php.sh:63` | **PHP-FPM fleet mounts `/fsx` at `/var/www`** | needs Drupal source files only |
| `scripts/deploy-host/use-env:165` | Deploy-host mounts `/fsx` at `/var/www` AND `/fsx/nginx` at `/etc/nginx/shared` | operator workbench needs both views |
| `cloudformation/cf-storage.yaml:284` | example mount string in the stack output | informational |

**The real security tell is line 63 — PHP-FPM mounts /fsx (the whole volume) just to access the Drupal subtree.** That means PHP workers running as `www-data` have the entire FSx namespace in their reach. A PHP read primitive (and there are many — file disclosure bugs in modules, debug output, careless `file_get_contents()` of user input) can read `/var/www/nginx/sites-enabled/drupal.conf` even though PHP has zero legitimate reason to. The nginx vhost leaks: upstream PHP-FPM pool names, FastCGI params, rate-limit zones, internal hostnames — everything an attacker would want to enumerate before pivoting. **Linux perms can mitigate (chmod the directory) but the layer-1 isolation is your proposed split.**

**Your proposed split (FSx layout):**

```
/fsx
├── www/                    ← mounted at /var/www on nginx + PHP fleets + deploy-host
│   ├── drupal/
│   ├── drupal-private/     ← salt.txt lives here
│   └── drupal-config/
└── nginx/                  ← mounted at /etc/nginx/shared on nginx fleet + deploy-host
    └── sites-enabled/
        └── drupal.conf
```

PHP-FPM workers can mount **only** `/fsx/www` → `/var/www`. From PHP's perspective, `/fsx/nginx` doesn't exist; there's no `..` path from `/var/www` that reaches it. Information-disclosure surface area for nginx config drops to zero on the PHP fleet.

Nginx instances still legitimately need both (static files + vhost configs), but the *architectural separation* still reads cleanly in audit docs: "Drupal application data and nginx configuration are sibling subtrees on FSx, not parent-child."

**Bonus: kills a hack we already have.** Today's use-env has this dance: mount `/fsx` at `/var/www`, then mkdir `/var/www/nginx` (which is really creating `/fsx/nginx`), THEN mount `/fsx/nginx` at `/etc/nginx/shared`. That's because `/fsx/nginx` has to exist before we can mount it. With sibling subtrees, `init-fsx-layout` creates both directories upfront (via a temporary root mount), and use-env never has to touch the FSx root again — it just mounts the two leaf subtrees in either order.
