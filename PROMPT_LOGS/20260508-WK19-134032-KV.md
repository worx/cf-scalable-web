### 13:40:58 - Response

Not Mac-only — it's pure bash. `set -a` (allexport mode) auto-exports any variable assignment that follows, until `set +a` turns it off. So:

```bash
set -a
source /etc/worxco/envs/sandbox
set +a
```

…exports every variable in the file, regardless of whether the file uses `export KEY=value` or just `KEY=value`. **Try this right now** on the deploy-host:

```bash
set -a
source /etc/worxco/envs/sandbox
set +a
export DRUPAL_DB_PASS="$(aws secretsmanager get-secret-value \
  --secret-id worxco/sandbox/drupal/db-password \
  --query SecretString --output text)"
env | grep DRUPAL_DB_HOST            # should now be the RDS endpoint
cd /var/www/sandbox/drupal
vendor/bin/drush status               # DB hostname should NOT be localhost
```

That should work even without updating the helper.

## On the missing `export` in the file

You said `/etc/worxco/envs/sandbox` has no `export` — that's because the file on disk is from **before** my recent change. To get the new format on disk, the helper itself needs updating:

```bash
cd ~/projects/cf-scalable-web && git pull
sudo install -m 0755 scripts/deploy-host/refresh-env-config /usr/local/sbin/refresh-env-config
sudo refresh-env-config sandbox
head -3 /etc/worxco/envs/sandbox     # should now show "export ..."
```

After that, plain `source /etc/worxco/envs/sandbox` works without the `set -a` wrapper.

## Both approaches are fine

Honestly, the `set -a` trick is more elegant — keeps the env file clean as `KEY=value`. Some teams prefer that. Putting `export` in the file is more "guaranteed to work however the user sources it."

For our case, `export` in the file makes it foolproof — any future operator just `source` and it works. No special wrapper to remember.

But the trick you remembered is exactly right and you'll use it again. It's the standard bash idiom for "source a env-file and auto-export."
