### 13:53:33 - Response

Found it. The destroy was **canceled by CloudFormation 1 second after it started** (18:10:54 → 18:10:55):

> `Delete canceled. Cannot delete export cf-deploy-host-sg-id as it is in use by cf-scalable-web-sandbox-deploy-peering.`

The peering stack from yesterday's deploy-allX run (Phase 1 deploys it) imports the deploy-host's security group ID via `Fn::ImportValue`. CloudFormation refuses to delete an exporter while an importer still references it, so the stack rolled back to `CREATE_COMPLETE`. Your `aws cloudformation wait stack-delete-complete` command is now spinning forever because the stack is *not* in a delete-complete state — it's back at CREATE_COMPLETE, and `wait` doesn't know to give up.

**Fix sequence:**

```bash
# 1. Kill the spinning wait command (Ctrl+C in your terminal)

# 2. Destroy the peering stack first — that removes the importer
make destroy-peering ENV=sandbox

# 3. Now destroy-deploy-host will actually succeed
make destroy-deploy-host CONFIRMED=yes

# 4. Redeploy fresh
make deploy-deploy-host

# 5. Re-run deploy-allX — phase 1 will re-deploy peering, phase 6/7 use existing secrets
make deploy-allX ENV=sandbox
```

**Worth a future improvement:** `destroy-deploy-host` should depend on `destroy-peering`, or at least pre-flight check for active importers and refuse with a clear message instead of letting CFN cancel a min into the operation. Want me to add that to the TODO too, or is this a "we'll see if it bites again" thing?
