`utf8mb3_tolower_ci` — that's an unusual and older MySQL collation. Not something Drupal itself would normally use, so it likely got in via legacy schema history on the prod side.

You only pasted the first query's result. Two possibilities:

1. **The second query returned rows** — some specific columns are using this collation → we alter just those
2. **The second query returned no rows** — nothing uses it explicitly, so it must be the **database-level default collation** for `zinew` → qmynd is choking on the DB-level default even though no column references it

Let's check both, plus a broader "where is utf8mb3 anywhere" sweep, so we know exactly what to fix.

sudo mysql zinew <<'SQL'

## Anticipated fix

Whatever we find, the cleanest resolution is to **upgrade the whole scratch DB from `utf8mb3` to `utf8mb4`**. Reasons:

- `utf8mb4` is a strict superset of `utf8mb3` (adds 4-byte characters like emoji) — no data loss
- Drupal 8+ officially requires `utf8mb4` anyway
- Both `utf8mb4_unicode_ci` (ID 224) and `utf8mb4_0900_ai_ci` (ID 255) are in qmynd's supported list
- Fixes the qmynd ECASE issue at the same time
- One conversion covers database, table, and column defaults in one shot

The conversion looks like this:

```bash
# Set the database default
sudo mysql -e "ALTER DATABASE zinew CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"

# Then generate one ALTER per table and pipe it into mysql
sudo mysql -N -e "
  SELECT CONCAT('ALTER TABLE zinew.\`', table_name,
                '\` CONVERT TO CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;')
    FROM information_schema.tables
   WHERE table_schema = 'zinew'
     AND table_type = 'BASE TABLE';
" | sudo mysql zinew
```

That auto-generates and runs one `ALTER TABLE ... CONVERT TO ...` for every table. `CONVERT TO` re-encodes column data if needed and updates all VARCHAR/TEXT/CHAR column collations in one shot.

Two things worth knowing before running this:

- **Time:** for a 2.4 GB DB with hundreds of tables it'll take a few minutes. Watch it — MySQL will hold locks on each table individually.
- **Safety:** as noted, this only touches the scratch DB on deploy-host. If anything goes sideways, `mysql zinew < /var/www/zinew.sql` restores the pristine dump.
