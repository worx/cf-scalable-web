feat(app-drupal): Phase C step 3 — cf-app-drupal.yaml lifecycle stack

Brings Drupal Secrets Manager entries and SSM parameters under proper
CloudFormation lifecycle. Until now, install-drupal.sh auto-created
secrets on first run — works for iteration but no audit trail, no
rotation policy, no clean tear-down. cf-app-drupal formalizes the
lifecycle for production-grade management.

What this stack OWNS (per environment):
  Secrets Manager (auto-generated passwords with proper exclusions)
    worxco/<env>/drupal/admin-password   24 chars by default
    worxco/<env>/drupal/db-password      32 chars, pgsql-safe charset

  SSM Parameters
    /<env>/drupal/db-name                  default: drupal
    /<env>/drupal/db-user                  default: drupal_user
    /<env>/drupal/site-name                default: drupal-<env>.test
    /<env>/drupal/install-path             default: /var/www/<env>/drupal
    /<env>/drupal/admin-username           default: admin
    /<env>/drupal/admin-email              default: admin@<site-name>

Stack deletion behavior:
  Secrets: scheduled deletion with default 30-day recovery window
           (CFN can't override; force-delete commands are printed in
            destroy output for emergency cleanup)
  SSM params: deleted immediately

Make targets:
  deploy-app-drupal ENV=...   Create/update the stack
  verify-app-drupal ENV=...   Show stack outputs + SSM params (table)
  destroy-app-drupal ENV=...  Delete (CONFIRMED=yes to skip prompt;
                              ENV=production requires 'WIPE PRODUCTION
                              SECRETS' as the strong affordance)

install-drupal.sh changes:
  Now resolves DB_NAME, DB_USER, SITE_NAME, ADMIN_USER, ADMIN_EMAIL
  from /<env>/drupal/* SSM parameters when present, falling back to
  hardcoded defaults when absent. This lets the script work cleanly:

    Recommended:    deploy-app-drupal → install-drupal
                    (stack creates secrets/params; install reads them)

    Backwards-compat: install-drupal alone
                      (auto-creates secrets, uses defaults for params)

    Mixed: any combination works — script reads what exists, generates/
           uses defaults for what doesn't.

Idempotency:
  - cf-app-drupal: standard CFN — re-deploy is a no-op if no params changed
  - install-drupal: existing read-or-generate logic handles either path

For an existing install (where install-drupal already auto-created the
secrets — like the current sandbox env), two recovery paths:
  Option A (clean): make remove-drupal ENV=<env>, then aws CLI delete-secret
                    --force-delete-without-recovery for both, then
                    deploy-app-drupal, then install-drupal
  Option B (preserve): aws cloudformation create-change-set with the
                       ImportResources flag — brings existing secrets
                       under stack management, preserves passwords.

## To validate step 3 on your sandbox

You have an existing install with auto-created secrets, so you have a choice:

### Option A — Validate end-to-end with a clean install (recommended)

```bash
# On deploy-host
make remove-drupal ENV=sandbox

# Force-delete the auto-created secrets (no 30-day window)
AWS_PROFILE=ZI-Sandbox aws secretsmanager delete-secret \
  --secret-id worxco/sandbox/drupal/admin-password \
  --force-delete-without-recovery --region us-east-1
AWS_PROFILE=ZI-Sandbox aws secretsmanager delete-secret \
  --secret-id worxco/sandbox/drupal/db-password \
  --force-delete-without-recovery --region us-east-1

# Now deploy the stack (creates fresh secrets via CFN)
# Run this from your Mac OR the deploy-host — it's just CFN, no marker needed
make deploy-app-drupal ENV=sandbox
make verify-app-drupal ENV=sandbox

# Reinstall — install-drupal will read the stack-managed secrets
make install-drupal ENV=sandbox
make verify-drupal ENV=sandbox
```
