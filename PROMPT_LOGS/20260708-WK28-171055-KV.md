### 17:11:36 - Response

Good observations across the board — let me unpack each.

## Why you're seeing 8-10 threads despite `workers=2`

pgloader's `workers` setting controls **per-table data-copy workers only**. SBCL and pgloader spin up a bunch of *other* threads on top of that:

| Thread role | Typical count |
|-------------|---------------|
| Main coordinator | 1 |
| MySQL source reader | 1 per active table |
| PostgreSQL destination writer | `workers` per active table (yours = 2) |
| DDL / schema tracking | 1 |
| Progress / logging | 1 |
| SBCL GC helper threads | 1-3, varies |
| MySQL wire-protocol driver | 1-2 |

Add those up: **~7-11 total** for a config like yours during data copy, plus a couple more during index rebuild phases. So 8-10 in htop is exactly what we'd expect. Your `workers=2` setting IS being honored — it's just describing a subset of the total threads.

## htop tips to see this more cleanly

- **`H`** — toggles showing/hiding user-space threads. Press it once to *hide* threads and see only the top-level `pgloader` process. That'll collapse those 8-10 lines into 1.
- **`t`** or **`F5`** — tree view. Shows parent-child hierarchy so you can see one `pgloader` with children indented under it.
- **`Shift+I`** — sort by CPU (usually the default anyway).

For this run I'd suggest **`t` for tree view + `H` to hide threads**. That way you see one `pgloader` line with a big total and you can watch the tree structure as it works.

## Why CPU is mostly zero

pgloader is **I/O bound**, not CPU bound. The dominant time cost is:

- Waiting for MySQL to return a batch of rows (network + disk read on your local MariaDB — but that's local so it's fast)
- Waiting for PostgreSQL RDS to acknowledge writes (**this is the biggest one** — every batch is a round trip across VPC networking, hitting RDS's WAL and index maintenance)
- Waiting for indexes to rebuild on PostgreSQL side after data copies complete

The CPU-bursts to 60-70% you saw are almost certainly:

- Data-conversion work in SBCL when a big batch of rows arrives (running your CAST rules — TINYINT→BOOLEAN, zero-date handling, DATETIME→TIMESTAMPTZ)
- Occasional GC pauses (SBCL freeing memory between batches)
- MariaDB spikes when it's answering a big `SELECT` for the next batch

Since it's network I/O bound, throwing more CPU at it wouldn't help. Same for RAM — you've got plenty of headroom.

## What this tells us about our config

You were right to worry the settings might be too conservative — this run is under-utilizing the box significantly. If pgloader finishes this run cleanly with no per-table errors, we know two things:

1. **The CAST rules cover the schema** — the "unknown-unknowns" are handled
2. **Memory headroom is enormous** — we could crank settings way back up

For a **future migration** (or a rerun of this one if we hit errors), we could safely go back to:

```
batch rows = 1000, prefetch rows = 1000,
workers = 4, concurrency = 2
```

That'd probably cut wall-clock in half or better. But **don't touch it mid-run** — we've committed to these settings for this pass, and the whole point is to get one clean end-to-end migration first.

## Metric to actually watch

Since CPU/RAM aren't the constraints, the metric that matters most is `pgloader.log` progress rate. Rough rule of thumb:

- Small tables (< 10K rows) → seconds each
- Medium tables (10K-500K rows) → 30 sec to a few min each
- Big tables (500K+ rows, especially `zi_zu_link_rev` at 1.23M) → 5-20 min each

If you `tail -f /tmp/pgloader/pgloader.log` and see per-table DONE lines coming through steadily, that's the healthiest signal. Total run for a 2.4 GB dump with this config is probably 60-90 min as originally estimated — but if network to RDS is being kind, could beat that.
