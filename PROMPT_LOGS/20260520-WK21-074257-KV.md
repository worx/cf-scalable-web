All 14 README links resolve to real files. Staged, ready to commit. Holding for your approval since you didn't explicitly say "commit" — and I want to layer my **feedback on documentation usability** before you decide.

**My ideas for making the doc system better for guests and contributors:**

1. **Standard Markdown links beat Obsidian wikilinks.** I used `[text](file.md)` throughout — these render in Obsidian, GitHub, IntelliJ, VS Code, and even pandoc → LaTeX → PDF. Obsidian-specific `[[wikilinks]]` would break everywhere else. Worth codifying as a doc-style rule.

2. **README as a navigation hub, not just a landing page.** What I added does this — the "Documentation map" section reorganizes by **what the reader wants to do** rather than alphabetically. Most projects' READMEs hide doc links in a wall of features text; this one greets users with "pick your goal."

3. **Frontmatter on memory files already gives us metadata for free.** They have `name`/`description`/`type` YAML frontmatter — Obsidian shows this as Properties in the right sidebar. We could standardize this across docs (not just memory): every doc gets a `name`/`description`/`audience` (operator/contributor/all). Then a future "make build-master" or "make build-doc-index" target could auto-generate a typed TOC from the frontmatter without us hand-maintaining the README list.

4. **Categorize by audience, not just topic.** I split the README by "I want to..." — that's audience-aware. Going further: tag every doc as `operator-facing`, `contributor-facing`, or `decision-record`. Obsidian's tag pane (currently disabled in your `.obsidian/core-plugins.json`) would then give readers a per-tag view. Cheap to enable.

5. **Graph view, sparingly.** Currently disabled in `core-plugins.json`. Enabling it lets readers see the doc-relationship graph visually — sometimes useful for "how do these concepts connect" exploration. Risk: empty graph is depressing; cluttered graph is overwhelming. Worth enabling once we have ~30+ docs cross-linked, not before.

6. **One file that's the "start here" reading sequence.** README serves that role now, but a dedicated `docs/READING-GUIDE.md` for new contributors ("if you've never seen this codebase before, read these in this order") would scale better than packing everything into README. Optional follow-up.

7. **The `master.pdf` concept already in TODO.** With standard Markdown links, pandoc can turn the whole docs/ tree into a single linked PDF easily. That'd be a great deliverable for someone who wants "the book" rather than browsing.

Anything in there resonate? Want to ship the current state, or layer some of these in first?
