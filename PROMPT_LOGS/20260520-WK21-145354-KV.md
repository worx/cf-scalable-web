```
fix(nginx): CSS/JS aggregate URLs need try_files fallback to PHP (lazy aggregation)

Browser opened Drupal site → page rendered HTML but unstyled (no CSS).
Diagnosis: every aggregated CSS/JS URL like
  /sites/default/files/css/css_eBw1FrQyQx7T6YR2Yfktw0BAzbAHs0Z5HK7v5jsjDzY.css
returned HTTP 404 from nginx.

Two layered causes, fixed in two commits:

(1) Previous commit (root:www-data 0770 on files/): Drupal couldn't
    write to /var/www/drupal/web/sites/default/files because perms
    locked www-data out. The dir was 0775 root:root; www-data fell
    through to the `other` triplet (r-x, no write).

(2) THIS commit: even with permissions fixed, the URLs still 404'd
    because Drupal 11 uses LAZY CSS aggregation — the aggregate
    file is built on-demand by PHP when its URL is first requested.
    Our nginx vhost had:

      location ~ ^/sites/.*/files/(css|js)/ {
        expires max;
        log_not_found off;
      }

    No try_files fallback. So nginx tried to serve the URL as a
    static file, didn't find it on disk, and 404'd — never giving
    PHP a chance to build the file. Drupal's compiled-on-demand
    css/js workflow was broken.

The styles/ (image styles) block already had `try_files $uri @rewrite`
as Drupal docs recommend. Just missing on the css|js one. Updated
both inline-template copies (install-drupal.sh writes this at install
time; publish-drupal-vhost.sh writes the same template via surgical
SSM dispatch — both kept in sync).

After this:
  - First request to a fresh aggregate URL → nginx tries file → 404
    on disk → falls through @rewrite → /index.php → Drupal builds
    aggregate, writes to FSx, returns CSS content
  - Subsequent requests → nginx serves from FSx directly (static),
    no PHP roundtrip needed
```

```
fix(install-drupal): chown root:www-data + chmod g+rwX on runtime-writable dirs

Add a final permissions step to install-drupal.sh that runs AFTER
drush site:install and the settings.php overwrite. Walks the three
directories Drupal writes to at runtime and sets them to root:www-data
with group-rwX:

  - $INSTALL_DIR/web/sites/default/files (CSS/JS aggregates,
    image styles, user uploads)
  - $PRIVATE_DIR (Drupal "private://" file system)
  - $CONFIG_DIR (config sync; Drupal writes a defensive .htaccess here)

Without this, the dirs end up root:root with default umask (0775 root
group, no write for www-data) because composer + drush run as the
script's effective user (root via SSM dispatch). Symptoms we hit:
  - CSS/JS aggregate URLs return 404 (Drupal can't write aggregates)
  - watchdog logs "Couldn't write .htaccess" security warnings for
    private:// and config-sync paths
  - Site renders content but no styling

salt.txt stays at the tighter 0640 (set immediately after this loop)
since it's a secret — operators only, not group-write.

Capital X in `u=rwX,g=rwX,o=` is deliberate: applies execute only to
directories, leaving files at 0660. Other (non-group) gets nothing —
locked out as a defense-in-depth measure even though those dirs only
exist on FSx with NFS access already restricted to the project SGs.

Combined with earlier commits in this arc (a405708 for the nginx
try_files fallback; 9d3425c for salt.txt perms) this makes a fresh
make deploy-allX produce a fully-styled Drupal site end-to-end with
no manual post-install cleanup needed.

Spec: docs/FSX-LAYOUT.md "File ownership and permissions" — keep
that doc in sync with this loop.
```
