## PostgreSQL error `28P01`

That's the SQLSTATE code for "invalid_password" — RDS accepted the connection, matched the username `drupal_user`, but the password didn't hash-match.

Two possibilities:

1. **The password in `/etc/worxco/envs/sandbox` doesn't match what's stored for `drupal_user` in PostgreSQL** (the values got out of sync somehow during rebuild)
2. **The password is correct, but pgloader's URL parser is mangling special characters** during URL-decoding despite the `jq @uri` encoding

## Diagnostic test — isolate which one

Let's bypass pgloader's URL handling entirely by using `psql` directly. It uses connection-string format (not URL), no encoding gymnastics:

```bash
sudo bash -c 'source /etc/worxco/envs/sandbox && \
  PGPASSWORD="$DRUPAL_DB_PASS" psql \
    "host=$DRUPAL_DB_HOST port=$DRUPAL_DB_PORT dbname=$DRUPAL_DB_NAME user=$DRUPAL_DB_USER sslmode=require" \
    -c "SELECT current_user, current_database();"'
```

**Two outcomes:**

| psql result | What it means | Next step |
|-------------|---------------|-----------|
| Prints `current_user \| current_database` row | Password is correct → pgloader's URL encoding is the culprit | We swap the URL format for a syntax pgloader can't mangle |
| `FATAL: password authentication failed for user "drupal_user"` | Password itself is wrong → env file and PostgreSQL are out of sync | We reset drupal_user's password (or re-run install-drupal to sync them) |

## If it's the URL-encoding culprit (probably)

pgloader supports specifying the connection outside the URL by putting each field on its own line. That eliminates all URL-decoding shenanigans. The syntax is:

```
LOAD DATABASE
  FROM mysql://worxco:localscratchpass@127.0.0.1:3306/zinew
  INTO postgresql:///drupal
    HOST '$DRUPAL_DB_HOST'
    PORT $DRUPAL_DB_PORT
    USER '$DRUPAL_DB_USER'
    PASSWORD '$DRUPAL_DB_PASS'
    SSL MODE 'require'
  ...
```

Or, even simpler — set `PGHOST`, `PGPORT`, `PGDATABASE`, `PGUSER`, `PGPASSWORD` as env vars and use a very minimal URL. pgloader honors those.
