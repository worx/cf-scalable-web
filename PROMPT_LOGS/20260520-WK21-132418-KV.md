**Message:**
```
feat(deploy-allX): heartbeat status loop during Phase 2 parallel wait

Previously, the wait for Phase 2's three parallel tracks (AMI build,
database, cache) used sequential `wait $PID; echo ✓ X finished`. This
notified on each completion BUT only in the static order
(cache→db→ami) and produced no output during the long quiet stretch
between database-done (~10m) and AMI-done (~25m). New users (and
sometimes the author) wondered if it was hung.

Replaced with a status loop that:
  - Checks each PID's liveness every 60s via `kill -0`
  - Prints a heartbeat line showing all three statuses with elapsed time:
      [12m30s] AMI=running  DB=done(rc=0)  Cache=done(rc=0)
  - Emits a one-shot ✓ message the moment any track finishes (in
    completion order, not a fixed order)
  - Captures each track's return code as it finishes for the final
    pass/fail check

Same correctness guarantees: doesn't proceed to Phase 3 until all
three return; surfaces all three return codes; fails loudly if any
nonzero. Just much friendlier UX during the wait.

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
```
