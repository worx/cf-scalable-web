### 20:10:39 - Response

**Classic macOS resolver negative-cache.** Here's what happened:

- macOS's resolver (`mDNSResponder`) cached an NXDOMAIN for `sandbox.envs.zoning-info.com` at some point earlier (likely during one of the dig probes BEFORE publish-dns wrote the record)
- That NXDOMAIN is cached until its negative-TTL expires (usually 5-15 min)
- `dig +short` queries DNS servers directly, bypassing the OS cache, so it correctly sees the new record
- `curl`/`ping`/Safari/etc. all go through the OS resolver, which serves them the cached NXDOMAIN

`dig` and `curl` disagreeing on whether a name exists is the unmistakable signature.

**Fix it right now:**

```bash
sudo dscacheutil -flushcache
sudo killall -HUP mDNSResponder
```

Then `curl https://sandbox.envs.zoning-info.com/` should work. And opening it in a browser. And `make smoke-test-public ENV=sandbox` should pass.

**The site has been up the whole time.** Step 4 proved that with the cert and ALB. The "failure" is your Mac's resolver lying to your local tools.

Once you confirm the flush worked, I'll add a diagnostic to smoke-test-public — if `dig +short` returns a record but `curl` returns 000, the script should detect the inconsistency and print exactly this fix command. That way the next time it bites (or bites someone else following the runbook), the script tells them what to do instead of just failing opaquely.
