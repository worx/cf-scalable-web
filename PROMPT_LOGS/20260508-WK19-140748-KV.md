### 14:08:52 - Response

Run these **from your Mac** (not the deploy-host shell — your SSM session dies when the instance is replaced):

```bash
# 1. Destroy peering first (it imports deploy-host SG, blocks deploy-host destroy otherwise)
make destroy-peering ENV=sandbox

# 2. Destroy the deploy-host instance + stack
make destroy-deploy-host

# 3. Recreate the deploy-host (new SG ID, new instance, fresh bootstrap.sh run)
make deploy-deploy-host

# 4. Recreate peering using the new deploy-host SG
make deploy-peering ENV=sandbox
```

Total wall time: ~10-15 minutes. Bootstrap takes ~3-5 of those.

## Before you start

Tear down the running pieces tied to the old deploy-host:
- Ctrl-C the drush rs in its terminal
- Ctrl-C the SSH tunnel terminal on your Mac

(Both will fail anyway when the instance dies. Tearing down cleanly is just tidier.)

## What survives, what doesn't

| Survives | Doesn't survive |
|----------|-----------------|
| FSx volume + Drupal install at `/var/www/sandbox/drupal` | Deploy-host EBS root disk |
| RDS database with all Drupal tables | Anything in `/home/ubuntu/` (gone with instance) |
| All Secrets Manager entries | Old `/etc/hosts` and `/etc/fstab` (regenerated by bootstrap) |
| All SSM parameters | drush rs process |
| FSx data, RDS data — all of it | SSH host key fingerprint (you'll see "Warning: ECDSA host key for ... has changed" — accept) |

## After step 4 finishes

SSH back in and validate the auto-everything path:

```bash
# Find the new instance ID
aws ec2 describe-instances \
  --filters "Name=tag:Name,Values=cf-deploy-host" "Name=instance-state-name,Values=running" \
  --query 'Reservations[].Instances[].InstanceId' --output text

# SSH-via-SSM (host key will have changed; accept the new one)
ssh deploy-host

# Verify the auto-everything path worked
df -h | grep /var/www                 # FSx mounted automatically?
grep "worxco-mount-env" /etc/fstab    # fstab entry written?
grep "worxco-fsx-hosts" /etc/hosts    # /etc/hosts entry from refresh-env-config?
ls /etc/worxco/envs/                  # cached env file present?
head /etc/worxco/envs/sandbox         # should have 'export ...' lines

# Existing Drupal install should still be there
ls /var/www/sandbox/drupal/.installed
make verify-drupal ENV=sandbox        # still healthy

# And browse it (drush rs in tmux this time so it persists)
make serve-drupal-local 2>/dev/null   # this is for local — not what we want
# Instead, use the cloud sequence:
source /etc/worxco/envs/sandbox
export DRUPAL_DB_PASS="$(aws secretsmanager get-secret-value \
  --secret-id worxco/sandbox/drupal/db-password \
  --query SecretString --output text)"
tmux new-session -d -s drupal-cloud "cd /var/www/sandbox/drupal && vendor/bin/drush runserver 0.0.0.0:8080"

# Mac (separate terminal): ssh -L 8080:localhost:8080 deploy-host
# Browser: http://localhost:8080 — should still load
```

If all that works, you've proven the whole pipeline regenerates from zero with no manual installs or fixes. Bedrock validation.
