No need to teardown for #1 — they're independent. The drush runserver process loaded settings.php at startup and doesn't re-read env vars; rewriting `/etc/worxco/envs/sandbox` doesn't affect what's already running. Same for the SSH tunnel — it's just a forwarder.

So for the **minimum** to validate #1, leave both running:

```bash
# In a NEW terminal on the deploy-host (don't touch the one running drush rs)
cd ~/projects/cf-scalable-web && git pull

sudo refresh-env-config sandbox
head -10 /etc/worxco/envs/sandbox     # should now show "export ..."

# Confirm a fresh sourcing exports correctly
source /etc/worxco/envs/sandbox
env | grep DRUPAL_DB_HOST             # exported value visible
```

If that all works, #1 is validated.

## When you'd want to teardown

Only if you want to **confirm a fresh drush rs picks up the new env-file format end-to-end**:

```bash
# In the terminal running drush rs: Ctrl-C
# Then:
source /etc/worxco/envs/sandbox        # plain source, no set -a wrapper
export DRUPAL_DB_PASS="$(aws secretsmanager get-secret-value \
  --secret-id worxco/sandbox/drupal/db-password \
  --query SecretString --output text)"
cd /var/www/sandbox/drupal
vendor/bin/drush runserver 0.0.0.0:8080

# Refresh the browser at http://localhost:8080 — should still work
```

That confirms **plain `source`** (no `set -a` wrapper) is now sufficient. If it works, the env-file fix is complete and `set -a` is no longer needed for sourcing.

The SSH tunnel from your Mac stays up the whole time — it doesn't care if drush rs restarts on the other side. Only thing tunnel-side that'd break is if a request was mid-flight when you Ctrl-C drush rs (you'd see "connection reset" in the browser; just refresh).

Either path is fine. The minimum path proves the fix mechanically works; the thorough path proves it works for the actual user-facing scenario. Up to you.
