## Where pgloader logs go

pgloader splits its output into **two channels** by default:

| Destination | What lands there |
|---|---|
| **stdout** (→ your `/tmp/zinew-pgloader.log` via `tee`) | Startup lines, final summary table, fatal errors |
| **`/tmp/pgloader/pgloader.log`** (pgloader's built-in log file) | Per-table detail — every `COPY`, every `CREATE INDEX`, every progress marker |

The three lines you're seeing on stdout are all you get **until the run completes**. Then a summary table lands there with per-table row counts and elapsed times.

The interesting live view is exactly where you're already looking: `/tmp/pgloader/pgloader.log`. That's the "watching the sausage get made" file.

## Why the split exists

pgloader's default log level for stdout is `NOTICE`, which suppresses everything except summary-level info. The file log at `/tmp/pgloader/pgloader.log` is `INFO` level, which streams the per-table activity. If you wanted the file log's detail to also show on stdout, you'd pass `--verbose` to pgloader — but that would make your terminal noisy for a run this big. What you have now is the sensible split.

## What to watch for as the run continues

In `tail -f /tmp/pgloader/pgloader.log` — expect to see this rhythm for each table:

```
COPY zinew.<table_name>
DONE copying zinew.<table_name>
```

Then after all tables are copied, an index-creation phase:

```
CREATE INDEX zinew.<table_name>_<idx_name>
CREATE UNIQUE INDEX zinew.<table_name>_<pkey>
```

Then foreign keys:

```
ALTER TABLE zinew.<table> ADD FOREIGN KEY ...
```

Then sequence resets:

```
SELECT setval(...) for zinew.<table>_id_seq
```

## When it finishes

When the whole thing completes, **the summary table lands in `/tmp/zinew-pgloader.log`** (the tee'd stdout file). It looks like:

```
       table name     errors       rows      bytes      total time
-----------------  ---------  ---------  ---------  --------------
  fetch meta data          0          <n>                    <t>ms
             DDL             0                                  <t>s
       zinew.node          0        <N>      <MB>            <t>s
       zinew.users         0         42       15kB           <t>ms
       ...
-----------------  ---------  ---------  ---------  --------------
              total             ...       ...        ...       <total time>
```

**Success looks like** every table row showing `errors = 0` and `rows > 0` (or `rows = 0` for genuinely-empty source tables). **Any row with `errors > 0`** = a table that had at least one row pgloader couldn't convert — those we'd address with additional CAST rules and rerun (usually a specific column type or a specific bad-data row).

Sit tight and let it cook. For a 2.4 GB dump this'll probably run 15-45 minutes depending on how many indexes are being rebuilt. When it's done, `cat /tmp/zinew-pgloader.log` will have the whole summary table.
