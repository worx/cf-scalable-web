### 11:16:54 - Response

You're right — I was reading from the **PHP/nginx box's POV**, not the deploy-host's. Critical detail I missed: **the FSx mount point differs between hosts**:

| Host | FSx mount point | Drupal path |
|---|---|---|
| deploy-host | `/var/www/sandbox/` | `/var/www/sandbox/drupal/` |
| PHP / nginx boxes | `/var/www/` | `/var/www/drupal/` |

Same files on FSx, different paths to reach them. **That's actually the root cause of the AssertionError**: drush ran on the deploy-host when Drupal was installed, compiled its service container with `/var/www/sandbox/drupal` baked in, then later PHP-FPM (which sees the files at `/var/www/drupal`) tries to use the same compiled container and the paths don't resolve.

**This changes the fix.** Don't run `drush cache:rebuild` from the deploy-host — it would just re-cache with `/var/www/sandbox/drupal` again, and PHP would hit the same error on the next request. Instead, **delete the compiled cache directly from FSx and let PHP-FPM rebuild it from its own perspective on the next request:**

```bash
# From the deploy-host (whichever path you're comfortable with — same file)
sudo rm -rf /var/www/sandbox/drupal/web/sites/default/files/php/*

# Verify the dir is empty
ls -la /var/www/sandbox/drupal/web/sites/default/files/php/
```

Then re-curl:
```bash
curl -sv -H "Host: drupal-sandbox.test" "http://sandbox-alb-95496877.us-east-1.elb.amazonaws.com/" --max-time 15 | head -40
```

The first request will be slow (Drupal recompiles the container against PHP-FPM's view of paths: `/var/www/drupal/...`). Subsequent requests will be fast and consistent.

---

**Deeper architectural issue worth talking about once this is working:** the mount-point asymmetry between deploy-host and PHP/nginx boxes is *by itself* a footgun. Anything that runs from both hosts (drush, install scripts, cache rebuilds, file paths in settings.php) has to know which side it's on. Two fixes worth considering for later:

1. **Mount FSx at `/var/www/$ENV/` on the PHP and nginx boxes too** — matches the deploy-host, eliminates the asymmetry. Requires updating `configure-php.sh` and `configure-nginx.sh` to mount with the env-prefix, plus updating nginx vhost roots accordingly. Then `cf-app-drupal`'s `DrupalInstallPathParameter` (`/var/www/${EnvironmentName}/drupal`) matches reality everywhere.

2. **Or mount FSx at `/var/www/` on the deploy-host** — matches the runtime boxes. Simpler for the runtime path, but loses the env-prefix that's nice for multi-tenancy.

Either way, you want **one canonical path** across the fleet. But for *right now* — just clear the cache and we'll see Drupal serve a real page.
