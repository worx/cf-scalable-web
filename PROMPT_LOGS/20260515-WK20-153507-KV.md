### 15:35:50 - Response

That's because the SSM-SessionManagerRunShell document explicitly does `exec bash -l`, which overrides ubuntu's login shell (the `chsh` only changes `/etc/passwd`, but SSM bypasses that). This is exactly the Option A scenario — keep bash as SSM's entry shell, but have bash auto-exec zsh.

**One-liner to add the auto-exec to your `~/.bashrc`:**

```bash
echo '[[ $- == *i* ]] && [ -x /usr/bin/zsh ] && exec zsh -l' >> ~/.bashrc
```

What it does:
- `[[ $- == *i* ]]` — only fires in interactive shells (won't trigger on scripts or cron)
- `[ -x /usr/bin/zsh ]` — degrades gracefully if zsh isn't installed for some reason
- `exec zsh -l` — replace bash with a zsh login shell (no extra process)

After running that, disconnect your SSM session and reconnect. The flow becomes:
1. SSM doc: `exec bash -l`
2. bash sources `~/.bashrc` → hits the auto-exec line → `exec zsh -l`
3. zsh starts, sources `~/.zshrc`, RPROMPT lights up

You should land directly in zsh with `[env:sandbox]` on the right.

**If you ever need to escape zsh** (e.g., debugging the shell itself): you can bypass with `NO_AUTO_ZSH=1 bash` — but I haven't added that guard, just mentioning the pattern. Right now there's no escape hatch; if you need one I'll add `[ "$NO_AUTO_ZSH" != 1 ]` to the condition.

**For the persistent commit** (when we do it): this line goes into `/etc/bash.bashrc` (system-wide) via `bootstrap.sh`, so even a freshly-rebuilt deploy-host comes up with the auto-exec already in place.
