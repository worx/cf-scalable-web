**Message:**
```
feat(https): HTTPS via ACM cert + ALB termination — sandbox.envs.zoning-info.com

End-to-end HTTPS for env-served URLs. ACM provisions and renews the cert
automatically; ALB terminates TLS; Drupal trusts the proxy so it
generates https:// URLs. No CertBot, no renewal cron, no key on disk.

Architecture:

  Browser  ──https──►  ALB (TLS terminator, ACM cert)
                            │
                            └─http──►  nginx (port 80, no TLS)
                                          │
                                          └─FastCGI──► PHP-FPM
                                                          │
                                                          └─ Drupal reads
                                                             X-Forwarded-Proto,
                                                             generates
                                                             https:// URLs

cf-compute-alb.yaml changes:
- Replace `ACMCertificateArn` parameter (pre-existing manual cert ARN
  approach) with `DomainName` + `EnvsSubZoneId`. The stack now CREATES
  the cert as part of `AWS::CertificateManager::Certificate` with DNS
  validation against the env's sub-zone (which lives in the same AWS
  account thanks to the delegation done per docs/DNS-SETUP.md).
- HTTPS listener (port 443) references the new in-stack cert via !Ref
  (no external ARN to track). TLS 1.3 baseline
  (ELBSecurityPolicy-TLS13-1-2-2021-06).
- HTTP listener (port 80) returns 301 → HTTPS when DomainName is set;
  falls back to forwarding to nginx when DomainName is empty (useful
  for initial env bring-up before DNS delegation exists).
- New output `AlbCertificateArn` (conditional) for debugging / external
  reference.

Per-env parameter files updated:
- sandbox: DomainName=sandbox.envs.zoning-info.com,
           EnvsSubZoneId=Z03149452QE09GQX23F48 (HTTPS ON)
- staging: DomainName="" EnvsSubZoneId="" (HTTP-only fallback)
- production: same as staging until prod is initialized

install-drupal.sh: settings.php template gains the reverse-proxy block:
   $settings['reverse_proxy'] = TRUE;
   $settings['reverse_proxy_addresses'] = ['10.0.0.0/8'];
which makes Drupal trust X-Forwarded-Proto from the ALB and generate
https:// URLs. Trust scope is the RFC1918 10/8 range — appropriate
since ALB-to-nginx and nginx-to-PHP-FPM are always intra-VPC.

Makefile smoke-test-public now curls https:// (and uses `curl -sL` so
it'd follow the 301 redirect from http:// if someone did test that
path).

Docs:
- docs/SSL-MANAGEMENT.md (new): full cert lifecycle (issuance,
  validation, renewal, rotation, destroy), Drupal-side config,
  troubleshooting, cost, "what ACM cannot do."
- docs/DNS-SETUP.md addendum: Step 4 in the one-time setup tells the
  operator to record the sub-zone ID into compute-alb-<env>.json for
  cert validation.

Notably NOT bumping RecipeVersion — all changes are CFN + script-side;
PHP-FPM/nginx AMIs don't need rebuilds. Drupal-side reverse_proxy reads
X-Forwarded-Proto, which the ALB sets automatically.

Migration: deploy-allX picks up cert creation on next compute-alb
deploy. Validation typically completes in 3-5 min; CFN waits for the
cert to reach ISSUED before creating the HTTPS listener.
```
