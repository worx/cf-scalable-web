### 16:58:29 - Response

Great catch — those numbers next to the table names in the "processing tables in this order" list are **not row counts**. Easy to misread because the format looks like it *should* be.

## What that number actually is

pgloader schedules table work based on **DDL complexity**, not data volume. The number next to each table in that list is the **count of DDL objects that pgloader will need to create for that table** — primarily indexes, but also unique constraints, primary keys, and foreign keys. Tables get processed in **descending order of that count** so that the "most work" tables start first and their long-tail index-rebuild time overlaps with data copy on smaller tables.

So `zinew.batch 19` means "batch has 19 indexes/constraints to build after data copy" — not "batch has 19 rows." The batch table happens to have a lot of indexes because Drupal's Batch API stores queued work items with many secondary lookups.

## Why data copy phase looks different

The `COPY zinew.zi_zu_link_rev with 1,233,329 rows` line is from a completely different phase of pgloader's log stream. That number **is** the actual row count — reported at COPY time as pgloader queries the source MySQL for the row count of that specific table right before streaming.

## The three phases pgloader logs

Here's the shape of a pgloader run, so the log makes more sense as it scrolls by:

| Phase | What logs look like | What the numbers mean |
|-------|---------------------|-----------------------|
| **1. DDL prep** | `NOTICE prepare pgsql database` <br>`NOTICE processing N tables in this order:` <br>`NOTICE zinew.batch 19` | Table name + index/DDL count. Sorted by DDL count desc. |
| **2. Data COPY** | `LOG COPY zinew.<table> with <N> rows` <br>`LOG DONE copying zinew.<table>` | Actual row count from source `SELECT COUNT(*)` |
| **3. Post-load DDL** | `LOG CREATE INDEX zinew.<table>_<name>` <br>`LOG ADD FOREIGN KEY zinew.<table>_<fk>` | Reconstructing indexes and constraints on the loaded data |

## What order data copy happens in

The COPY phase can shuffle around from the "processing tables in this order" list because with `workers > 1`, pgloader can be copying several tables in parallel and their DONE lines interleave. Your current config has `workers = 2, concurrency = 1`, so:

- **`concurrency = 1`** — only one table's data is being copied at a time
- **`workers = 2`** — but each copy uses 2 threads (one reading MySQL, one writing PostgreSQL)

So in your run, tables will COPY one at a time, roughly in the order shown in the "processing tables" list (adjusted somewhat for size — big tables like `zi_zu_link_rev` may get pulled forward because pgloader wants to start the long-running ones ASAP so they finish by the time smaller stragglers wrap up).

The 1.2M-row `zi_zu_link_rev` starting first is exactly right — that's your biggest table getting kicked off early. If pgloader waited to do it last, it'd be a bottleneck at the end. Starting it first is the smart schedule.

## Small honest caveat

I'm confident about the phases and about the number *not* being rows. On the specific meaning (is it index count vs total DDL count vs "score" vs something else) — pgloader's source is Common Lisp and its exact scheduling metric changes across versions. If it turns out to be columns or bytes instead of indexes for your version, the general principle still applies: it's a scheduling priority, not a row count.
