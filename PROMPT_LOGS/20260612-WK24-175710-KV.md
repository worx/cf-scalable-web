Operator asked a clarifying question on the INTO clause of the pgloader

dbadmin) but the INTO references DRUPAL_DB_USER (drupal_user) — why

Answer: dbadmin is the RDS-provisioned master user; drupal_user is
the application user that install-drupal.sh creates LATER. If you've
never run install-drupal-remote against a sandbox env, drupal_user
doesn't exist in PostgreSQL and pgloader fails with "password
authentication failed for user drupal_user."

Added explicit prerequisite block at the start of step 6:
  1. Note dbadmin vs drupal_user distinction
  2. Recommend `make install-drupal-remote ENV=sandbox` as the
     fastest way to bootstrap the DB user + settings.php + salt.txt
  3. Acknowledge the fresh Drupal 11 install is wasted work — its
     schema/data get dropped by pgloader's `include drop` clause —
     but the DB user and the settings.php files it creates are what
     we need to keep
  4. Note that if we end up doing this repeatedly, splitting out a
     "create DB user only" target would be the right cleanup

The /etc/worxco/envs/sandbox env file already has DRUPAL_DB_HOST,
DRUPAL_DB_PORT, DRUPAL_DB_NAME, DRUPAL_DB_USER, and DRUPAL_DB_PASS
populated correctly (from SSM + Secrets Manager) — operator doesn't
have to make those up. Only the PG-side user creation was missing.

## Quick summary for your shell session

**Step 0 (the prerequisite I just documented):** if you haven't already done it for this sandbox cycle, from your Mac:

```bash
make install-drupal-remote ENV=sandbox
# ~5 min. Side effect: drupal_user gets CREATEd in PG with the right
# password and grants. That's the part we want to keep.
```

You can verify drupal_user exists afterward — from the deploy-host:
```bash
source /etc/worxco/envs/sandbox
PGPASSWORD="$DRUPAL_DB_PASS" psql -h "$DRUPAL_DB_HOST" -p "$DRUPAL_DB_PORT" \
  -U "$DRUPAL_DB_USER" -d "$DRUPAL_DB_NAME" -c "SELECT current_user, current_database();"
# Should print: drupal_user | drupal
```

If that works, pgloader's INTO URL will work.

**Then continue with step 6** as documented. pgloader's `include drop` will discard the fresh Drupal 11 schema and replace it with prod's. Don't worry about clobbering anything important — that's the point.

If the install-drupal-remote step itself takes a weird path or fails (since prior sessions may have left this sandbox in various states), drop the output here and we'll work through it.
