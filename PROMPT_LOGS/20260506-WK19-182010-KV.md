fix(deploy-host): Conditional pre-delete of SSM document; document gotchas

The unconditional pre-delete of SSM-SessionManagerRunShell in
make deploy-deploy-host was silently breaking SSM session preferences.

Bug: The Makefile always ran `aws ssm delete-document` before
CloudFormation deploy. On an UPDATE, CFN's SSMSessionPreferences
resource was already CREATE_COMPLETE — CFN saw no template change
and did nothing. Result: stack happy, document gone, sessions fall
back to default (ssm-user with sh) instead of ubuntu with bash.

Fix: pre-delete only when the stack does NOT exist. The pre-delete
is meant to clean up orphans from a prior lifecycle (where the doc
might persist after a stack delete due to ordering edge cases), not
to re-bootstrap on every deploy. When the stack already exists, CFN
owns the document and the pre-delete is harmful.

Recovery for already-orphaned state: aws ssm create-document with
the same content. CFN's physical-id is just the document name, so
re-creating it puts CFN's state back in sync.

Also documents two new gotchas for future reference:
  - !Ref on SG without VpcId returns name, not ID (caused yesterday's
    cf-deploy-peering failure)
  - Unconditional SSM doc pre-delete (this fix)

Co-Authored-By: Claude Opus 4.6 (1M context) <noreply@anthropic.com>
