## Things worth flagging in prod's settings.php

Nothing that blocks the migration, but three things to be aware of so you're not surprised:

**1. Config sync directory location**
- Prod: `'sites/default/files/sync'` (relative to Drupal root)
- Sandbox: `'/var/www/drupal-config'` (absolute)
- **Impact:** Drupal bootstrap doesn't care (config comes from the DB). Only `drush cex` / `drush cim` would care. Not a blocker for first-boot smoke test.

**2. Flysystem/S3 for user files**
- Prod's settings.php configures a `flysystem` bucket `zi-documents` for file uploads
- Our sandbox settings.php has no flysystem block
- **Impact:** anything stored on the `zi-documents` S3 bucket will 404 in sandbox until we either (a) add flysystem config pointing at a sandbox S3 bucket or (b) point sandbox flysystem at prod's `zi-documents` read-only, or (c) sync `zi-documents` → sandbox bucket. For a first browser test, this is fine — you'll see missing images for S3-hosted content, but the site structure will work. We can address in a follow-up.

**3. Trusted host patterns**
- Prod: `zoning-info.com` variants
- Sandbox: `elb.amazonaws.com` (which is what your ALB DNS resolves to)
- **Impact:** you'll access the sandbox site via its ALB DNS (something like `sandbox-drupal-alb-XXX.us-east-1.elb.amazonaws.com`), which is already covered. No change needed.

**4. Hardcoded prod DB creds in prod settings.php** ← **CRITICAL**

Prod's settings.php has the Aurora MySQL credentials hardcoded. **If prod's settings.php ever landed on sandbox, sandbox would connect to prod's Aurora and write to prod.** That's why our double-exclude of `settings.php` (in tar AND in rsync) is important.
