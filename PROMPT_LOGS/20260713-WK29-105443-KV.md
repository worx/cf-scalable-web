### 10:55:38 - Response

Great instinct. Yes, this is a clean-up worth doing before we build the `make dump-*` targets that upload scripts. Object prefixes are the S3-native way to do exactly what you're describing — no real folders, but every tool (aws CLI, Console UI, lifecycle rules) treats them as folders when convenient.

## Current bucket layout vs proposed

**Current** (as we've built it so far):
```
s3://sandbox-migration-kv-worxco/
├── logs/                     ← already prefixed by _common.sh's log_upload
│   └── YYYY-MM-DD/
│       └── *.log
├── zinew.sql                 ← dumps at root
├── drupal-codebase.tar.gz
├── drupal-files.tar.gz
└── drupal-private.tar.gz
```

**When we start pushing scripts**, if we don't prefix:
```
s3://sandbox-migration-kv-worxco/
├── logs/
├── _common.sh                ← scripts mingled with dumps
├── dump-mysql.sh
├── dump-codebase.sh
├── dump-files.sh
├── dump-private.sh
├── restore-mysql.sh
├── ... (more scripts as we build)
├── zinew.sql
├── drupal-codebase.tar.gz
└── ... etc
```

That gets messy fast. `aws s3 ls s3://bucket/` becomes a long unsorted list of mixed-purpose objects.

## Two clean alternatives

**Option A — only prefix scripts, leave dumps at root:**
```
s3://bucket/
├── scripts/
│   ├── _common.sh
│   ├── jumpbox/dump-mysql.sh
│   └── (etc.)
├── logs/
├── zinew.sql
└── drupal-*.tar.gz  (4 dump files)
```

**Option B — prefix everything: scripts, dumps, logs each get their own top-level:**
```
s3://bucket/
├── scripts/
│   ├── _common.sh
│   ├── jumpbox/*.sh
│   └── deploy-host/*.sh
├── dumps/
│   ├── zinew.sql
│   └── drupal-*.tar.gz
└── logs/
    └── YYYY-MM-DD/*.log
```

## My honest recommendation: Option B

Reasons:
1. **Total symmetry** — bucket root has three folders, zero loose files. `aws s3 ls s3://bucket/` returns three tidy `PRE` entries and you know exactly where everything lives.
2. **Mirrors the local repo layout** — `migration/scripts/` → `s3://bucket/scripts/`. Mental model transfers with zero context switch.
3. **Better lifecycle rules later** — if you ever want "expire dumps at 30 days, keep scripts indefinitely, keep logs 90 days," each prefix gets its own rule. Currently the CFN template has one blanket 30-day rule; with prefixes we could differentiate cleanly.
4. **Easier bulk ops** — `aws s3 rm --recursive s3://bucket/dumps/` cleans dumps without touching scripts. Right now cleaning dumps means enumerating them by name.
5. **Cost of change is trivial** — 4 DUMP_S3_KEY defaults to update in the scripts we already wrote, plus we do it BEFORE building any Makefile targets that reference these keys.

Option A is fine too — mostly a cosmetic preference. But since we're paying the cost of updating anyway, going all the way to Option B costs nothing extra and reads cleaner.

## Cost breakdown if you go with Option B

Concrete changes:

1. **4 script edits** — change `DUMP_S3_KEY` defaults:
   - `dump-mysql.sh`: `zinew.sql` → `dumps/zinew.sql`
   - `dump-codebase.sh`: `drupal-codebase.tar.gz` → `dumps/drupal-codebase.tar.gz`
   - `dump-files.sh`: `drupal-files.tar.gz` → `dumps/drupal-files.tar.gz`
   - `dump-private.sh`: `drupal-private.tar.gz` → `dumps/drupal-private.tar.gz`

2. **1 script edit** — update `restore-mysql.sh` default:
   - `zinew.sql` → `dumps/zinew.sql`

3. **Future scripts / Makefile targets** — will naturally use `scripts/` and `dumps/` prefixes when we build them.

4. **_common.sh** — already uses `logs/` prefix, no change needed.

5. **CFN bucket template** — no strict change required, but we could optionally tighten the lifecycle rule to only target `dumps/` (so scripts aren't accidentally expired at 30 days). Optional; can defer.

Total time: **~5 minutes** to make the DUMP_S3_KEY updates in the 5 script files.

**Want me to make the changes for Option B?** Or if you prefer A, that's a simpler 1-minute change (just leave DUMP_S3_KEY defaults alone; only prefix scripts when the Makefile targets upload them). Your call.
