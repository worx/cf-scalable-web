**Message:**
```
feat(make): wait-amis + update-ami-params; build-amis now blocks until AVAILABLE

P1 cleanups from the 2026-05-15 deploy-race trap. Two related fixes:

1. `make build-amis` previously kicked off Image Builder pipelines and
   returned immediately. Chained `update-ami-param` then found the
   PREVIOUS AVAILABLE AMI (because the new one wasn't built yet) and
   silently wrote that to SSM, so `deploy-compute` rolled to the same
   AMI we already had. Hit this twice during the debug arc.

   Fix: split into `build-amis-async` (fire-and-forget, same as old
   behavior) and `build-amis` which is now `build-amis-async wait-amis`.
   Default is safe; explicit opt-in for fire-and-forget.

2. New `wait-amis` target polls Image Builder every 30s and shows live
   status per pipeline (nginx=BUILDING, php-fpm-74=TESTING, ...).
   Exits 0 when all three reach AVAILABLE, exits 1 if any hits FAILED
   / CANCELLED / DEPRECATED / DELETED or the 40 min hard cap trips.

3. New `update-ami-params` (plural) wraps `update-ami-param` x3 so the
   API isn't a footgun. The original singular-with-PIPELINE-var design
   was easy to forget — caller had to remember to invoke three times.
   The new plural one is a one-liner.

4. `deploy-ami-params` alias for muscle memory — matches the surrounding
   `deploy-image-builder` / `deploy-compute` naming convention. The name
   I instinctively typed during the debug arc, only to discover it
   didn't exist. Now it does.

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
```
