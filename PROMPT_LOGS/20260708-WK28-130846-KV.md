### 13:10:19 - Response

Same error — which tells me the ALTER USER approach isn't the whole story. Here's what's actually going on:

## The real culprit — MySQL server's default authentication plugin

MySQL's authentication handshake works like this:

1. Client (pgloader) opens TCP to MySQL
2. **MySQL's very first packet announces the *server's* default auth plugin** (not the user's)
3. Client responds using that plugin

MySQL 8's server default is `caching_sha2_password`. pgloader's `qmynd` driver bails on that plugin **before ever getting to the user record**, so changing `worxco@localhost`'s plugin never gets the chance to matter — the connection dies at the handshake.

The `ALTER USER` you ran was correct *and* insufficient. We also need to change the **server-wide default**.

## Fix — server config + restart

```bash
# Create a small drop-in config that overrides the default plugin
sudo tee /etc/mysql/mysql.conf.d/pgloader-compat.cnf > /dev/null <<'EOF'
[mysqld]
default_authentication_plugin=mysql_native_password
EOF

# Restart MySQL to pick it up
sudo systemctl restart mysql

# Verify the change took
sudo mysql -e "SHOW VARIABLES LIKE 'default_authentication_plugin';"
# Expected: mysql_native_password
```

## Also — belt-and-suspenders: create worxco@127.0.0.1

MySQL matches users by `(user, host)`. `worxco@localhost` in MySQL 8 specifically matches **socket** connections, not TCP-to-127.0.0.1. Even with the server default fixed, we want an explicit user record for the TCP path:

```bash
sudo mysql <<'SQL'
CREATE USER IF NOT EXISTS 'worxco'@'127.0.0.1'
  IDENTIFIED WITH mysql_native_password
  BY 'localscratchpass';
GRANT ALL PRIVILEGES ON zinew.* TO 'worxco'@'127.0.0.1';
FLUSH PRIVILEGES;
SQL
```
