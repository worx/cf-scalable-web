fix(drupal-vhost): write drupal.conf to /etc/nginx/shared, not /var/www/nginx

Regression introduced by the FSx sibling-isolation refactor (commit
6f198c3). Both install-drupal.sh and publish-drupal-vhost.sh hard-coded
NGINX_VHOST_DIR=/var/www/nginx/sites-enabled. Under the OLD FSx layout
(/var/www = FSx root), that path resolved to /fsx/nginx/sites-enabled
which nginx instances see at /etc/nginx/shared/sites-enabled. Worked.

Under the NEW FSx layout (/var/www = /fsx/www sibling subtree),
/var/www/nginx/sites-enabled resolves to /fsx/www/nginx/sites-enabled —
INSIDE the www subtree, not in the nginx subtree. nginx instances mount
/fsx/nginx, so they never see drupal.conf in the new location. Result:
freshly-installed Drupal's vhost gets silently written to a dead path,
nginx serves the baseline default_server's 404 ("no vhost configured
for this host") to every request.

Past incident: 2026-05-23, fresh greenfield deploy after the FSx
refactor — install-drupal completed cleanly, smoke-test-drupal returned
404, drupal.conf was at /fsx/www/nginx/sites-enabled/ (invisible to
nginx), /fsx/nginx/sites-enabled/ was empty.

Fix:
1. Both scripts now write to /etc/nginx/shared/sites-enabled/ directly.
   Since the FSx refactor, the deploy-host also mounts /fsx/nginx at
   /etc/nginx/shared — so the writer and the readers see the same path.
   Removed the now-stale `sudo chmod 755 /var/www/nginx` line.

2. install-drupal-full now has 6 steps (was 4):
     1. install-drupal-remote
     2. publish-drupal-vhost      ← NEW (defense: always refresh vhost)
     3. reload-nginx              ← NEW (paired with step 2)
     4. smoke-test-drupal
     5. publish-dns
     6. smoke-test-public
   install-drupal-remote skips the Drupal install when the .installed
   marker is present, which means it ALSO skips re-writing drupal.conf

   the vhost is always current with whatever the current scripts
   produce — defense against any future "marker exists but vhost is
   stale/missing/wrong" failure mode (today's bug being one example).
