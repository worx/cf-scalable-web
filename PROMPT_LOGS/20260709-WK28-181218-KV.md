### 18:14:16 - Response

Aha — the pgsql driver is there in core, so that's not the issue. But look at the error message combined with what we know: **drush can read the codebase but can't read `settings.php`**. That's a permissions issue.

Recall my Step 4 chmod pattern set:

```bash
sudo chmod 640 /var/www/drupal/web/sites/default/settings.php
sudo chown root:www-data /var/www/drupal/web/sites/default/settings.php
```

That means `settings.php` is:
- Readable by `root`
- Readable by anyone in the `www-data` group
- **Unreadable by everyone else**, including your `ubuntu` user

When you run `./vendor/bin/drush` as `ubuntu`, PHP's `include()` of settings.php **fails silently** (Drupal catches parse/include errors gracefully during bootstrap). Then Drupal moves on with an empty `$databases` array, and hits the error you saw: *"The specified database connection is not defined: default."*

## Confirm in 3 quick checks

```bash
# 1. Confirm the perms
ls -la /var/www/drupal/web/sites/default/settings.php

# 2. Confirm ubuntu is NOT in www-data group
groups ubuntu

# 3. Prove ubuntu can't read it
cat /var/www/drupal/web/sites/default/settings.php > /dev/null
# Expected: "Permission denied"
```

## The fix — three options

Pick whichever fits your setup:

**A) Run drush with sudo (one-off, easiest)**

```bash
sudo -E ./vendor/bin/drush status
```

The `-E` preserves your sourced env vars across the sudo. Should just work.

**B) Add ubuntu to www-data group (persistent, cleanest for admin work on deploy-host)**

```bash
sudo usermod -aG www-data ubuntu
# Log out and back in for group change to take effect,
# or start a new sub-shell with:
newgrp www-data

# Then verify
groups
# should include 'www-data'

# And retry
./vendor/bin/drush status
```

This is what I'd recommend if you plan to run admin commands (drush, composer, etc.) from deploy-host regularly — you avoid needing sudo for every drush invocation.

**C) Make settings.php world-readable (NOT recommended)**

Would work but weakens security. The file has your DB password when read as plaintext (well, it's env-var indirected, but still). Don't do this.
