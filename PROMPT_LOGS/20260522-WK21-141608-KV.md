fix(ssm-dispatch): target ASG view, not tag-filtered EC2 describe-instances

`reload-nginx.sh` and `restart-php-fpm.sh` enumerated their SSM targets
by `aws ec2 describe-instances --filters tag:aws:autoscaling:groupName=...`
with state=running. That approach is fragile during instance refresh:
EC2's tag index briefly returns terminated instance IDs alongside live
ones, and SSM rejects the whole batch with `InvalidInstanceId — Instances
not in a valid state for account` rather than per-instance.

Hit today during deploy-allX: nginx ASG churned during the install
window (instances 502'd before Drupal install finished, ASG replaced
them on ELB health-check failure). When reload-nginx.sh ran post-install,
its tag-filter query returned 3 IDs — the current healthy one plus the
two already-terminated originals. SSM rejected the batch, and the
deploy-allX failed at the very last step despite a successful install.

Fix: enumerate targets via `aws autoscaling describe-auto-scaling-groups
--query Instances[?LifecycleState=='InService' && HealthStatus=='Healthy']`.
The ASG's own Instances[] list is authoritative for "what's actually
serving traffic" and reflects refreshes immediately, so we won't address
zombie instances. Also makes the warning more actionable when there are
zero targets ("ASG may be mid-refresh or scaled to 0").

Files Modified:
- scripts/reload-nginx.sh
- scripts/restart-php-fpm.sh

Tests: live query against sandbox-nginx-asg returns exactly the 2 current
       InService+Healthy instances (verified against ASG console view)
