### 11:27:22 - Response

**The smoking gun on path resolution:** At runtime under PHP-FPM, **everything** correctly resolves to `/var/www/drupal/web`. No `/var/www/sandbox/` anywhere. So the wrong path is NOT being computed from `__DIR__` or any filesystem realpath — it's being read from a **stale cached container in the DATABASE** (Drupal stores its compiled container in `cache_container` and `cache_bootstrap` tables, not just the filesystem). When we deleted the filesystem cache, Drupal happily picked up the DB-cached container which has `/var/www/sandbox/drupal` baked in from when drush ran on the deploy-host.

Clear the DB cache tables:

```bash
# From the deploy-host
psql-env sandbox -d drupal -c "TRUNCATE TABLE cache_bootstrap, cache_container, cache_discovery, cache_config, cache_default, cache_data, cache_menu, cache_dynamic_page_cache, cache_render, cache_page;"
```

That clears every cache bin that could be holding the wrong path. Drupal will rebuild from scratch on the next request, this time using PHP-FPM's view of paths (`/var/www/drupal/`).

You can also do just the critical ones if you want to be conservative:
```bash
psql-env sandbox -d drupal -c "TRUNCATE TABLE cache_bootstrap, cache_container, cache_discovery;"
```

Then re-curl:
```bash
curl -s -H "Host: drupal-sandbox.test" "http://sandbox-alb-95496877.us-east-1.elb.amazonaws.com/" --max-time 15 | head -20
```

You should finally see real Drupal HTML (probably a redirect to `/user/login` or the welcome page).

**Long-term architecture insight from this whole journey:** the deploy-host and PHP-fleet seeing different mount paths (`/var/www/sandbox/` vs `/var/www/`) is the *cause* of this trap. Whenever drush runs on the deploy-host, it compiles things with `/var/www/sandbox/` paths and stores them in shared FSx/DB cache; PHP-FPM then reads those and fails. Either:

1. **Mount FSx at `/var/www/$ENV/` on PHP/nginx boxes too** — every host sees the same path, no asymmetry. Plus better multi-tenancy story.
2. **Drush only ever runs from a PHP box** — never from the deploy-host directly. Riskier; deploy-host is the natural place for ops tooling.

For sandbox right now: truncate the DB cache and move on. The architectural fix is a follow-up worth doing before adding a second environment.
