Couple of things — first, the tmux scrollback, then where to look for proper logs.

**Recover tmux scrollback (right now, before the buffer rotates):**

```bash
# In the same tmux session — dumps full scrollback to a file
tmux capture-pane -p -S -100000 > /tmp/deploy.log
wc -l /tmp/deploy.log
less +G /tmp/deploy.log     # opens at the end
```

Or interactively: `Ctrl-b [` enters copy mode → PgUp/arrows to scroll, `q` to exit.

**Verify deploy state without trusting the scrollback** (these are the authoritative sources):

```bash
# Stack states — anything ROLLBACK or FAILED is bad news
aws cloudformation describe-stacks --region us-east-1 \
  --query 'Stacks[?starts_with(StackName, `cf-scalable-web-sandbox`)].[StackName,StackStatus,LastUpdatedTime]' \
  --output table

# Image Builder pipeline executions — what build-amis kicked off
aws imagebuilder list-image-pipeline-images \
  --image-pipeline-arn $(aws ssm get-parameter --name /sandbox/image-builder/php83-pipeline-arn --query Parameter.Value --output text) \
  --query 'imageSummaryList[0].[name,state.status,dateCreated]' --output table

# AMI rollout — did compute pick up the new ones?
aws ec2 describe-images --owners self \
  --filters "Name=tag:Environment,Values=sandbox" \
  --query 'Images[*].[Name,CreationDate,ImageId]' --output table | head -20

# ASG instance counts — anything 0/0 means the ASG didn't bring up new boxes
aws autoscaling describe-auto-scaling-groups \
  --query 'AutoScalingGroups[?contains(AutoScalingGroupName,`sandbox`)].[AutoScalingGroupName,DesiredCapacity,length(Instances)]' \
  --output table

# ALB target health — the actual proof
make health-check ENV=sandbox
```

**Where Image Builder writes its own logs** (more durable than tmux):
- Build/test step output goes to S3 at `s3://${env}-image-builder-${BucketSuffix}/build-logs/...` (configured in `cf-image-builder.yaml` → `InfrastructureConfiguration` → `Logging.S3Logs`).
- Each component's bash output is captured per-step. If a build failed, the `build.log` for the failing step has the full output.

If `make` itself died mid-pipeline rather than running clean to completion, the most likely culprit is one of:
- AMI build failed (cfn-lint passes but a component step like `apt-get install` failed) → check S3 build logs
- AMI built but `deploy-compute` couldn't roll the ASG (capacity, IAM, etc.) → CFN events on the compute-php stack
